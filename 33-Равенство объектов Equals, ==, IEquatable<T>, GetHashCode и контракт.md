### Вступление

Ты завёл доменный тип — скажем, `Money(Amount, Currency)` — и положил несколько таких объектов в `HashSet<Money>`, чтобы убрать дубликаты. Два объекта с `Amount = 100` и `Currency = "USD"` по всякой человеческой логике одинаковые. Но `set.Count` показывает два элемента, а `set.Contains(new Money(100, "USD"))` возвращает `false`. Объект будто провалился сквозь коллекцию.

Или другой день: ты сравниваешь два экземпляра оператором `==`, получаешь `true`, переписываешь тот же код на generic-метод `T Find<T>(...)` — и там сравнение перестаёт работать, хотя объекты те же.

Оба случая — это одна и та же тема, которую на собеседовании спрашивают почти всегда: **как в .NET устроено равенство**. Проблема, которую тема решает, простая по формулировке и коварная в деталях — «когда два объекта считаются равными» в языке имеет не один ответ, а несколько, и они не обязаны совпадать. Есть равенство по ссылке (это _тот же самый_ объект в памяти) и равенство по значению (это _логически такой же_ объект). Есть четыре разных механизма, через которые это равенство выражается, и они выбираются в разные моменты — часть компилятором, часть в рантайме. И есть жёсткий контракт между `Equals` и `GetHashCode`, нарушение которого не даёт ошибку компиляции, а тихо ломает все хеш-коллекции. Разберём это так, чтобы пережить любой follow-up.

### Mental model

> **Тождество (`ReferenceEquals`) — «это один и тот же объект». Равенство по значению (`Equals` / `==` / `IEquatable<T>`) — «это логически такой же объект», и его нужно определить самому.** Как только ты переопределяешь равенство по значению, `GetHashCode` обязан согласовываться с ним — иначе хеш-коллекции теряют объект.

Первое, что стоит держать в голове — карта того, какие вообще бывают «поверхности» равенства и чем они отличаются по механике.

### Как это работает

#### Четыре механизма — и что каждый делает по умолчанию

**`object.Equals(object? obj)`** — виртуальный метод, определённый на `System.Object`, значит есть у всех. По умолчанию для класса он делает `ReferenceEquals(this, obj)` — то есть сравнивает ссылки. Пока ты его не переопределил, «равенство по значению» у класса _совпадает_ с тождеством. Именно поэтому два разных объекта с одинаковыми полями по умолчанию не равны.

**`ReferenceEquals(a, b)`** — статический метод, который всегда сравнивает ссылки и _игнорирует любые переопределения_. Это способ спросить «это буквально один и тот же объект в куче?», даже если тип переопределил `Equals` и `==`. Тонкость: для value-типов `ReferenceEquals` всегда возвращает `false`, потому что оба аргумента боксируются в разные объекты.

**Статический `object.Equals(a, b)`** — безопасная обёртка: сначала обрабатывает `null` (оба `null` → `true`, один `null` → `false`), потом вызывает виртуальный `a.Equals(b)`. Этим удобно пользоваться внутри перегрузки `==`, чтобы не писать проверки на `null` руками.

**`operator ==` / `operator !=`** — это _статические операторы_, а не виртуальные методы. Их привязка решается компилятором по **статическому типу** переменных в точке вызова. Для reference-типов по умолчанию `==` сравнивает ссылки; для value-типов у пользовательского `struct` оператора `==` вообще нет (будет ошибка компиляции), пока ты его не напишешь — исключение составляют встроенные примитивы (`int`, `double`, …), у которых `==` встроен.

**`IEquatable<T>.Equals(T? other)`** — типобезопасный интерфейс:

```csharp
public interface IEquatable<T>
{
    bool Equals(T? other);
}
```

Он нужен не «для красоты», а по двум причинам. Первая — для структур он **избавляет от boxing**: `Equals(object)` требует упаковать значение в объект, `Equals(T)` работает напрямую со значением. Вторая — generic-коллекции (`List<T>.Contains`, `Dictionary<TKey,…>`, `HashSet<T>`) реально вызывают именно `IEquatable<T>.Equals`, а не `object.Equals`, через `EqualityComparer<T>.Default`. Без `IEquatable<T>` они падают на медленный fallback через `object.Equals` с приведением типов.

#### Ловушка №1: `==` привязывается по статическому типу, `Equals` — виртуально

Это самый частый способ завалить кандидата на follow-up. Возьмём класс, где переопределены и `Equals`, и `==` для сравнения по значению. Один и тот же объект под разными статическими типами даёт **разные** результаты `==`:

Практический вывод: если тип используется полиморфно (кладётся в `List<object>`, `object[]`, сравнивается под интерфейсом), полагаться на `==` для равенства по значению нельзя — нужен `Equals`. Это же причина, почему LINQ, `Dictionary` и прочие коллекции внутри вызывают `Equals`/`GetHashCode`, а не `==`: оператор им недоступен, они работают через `EqualityComparer<T>.Default`.

#### Контракт `Equals` ↔ `GetHashCode`

`GetHashCode()` возвращает `int` — числовой «отпечаток» объекта, по которому хеш-коллекции (`Dictionary`, `HashSet`, `Lookup`) сначала находят _бакет_, а уже внутри бакета сравнивают кандидатов через `Equals`. Отсюда железное правило:

> **Если `a.Equals(b) == true`, то `a.GetHashCode() == b.GetHashCode()` обязательно.** Обратное не требуется: разные объекты могут иметь одинаковый хеш (это допустимая коллизия, её разрулит `Equals`).

Формально контракт `Equals` — это отношение эквивалентности: рефлексивность (`a.Equals(a)`), симметрия (`a.Equals(b) == b.Equals(a)`), транзитивность, консистентность (результат не меняется, пока объект не изменился) и `a.Equals(null) == false`. Контракт `GetHashCode` — равные объекты дают равный хеш; хеш стабилен, пока объект используется как ключ; метод не бросает исключений и дёшев.

Что происходит, если переопределить `Equals`, но забыть `GetHashCode` (компилятор, к слову, предупредит — **CS0659**):

![[Pasted image 20260701161551.png]]

Канонический шаблон для класса, закрывающий весь контракт разом. Обрати внимание: `sealed` делает приведение через `as` безопасным по симметрии (об этом — в 🔬), а `HashCode.Combine` (появился в .NET Core 2.1) избавляет от ручного `x * 397 ^ y`:

```csharp
public sealed class Money : IEquatable<Money>
{
    public decimal Amount { get; }
    public string Currency { get; }

    public bool Equals(Money? other) =>
        other is not null &&
        Amount == other.Amount &&
        Currency == other.Currency;

    public override bool Equals(object? obj) => Equals(obj as Money);

    public override int GetHashCode() => HashCode.Combine(Amount, Currency);

    public static bool operator ==(Money? left, Money? right) =>
        left is null ? right is null : left.Equals(right);

    public static bool operator !=(Money? left, Money? right) => !(left == right);
}
```

**Ловушка изменяемого ключа.** Если поле, участвующее в `GetHashCode`, изменить _после_ того, как объект положен в `Dictionary`/`HashSet`, хеш объекта поменяется, но он останется в старом бакете. Теперь коллекция ищет его в новом бакете — и не находит: запись становится недостижимой («потерянный» ключ). Отсюда практическое правило: **ключи хеш-коллекций должны быть иммутабельны по полям равенства.** Именно поэтому `record` и типы-ключи обычно делают неизменяемыми.

### 🔬 Глубже / на синьора

#### Дефолтное равенство структур: почему оно медленное и как именно

У `struct` `Equals`/`GetHashCode` унаследованы от `System.ValueType`, и его реализация — двухпутевая. Есть быстрый путь (побайтовый `memcmp`) и медленный (reflection по полям с boxing на каждое поле). Выбор делает нативная проверка `CanCompareBits`:

![[Pasted image 20260701161605.png]]

`CanCompareBits` истинна, только если тип не содержит ссылок (managed-указателей), «плотно упакован» (нет padding-байтов между полями — иначе мусор в padding даст ложное неравенство), не содержит `float`/`double` (у них `+0.0 == -0.0` при разных битах и особая логика `NaN`, поэтому побайтово сравнивать нельзя) и не переопределяет сам `Equals`/`GetHashCode` в полях. Малейшее ссылочное поле или `double` — и `Equals` уходит в reflection: перебор полей через `FieldInfo`, упаковка значения каждого поля в `object` и вызов `.Equals`. На вставке 1000 элементов в `HashSet` это сотни тысяч рефлексивных вызовов.

Дефолтный `ValueType.GetHashCode` ещё коварнее: в медленном пути он для дистрибуции берёт хеш **первого нестатического поля** и «мешает» с id типа. Если у структуры первое поле часто одинаковое, все объекты валятся в один бакет — `HashSet` вырождается в связный список, вставка становится O(N²). **Вывод для синьора:** любую структуру, которая пойдёт в хеш-коллекцию или будет часто сравниваться, обязательно снабжают ручными `Equals`/`GetHashCode` + `IEquatable<T>`. Для C# 10+ есть дешёвый путь — `record struct` синтезирует всё это корректно.

#### Дефолтный `GetHashCode` reference-типа — это не адрес

Частый миф: «хеш ссылочного типа — это адрес объекта». Неверно, и по фундаментальной причине: GC при компактизации **двигает** объекты, а хеш обязан быть стабильным всю жизнь объекта. Поэтому в .NET Core/5+ дефолтный identity-hash — это псевдослучайное 32-битное число, вычисляемое лениво при первом вызове (из thread-local генератора) и кэшируемое в заголовке объекта (в sync block / object header). В .NET Framework использовался индекс sync block. В обоих случаях значение отражает _identity_ (разные экземпляры почти всегда дают разный хеш), но никак не привязано к адресу.

#### `is` против `GetType()` в `Equals` — ловушка симметрии

Если писать `Equals` через `obj is Person p`, то в иерархии наследования симметрия ломается: базовый объект «is `Base`» вернёт `true` для наследника, а наследник для базового — `false`, и `a.Equals(b) != b.Equals(a)`. Для value-based равенства по всей иерархии надо сравнивать точный тип: `GetType() == obj.GetType()`. Либо — что чаще и проще — **запечатать тип** (`sealed`), тогда `as`/`is` безопасны. Records решают это за тебя: сгенерированный `Equals` сравнивает `EqualityContract` (тип), поэтому наследник и предок никогда не равны.

#### `IEqualityOperators<TSelf, TOther, TResult>` (.NET 7+)

С приходом generic math появился статический абстрактный интерфейс `IEqualityOperators<,,>`, который выносит `operator ==`/`!=` в контракт. Он позволяет generic-коду ограничивать параметр типом, у которого есть `==` — то, что раньше было невозможно, так как операторы статические и не участвовали в интерфейсах. На собесе достаточно знать, что он существует и зачем (обобщённый код над операторами равенства).

#### Forward-reference

`record` и `record struct` — компилятор синтезирует `Equals`, `GetHashCode`, `==`, `!=` с семантикой равенства по значению (плюс `Deconstruct`, `ToString`). Это прямой ответ на всю ручную рутину из этой главы. Механику синтеза и подводные камни (наследование records, `EqualityContract`, `with`-копии) разбираем в **главе про Records**. Также рядом стоит **глава про `IEqualityComparer<T>` / `EqualityComparer<T>.Default`** — там про _внешнее_ равенство: как коллекции выбирают компаратор, кастомные компараторы и `StringComparer` (например, регистронезависимый словарь).

Про строки коротко, потому что это частый вопрос: `string`-овый `==` и `string.Equals(string)` по умолчанию **ordinal** (побайтовое сравнение code units, культуронезависимое). А вот `string.CompareTo` — **культурозависимый** (это про упорядочивание, не про равенство), что регулярно ловит людей: `"a" == "a"` ordinal, но сортировка через `CompareTo` учитывает культуру. Для явного контроля — `StringComparer.Ordinal` / `OrdinalIgnoreCase` (см. главу про компараторы).

### Interview layer

#### Вопросы, по частоте/сеньорности

**1. В чём разница между `==` и `Equals()`? (спрашивают почти всегда)**

- → _likely follow-up:_ почему для reference-типов они по умолчанию ведут себя одинаково?
- → _trap:_ `object o1 = p1; object o2 = p2; o1 == o2` — что вернёт, если `Person` переопределяет `==` для сравнения по значению? (`false` — `==` привязан по статическому типу `object` → ссылочное сравнение.)

**2. Почему при переопределении `Equals` надо переопределять `GetHashCode`? (почти всегда)**

- → _likely follow-up:_ что именно сломается и в каких коллекциях?
- → _trap:_ а если `GetHashCode` всегда возвращает `42` — код сломается? (Нет — контракт формально соблюдён, `Equals` разрулит коллизии. Но всё валится в один бакет → поиск деградирует до O(n).)

**3. Зачем `IEquatable<T>`, если уже есть `Equals(object)`? (часто)**

- → _likely follow-up:_ кто его реально вызывает?
- → _trap:_ для `struct` без `IEquatable<T>` что происходит при вставке в `Dictionary`? (Fallback на `ValueType.Equals` → reflection + boxing на каждое поле; при неудачном первом поле — ещё и коллизии в хеше.)

**4. Сформулируй контракт `GetHashCode`. (часто)**

- → _likely follow-up:_ должен ли хеш быть уникальным? (Нет, только: равные объекты → равный хеш.)
- → _trap:_ можно ли строить `GetHashCode` на изменяемом поле? (Нельзя, если объект — ключ: мутация после вставки «теряет» запись.)

**5. Как ведёт себя дефолтное равенство `struct` и почему это проблема? (mid+/senior)**

- → _trap:_ при каких условиях `ValueType.Equals` уходит на медленный reflection-путь? (Ссылочное поле, `float`/`double`, padding, свой `Equals` в поле — всё, что делает `CanCompareBits` ложной.)

**6. Дефолтный `GetHashCode` ссылочного типа — это адрес? (senior)** (Нет; identity-hash из sync block/header, стабилен несмотря на перемещения GC.)

**7. Почему `Equals` через `is` может нарушить симметрию? (senior)** (В иерархии `base is Derived` асимметрично; нужен `GetType()`-сравнение или `sealed`.)

#### Типичные поверхностные ответы и почему они проваливаются

- **«`==` сравнивает значения, `Equals` — ссылки.»** Перевёрнуто и неверно. По умолчанию у reference-типов _оба_ сравнивают ссылки; у value-типов оба — значения. Разница не в «значения vs ссылки», а в **привязке**: `==` статическая (компилятор), `Equals` виртуальная (рантайм).
- **«`GetHashCode` возвращает адрес объекта.»** Провал: GC двигает объекты, хеш обязан быть стабильным. Это identity-hash в заголовке.
- **«Достаточно переопределить `Equals`.»** Нет — хеш-коллекции сначала бьют по бакету через `GetHashCode`; без него равные объекты расходятся по бакетам и теряются.
- **«`IEquatable<T>` — это просто для скорости.»** Половина правды. Конкретика: он убирает boxing для структур и cast для классов, и его _реально_ вызывают generic-коллекции через `EqualityComparer<T>.Default`; без него — медленный fallback на `object.Equals`.

#### Model answers

**Q: Разница между `==` и `Equals`?**

**RU:** `Equals` — виртуальный метод, диспетчеризуется по реальному типу объекта в рантайме; `==` — статический оператор, компилятор выбирает его реализацию по статическому типу переменных. По умолчанию у ссылочных типов оба сравнивают ссылки, у значимых — значения. Ключевое следствие: если положить объект в переменную типа `object`, `==` даст ссылочное сравнение даже при переопределённом операторе, а `Equals` останется корректным. Поэтому коллекции и LINQ используют `Equals`/`GetHashCode`, а не `==`.

**EN:** `Equals` is a virtual method dispatched on the object's runtime type; `==` is a static operator whose implementation the compiler picks from the variables' _static_ type. By default reference types compare by reference and value types by value in both. The key consequence: assign the object to an `object`-typed variable and `==` falls back to reference comparison even if the type overloads it, while `Equals` still dispatches correctly. That's why collections and LINQ rely on `Equals`/`GetHashCode`, not `==`.  
_(senior add-on:_ for value types `Equals(object)` boxes, so implement `IEquatable<T>`; and `==` isn't even defined on a custom struct until you write it.)*

**Q: Почему `Equals` без `GetHashCode` ломает коллекции?**

**RU:** Хеш-коллекции (`Dictionary`, `HashSet`) сначала находят бакет по `GetHashCode`, и только внутри бакета сравнивают через `Equals`. Если я переопределил `Equals`, но оставил дефолтный `GetHashCode`, два логически равных объекта получают разный хеш, попадают в разные бакеты — и коллекция их не сматчит: дубликаты не отсеиваются, `Contains`/`TryGetValue` промахиваются. Контракт: равные по `Equals` объекты обязаны давать равный хеш.

**EN:** Hash-based collections locate a bucket via `GetHashCode` first, then compare within the bucket via `Equals`. If I override `Equals` but keep the default `GetHashCode`, two logically equal objects get different hashes, land in different buckets, and never get matched — duplicates slip through, `Contains`/`TryGetValue` miss. The contract is: objects equal by `Equals` must return equal hash codes.  
_(senior add-on:_ returning a constant hash is _correct_ but degrades every lookup to O(n) — one bucket; and hash fields must be immutable while the object is a key.)*

**Q: Зачем `IEquatable<T>`?**

**RU:** Он даёт типобезопасное сравнение без boxing. Для структуры `Equals(object)` вынуждает упаковать значение в объект; `Equals(T)` работает напрямую. И generic-коллекции через `EqualityComparer<T>.Default` вызывают именно `IEquatable<T>.Equals`, если тип его реализует, иначе падают на медленный `object.Equals`. Поэтому для типов-ключей я всегда реализую `IEquatable<T>` вместе с `Equals(object)` и `GetHashCode`.

**EN:** It gives type-safe comparison without boxing. For a struct, `Equals(object)` forces a box; `Equals(T)` works on the value directly. And generic collections, via `EqualityComparer<T>.Default`, call `IEquatable<T>.Equals` when the type implements it, otherwise falling back to the slower `object.Equals`. So for key types I always implement `IEquatable<T>` alongside `Equals(object)` and `GetHashCode`.
