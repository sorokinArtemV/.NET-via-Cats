

Калибровка **mid+**, упор на dispatch. Ответы свёрнуты (`<details>` раскрывается в Obsidian reading mode).

---

### 1. Базовый override
```csharp
class A { public virtual string M() => "A"; }
class B : A { public override string M() => "B"; }
A a = new B();
Console.WriteLine(a.M());
```
<details><summary>Ответ</summary>

`B`. `override` заменяет запись в слоте vtable; dispatch идёт по **фактическому** типу объекта (B), независимо от статического типа ссылки (A).
</details>

---

### 2. `new` через базовую ссылку
```csharp
class A { public virtual string M() => "A"; }
class B : A { public new string M() => "B"; }
A a = new B();
Console.WriteLine(a.M());
```
<details><summary>Ответ</summary>

`A`. `new` не трогает слот, а заводит отдельный член. Через ссылку статического типа A компилятор видит `A.M` → `A`.
</details>

---

### 3. Тот же `new`, но через derived-ссылку
```csharp
B b = new B();   // классы из задания 2
Console.WriteLine(b.M());
```
<details><summary>Ответ</summary>

`B`. Статический тип B «видит» свой скрывающий член. `new` диспетчеризуется по статическому типу — и тут он B.
</details>

---

### 4. Три уровня вперемешку
```csharp
class A { public virtual string M() => "A"; }
class B : A { public override string M() => "B"; }
class C : B { public new string M() => "C"; }

A a = new C();
B b = new C();
C c = new C();
Console.WriteLine($"{a.M()} {b.M()} {c.M()}");
```
<details><summary>Ответ</summary>

`B B C`. C использует `new`, поэтому **виртуальный слот** (введённый A, переопределённый B) остаётся на `B.M`. Через A и B идёт виртуальный dispatch по слоту → `B`. Через C статический тип видит новый член `C.M` → `C`.
</details>

### 5. `base.M()` внутри override
```csharp
class A { public virtual void Log() => Console.Write("A"); }
class B : A { public override void Log() { base.Log(); Console.Write("B"); } }
A a = new B();
a.Log();
```
<details><summary>Ответ</summary>

`AB`. `a.Log()` виртуально уходит в `B.Log`; внутри `base.Log()` — **невиртуальный** вызов `A.Log` (`call`, обходит vtable) → печатает `A`, затем `B`.
</details>

### 6. Виртуальный вызов в конструкторе базового класса
```csharp
class A { public A() => Init(); protected virtual void Init() { } }
class B : A
{
    private string _name;
    public B() { _name = "set"; }
    protected override void Init() => Console.WriteLine(_name ?? "null");
}
new B();
```
<details><summary>Ответ</summary>

`null`. Порядок при `new B()`:
1. инициализаторы полей B — здесь их нет;
2. тело ctor **A**, который виртуально зовёт `Init()` → диспетчеризуется в `B.Init` (фактический тип B), но `_name` ещё `null`, потому что тело ctor B не выполнялось;
3. тело ctor B ставит `_name = "set"` — поздно.

**Контраст:** будь `private string _name = "set";` (field initializer), он выполнился бы на шаге 1, **до** ctor A → напечатало бы `set`. Отсюда правило: не вызывай `virtual` из конструктора.
</details>

### 7. Абстрактный метод не переопределён
```csharp
abstract class Shape { public abstract double Area(); }
class Circle : Shape { }   // ?
```
<details><summary>Ответ</summary>

Ошибка компиляции (CS0534): `Circle` — конкретный класс и обязан реализовать `Area()`. Варианты: либо `override double Area() => …`, либо сделать `Circle` тоже `abstract`.
</details>

### 8. `sealed override` блокирует внука
```csharp
class A { public virtual void M() { } }
class B : A { public sealed override void M() { } }
class C : B { public override void M() { } }   // ?
```
<details><summary>Ответ</summary>

Ошибка компиляции (CS0239) на `C.M`: `B.M` запечатан `sealed override`, дальнейшее переопределение запрещено. `C` может скрыть через `new`, но не `override`.
</details>

### 9. Override + overload: `string` как `object`
```csharp
class Base
{
    public virtual void M(object o) => Console.Write("obj");
    public virtual void M(string s) => Console.Write("str");
}
object x = "hi";
new Base().M(x);
```
<details><summary>Ответ</summary>

`obj`. Перегрузка выбирается на компиляции по **статическому** типу аргумента (`object`), хотя значение — строка. «Overloading by static type». Чтобы попасть в `M(string)`, нужен статический тип string: `M((string)x)`.
</details>

### 10. Ковариантный возвращаемый тип (C# 9)
```csharp
class Animal { }
class Dog : Animal { }
class Shelter             { public virtual Animal Adopt() => new Animal(); }
class DogShelter : Shelter { public override Dog Adopt() => new Dog(); }   // ?
```
<details><summary>Ответ</summary>

Компилируется (C# 9+): covariant return types разрешают override сужать возвращаемый тип до более производного (`Dog` вместо `Animal`). До C# 9 — ошибка «return type must match».
</details>

### 11. Смена модификатора доступа в override
```csharp
class A { protected virtual void M() { } }
class B : A { public override void M() { } }   // ?
```
<details><summary>Ответ</summary>

Ошибка компиляции (CS0507): override обязан сохранять accessibility базового члена. `protected virtual` нельзя переопределить как `public`.
</details>

### 12. `override` поверх невиртуального
```csharp
class A { public void M() { } }
class B : A { public override void M() { } }   // ?
```
<details><summary>Ответ</summary>

Ошибка компиляции (CS0506): `A.M` не помечен `virtual`/`abstract`/`override` — переопределять нечего. Можно только скрыть через `new`.
</details>

### 13. Сокрытие без `new`
```csharp
class A { public void M() { } }
class B : A { public void M() { } }   // ?
```
<details><summary>Ответ</summary>

Компилируется, но даёт **предупреждение** CS0108: «hides inherited member; use the `new` keyword if hiding was intended». Поведение — сокрытие (как `new`), просто компилятор просит сделать намерение явным.
</details>

### 14. Override свойства
```csharp
class A { public virtual string Name => "A"; }
class B : A { public override string Name => "B"; }
A a = new B();
Console.WriteLine(a.Name);
```
<details><summary>Ответ</summary>

`B`. Свойства диспетчеризуются так же, как методы: `override` геттера заменяет слот, dispatch по фактическому типу.
</details>

### 15. Диспетчеризация через интерфейс
```csharp
interface IShape { string Draw(); }
class Circle : IShape { public string Draw() => "circle"; }
IShape s = new Circle();
Console.WriteLine(s.Draw());
```
<details><summary>Ответ</summary>

`circle`. Вызовы через интерфейс диспетчеризуются по **фактическому** типу объекта (через interface map), даже без ключевого слова `virtual` на методе класса — реализация интерфейса по сути виртуальна.
</details>

### 16. `new virtual` — переввод виртуальности
```csharp
class A { public virtual string M() => "A"; }
class B : A { public new virtual string M() => "B"; }
class C : B { public override string M() => "C"; }

A a = new C();
B b = new C();
Console.WriteLine($"{a.M()} {b.M()}");
```
<details><summary>Ответ</summary>

`A C`. `new virtual` в B **заводит новый виртуальный слот**, не трогая слот A. `C.override` переопределяет слот **B**, а не A. Поэтому через статический тип A работает слот A (никем не переопределён) → `A`; через B — новый слот, переопределённый C → `C`.
</details>

### 17. Конструктор у абстрактного класса
```csharp
abstract class Repository
{
    protected readonly string _conn;
    protected Repository(string conn) => _conn = conn;
}
```
<details><summary>Ответ</summary>

Да, у абстрактного класса есть конструктор — он не для создания экземпляра напрямую (нельзя), а вызывается из конструктора наследника через `base(conn)` для инициализации общего состояния. Абстрактный класс — это «класс с дыркой», а не «не класс».
</details>

### 18. Template Method: невиртуальный зовёт виртуальный
```csharp
class A { public void Run() => Step(); protected virtual void Step() => Console.Write("A"); }
class B : A { protected override void Step() => Console.Write("B"); }
A a = new B();
a.Run();
```
<details><summary>Ответ</summary>

`B`. Невиртуальный `Run` вызывает виртуальный `Step()`, который диспетчеризуется по фактическому типу (B) → `B.Step`. Это и есть паттерн Template Method: базовый класс фиксирует алгоритм, наследник подменяет шаги.
</details>

### 19. `base` к абстрактному члену
```csharp
abstract class A { public abstract void M(); }
class B : A { public override void M() => base.M(); }   // ?
```
<details><summary>Ответ</summary>

Ошибка компиляции (CS0205): нельзя вызвать абстрактный базовый член — у него нет реализации, к которой мог бы привязаться невиртуальный `base.M()`. `base` работает только когда у базы есть тело.
</details>

### 20. Когда `new` вообще оправдан
```csharp
// override vs new — что выбрать?
```
<details><summary>Ответ</summary>

`override` — настоящий полиморфизм через vtable; почти всегда то, что нужно. `new` — сокрытие: ломает полиморфизм (через базовую ссылку бежит базовый член) и обычно сигнализирует об ошибке дизайна.

Единственный по-настоящему легитимный кейс `new` — **версионирование библиотек**: базовый класс из чужой сборки добавил метод с твоей сигнатурой, и `new` явно фиксирует «я сознательно скрываю», убирая предупреждение.

Эвристика: ловишь себя на `new` в собственной иерархии — почти наверняка нужен `override` или переименование метода.
</details>
