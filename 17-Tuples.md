## Tuple / ValueTuple — лёгкий составной тип

### Зачем эта тема

Частая задача — вернуть из метода **несколько значений** без отдельного класса/DTO. До C# 7 варианты были так себе: `out`-параметры (некрасиво и не композируется), `Tuple<...>` (имена только `Item1`/`Item2`, отдельный объект в куче), анонимные типы (имена настоящие, но тип невыразим — нельзя вернуть из метода). `ValueTuple` (C# 7) закрыл это: value-тип с человеческими именами и удобным синтаксисом. Глава — про различия двух «кортежей» и про места, где на них ловят (особенно «имена в рантайме»).

### Mental model

Кортеж — **лёгкий безымянный составной тип** для «склеить несколько значений вместе». `ValueTuple` (C# 7, синтаксис `(a, b)`) — **value-тип с compile-time именами**; старый `Tuple<...>` — **immutable reference-тип** с именами `ItemN`.

### Tuple vs ValueTuple

||`Tuple<...>`|`ValueTuple` / `(a, b)`|
|---|---|---|
|Категория|class (reference type)|struct (value type)|
|Мутабельность|immutable (свойства read-only)|mutable (публичные поля)|
|Имена|только `Item1`, `Item2`…|именованные (compile-time)|
|Создание|`Tuple.Create(1, "x")`|`(1, "x")`|
|Версия|.NET 4.0 / до C# 7|C# 7.0 (тип — .NET 4.7)|
|Аллокация|отдельный объект в куче|где живёт контейнер|

**Про «быстрее» и «на стеке».** Частая полуправда — «`ValueTuple` всегда на стеке, поэтому быстрее». Точнее: `ValueTuple` — struct, поэтому **не аллоцирует отдельный объект в куче** и не требует разыменования. Но живёт он там, где живёт его контейнер: локальная переменная — на стеке, а поле класса / элемент массива / захват в замыкание / поле async-стейта → **в куче**. И «быстрее» — с оговоркой: это mutable struct, и при передаче он **копируется целиком**, так что для крупных кортежей копирование может стоить дороже одной ссылки.

### Имена — это compile-time трюк

Самое важное и любимое на собеседовании: **имена элементов кортежа существуют только на этапе компиляции**. В IL они стираются до `Item1`, `Item2`.

![[Pasted image 20260624164849.png]]

Где имена тогда «живут»? В локальном коде их подставляет компилятор (и IntelliSense). Через границу метода — на **публичной сигнатуре** они сохраняются особым атрибутом `TupleElementNamesAttribute` (его эмитит компилятор; писать его руками нельзя), чтобы вызывающий код, в том числе из другой сборки, видел `result.Min`, а не `result.Item1`. Но как только кортеж проходит через `object`, рефлексию или `dynamic` — имён нет, остаются только `Item1`, `Item2`. Это и есть ответ на засыпку «а в рантайме имена есть?» — нет.

### Деконструкция

Кортеж можно разложить на отдельные переменные:

csharp

```csharp
var (x, y) = point;                       // var снаружи — вывод типов
(int x, int y) = point;                   // явные типы
(var min, var max, _) = ComputeStats();   // _ — отбросить элемент
```

Ключевой нюанс: **деконструкция не уникальна для кортежей**. Любой тип, у которого есть метод `Deconstruct(out …)` (хоть свой, хоть extension), раскладывается тем же синтаксисом — это duck-typing компилятора:

csharp

```csharp
public void Deconstruct(out string first, out string last) { first = First; last = Last; }
var (f, l) = person;   // работает, потому что у Person есть Deconstruct
```

### Равенство кортежей

Здесь две _разные_ вещи, которые путают:

- **`Equals` / `GetHashCode`** — настоящие override прямо на структуре `ValueTuple` (с C# 7.0). Они поэлементны и корректны — именно поэтому кортеж **хорош как ключ словаря** или элемент `HashSet`: ручной `Equals`/`GetHashCode` писать не нужно, в отличие от обычной структуры с её медленным дефолтом (см. главу Equality).
- **Оператор `==` / `!=`** — появился в **C# 7.3**, но это **не** оператор на типе: компилятор через lowering **синтезирует** поэлементное сравнение в точке использования (для дженерик-структуры оператор невыразим). Сравнение поэлементное, с короткой замкнутостью, **имена игнорируются** — `(X:1, Y:2) == (A:1, B:2)` это `true`.

Нюанс на засыпку: у **1-элементного** `ValueTuple<T>` оператора `==` нет — `ValueTuple.Create(1) == ValueTuple.Create(1)` не компилируется (`CS0019`), потому что в языке нет понятия унарного кортежа и синтез для него не делается (хотя сам тип `ValueTuple<T>` существует — он служит звеном `TRest` в больших кортежах).

Оговорка-ловушка: кортеж — mutable struct, поэтому, положив его ключом в словарь, **не мутируй** его поля между операциями — это та же mutable-key проблема, что в главе Equality.

### Когда НЕ использовать

Кортеж хорош для **приватного** возврата нескольких значений, локальных хелперов, временных агрегатов. Плохо — в **публичном доменном API**: `Item1`/`Item2` протекают наружу, в рантайме нет имён, нет валидации и поведения, нельзя добавить метод. Если структура данных живёт в публичном контракте или имеет смысл сама по себе — заводи `record` (даёт имена, иммутабельность и value-equality «из коробки») или именованный тип. Кросс-ссылка на главу про records.

### 🔬 Глубже / на синьора

**Кортежи больше 7 элементов — вложенность через `TRest`.** У `ValueTuple` арность до 7 «прямая», а восьмой и дальше упакованы в последний элемент `TRest`, который сам является кортежем. `(1,2,3,4,5,6,7,8)` в IL — это `ValueTuple<int,int,int,int,int,int,int, ValueTuple<int>>`. Компилятор прячет это: `t.Item8` он транслирует в `t.Rest.Item1`. Поэтому формально кортеж может быть сколь угодно большим, но под капотом это вложенная матрёшка по 7.

**Mutable-struct следствия.** Поля `ValueTuple` — публичные и изменяемые (а не свойства, как у `Tuple`), потому что иначе value-семантика с деконструкцией была бы неудобна. Цена — обычные грабли mutable-структур: передаёшь по значению → мутируешь копию, а не оригинал; кортеж в `readonly`-поле не даст менять `Item1`. Для возвращаемых агрегатов это не мешает, но как «изменяемое состояние» кортеж держать не стоит.

**Контраст с анонимными типами.** Анонимный тип (`new { X = 1, Y = 2 }`) — наоборот: **reference type**, а имена у него **настоящие** (в метаданных), но сам тип **невыразим** в коде — его нельзя указать как возвращаемый, поэтому из метода не вернёшь (только `object`/`dynamic`). Кортеж ровно зеркален: имена «ненастоящие» (стираются), зато тип выразим и легко возвращается. Отсюда практический выбор: вернуть несколько значений — кортеж; локальная проекция в LINQ внутри одного метода — анонимный тип.

**Версии.** Сам тип `System.ValueTuple` появился в .NET Framework 4.7; на более старых таргетах синтаксис `(a, b)` работает, если подключить NuGet-пакет `System.ValueTuple`. `Tuple<...>` — с .NET 4.0.

### Interview layer

**Вопросы по частоте:**

**1. Чем `Tuple` отличается от `ValueTuple`?**  
→ follow-up: почему `ValueTuple` обычно быстрее?  
→ trap: он всегда на стеке?  
Суть: `Tuple` — class/immutable/`ItemN`; `ValueTuple` — struct/mutable/именованный. Не аллоцирует отдельный объект, но живёт там, где контейнер (поле/массив/замыкание → куча).

**2. Есть ли у `ValueTuple` имена в рантайме?**  
→ follow-up: где тогда живут имена?  
→ trap: что увидишь через рефлексию или после каста к `object`?  
Суть: имена — compile-time; на сигнатурах методов хранятся через `TupleElementNamesAttribute`; в рантайм-значении только `Item1`, `Item2`.

**3. Можно ли `ValueTuple` ключом словаря?**  
→ trap: надо ли вручную переопределять `Equals`/`GetHashCode`?  
Суть: да, хороший ключ — `Equals`/`GetHashCode` уже корректно переопределены на структуре; вручную ничего не нужно (но не мутируй ключ).

**4. Что такое деконструкция?**  
→ follow-up: только для кортежей?  
Суть: разложение на переменные; работает для любого типа с методом `Deconstruct` (duck-typing).

**5. (senior)** Кортеж в публичном API — норм? → нет: `Item1` протекает, нет имён в рантайме и валидации; бери `record`/именованный тип.

**6. (senior)** Что внутри кортежа из 10 элементов? → вложенность `TRest`: `ValueTuple<…7, ValueTuple<…>>`.

**Шаблонно-неверные ответы:**

- _«`ValueTuple` всегда на стеке»_ — нет, живёт где контейнер; в поле класса/массиве/замыкании — в куче.
- _«Имена кортежа — настоящая фича типа»_ — нет, compile-time сахар; в IL `Item1`/`Item2`.
- _«Структуру нельзя ключом словаря без ручного `Equals`»_ — `ValueTuple` уже переопределяет `Equals`/`GetHashCode`, в отличие от обычной структуры.
- _«`Tuple` и `ValueTuple` — одно и то же, просто синтаксис»_ — нет: class/struct, immutable/mutable, свойства/поля.

**Model answer (mid+):**

> Кортеж — это лёгкий составной тип, чтобы вернуть несколько значений без отдельного класса. `ValueTuple` (синтаксис `(a, b)`, C# 7) — структура с публичными полями и compile-time именами; старый `Tuple<...>` — иммутабельный класс с `Item1`/`Item2`. `ValueTuple` обычно быстрее, потому что не аллоцирует отдельный объект в куче, но он живёт там, где его контейнер: в поле или массиве это будет куча. Главное помнить про имена: они существуют только на этапе компиляции и стираются в IL до `Item1`, `Item2` — на сигнатурах методов их доносит атрибут `TupleElementNamesAttribute`, но через `object` или рефлексию имён уже нет. Ключом словаря `ValueTuple` годится: `Equals`/`GetHashCode` у него корректно переопределены. А вот в публичный доменный API кортеж не отдаю — там лучше `record`.

> _(senior add-on):_ оператор `==` для кортежей (C# 7.3) — это не оператор на типе, а синтез компилятора поэлементно; `Equals`/`GetHashCode` — настоящие override на структуре. Кортежи больше 7 элементов внутри вложены через `TRest` (`ValueTuple<…7, ValueTuple<…>>`), `Item8` транслируется в `Rest.Item1`. И зеркальный контраст: анонимный тип — reference с настоящими именами, но невыразимым типом; кортеж — наоборот.
