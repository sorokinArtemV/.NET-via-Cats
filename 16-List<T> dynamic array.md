### Зачем эта тема

`List<T>` — самый частый контейнер в C# и заодно лучший ответ на вопрос «зачем вообще нужны generics». До них был `ArrayList`: он хранил `object`, поэтому каждый `int` при добавлении **боксировался** в кучу, а на чтении требовался каст без проверки типа на этапе компиляции. `List<T>` это закрывает: типобезопасность на компиляции, **никакого boxing** для value-типов, а рантайм шарит один IL-код между всеми ссылочными `T` и специализирует код под каждый value-тип. Дальше — как он устроен и где на этом ловят.

### Mental model

`List<T>` — **обёртка над обычным массивом `T[]`**, который сам пересоздаётся, когда места не хватает. Почти всё поведение (стоимость операций, ловушки со структурами, рост) — следствие этого одного факта.

### Count vs Capacity

Две независимые величины:

- **`Count`** — сколько элементов сейчас в списке.
- **`Capacity`** — сколько влезает во внутренний массив до следующего пересоздания.

csharp

```csharp
var list = new List<int>();
// Count = 0, Capacity = 0  (внутренний массив пуст, аллокации ещё нет)
list.Add(1);
// Count = 1, Capacity = 4  (первый Add выделил массив на 4)
```

`Capacity ≥ Count` всегда. Понимание, что это разные числа, — фундамент для роста и для оптимизаций ниже.

### Рост и amortized-стоимость

Когда `Count == Capacity` и приходит новый `Add`, список **не может** дописать в текущий массив — массивы фиксированного размера. Он выделяет новый массив (`new List<T>()` стартует с 0 → 4 при первом `Add`, дальше каждый раз **×2**: 4 → 8 → 16…), **копирует** все элементы в него, и старый отдаёт сборщику.

Поэтому `Add` — **amortized O(1)**, но в момент resize это O(n) (копирование всех элементов). «Амортизированная» означает «в среднем по всем `Add`»: при росте от 1 до n удвоением суммарное число копирований — геометрический ряд `1 + 2 + 4 + … + n ≈ 2n`. То есть n добавлений стоят суммарно ~2n копирований → константа на одну операцию, несмотря на редкие дорогие resize.

### Сложность операций

- **Индексатор `list[i]`** — O(1): прямой доступ в массив.
- **`Add`** — amortized O(1) (см. выше).
- **`Insert(i, x)` / `RemoveAt(i)` / `Remove(x)`** — O(n): нужно **сдвинуть** все элементы правее точки вставки/удаления.
- **`Remove(x)`** удаляет **первое** вхождение (по `EqualityComparer<T>.Default`), `RemoveAll(predicate)` — все подходящие за один проход.

Идиома удаления в цикле — **с конца**, чтобы не ловить каскадные сдвиги и не сбивать индексы:

csharp

```csharp
for (int i = list.Count - 1; i >= 0; i--)
    if (ShouldRemove(list[i]))
        list.RemoveAt(i);
// либо просто: list.RemoveAll(ShouldRemove);
```

### Семантика хранения: ref vs value

Для ссылочных `T` список хранит ссылки, для value-типов — **копии значений** прямо в массиве. Отсюда классическая ловушка:

csharp

```csharp
struct Point { public int X, Y; }

var list = new List<Point> { new Point() };
list[0].X = 5;   // CS1612: Cannot modify the return value... (НЕ компилируется)

var arr = new Point[1];
arr[0].X = 5;    // OK
```

Разница в том, что **индексатор `List<T>` возвращает копию** структуры: индексатор — это property-getter, то есть метод, и он отдаёт _значение_, а не место в памяти; менять у временной копии бессмысленно, и компилятор это запрещает (`CS1612`). А доступ к элементу **массива** — это не property-getter, а встроенная операция языка (`ldelema`), которая даёт managed reference на сам элемент, поэтому запись проходит. Любимый mid+ вопрос на понимание «value vs reference + как работает индексатор». Обходной путь — переприсвоить целиком (`list[0] = new Point { X = 5 }`) или использовать `CollectionsMarshal.AsSpan` (senior-блок).

### Управление ёмкостью

Каждый resize — это аллокация + копирование. Если размер известен заранее, его задают сразу:

csharp

```csharp
var list = new List<int>(expectedCount);   // сразу нужная Capacity, без промежуточных удвоений
list.EnsureCapacity(expectedCount);        // то же, но на уже существующем списке (.NET 6+)
list.TrimExcess();                          // ужать Capacity до Count, отдать лишнюю память
```

Важно: **ёмкость сама не уменьшается** после удалений — удалил половину элементов, `Capacity` остался прежним. Хочешь вернуть память — `TrimExcess()` вручную (имеет смысл после массовых удалений, когда список больше не будет расти). Предвыделение оправдано, когда знаешь примерный размер: парсинг файла, выборка из БД, известное число итераций.

### Группы методов

|Категория|Методы|
|---|---|
|Добавление|`Add`, `AddRange`, `Insert`, `InsertRange`|
|Удаление|`Remove`, `RemoveAt`, `RemoveAll`, `RemoveRange`, `Clear`|
|Поиск|`IndexOf`, `Contains`, `Find`, `FindAll`, `BinarySearch` (на отсортированном)|
|Порядок|`Sort`, `Reverse` (см. главу IComparable — `Sort` нестабилен)|
|Конвертация|`ToArray`, `AsReadOnly`, `ConvertAll`|

`AddRange` вместо `Add` в цикле — один расчёт нужной ёмкости и меньше промежуточных копирований, когда источник знает свой размер.

### Сравнение с другими коллекциями

|Коллекция|Основа|Доступ по индексу|Вставка в середину|
|---|---|---|---|
|`List<T>`|массив|O(1)|O(n) (сдвиг)|
|`T[]`|массив|O(1)|— (фикс. размер)|
|`LinkedList<T>`|связный список|O(n)|O(1) при наличии узла|
|`HashSet<T>`|хеш-таблица|—|— (без порядка, без дубликатов)|

Нюанс про `LinkedList<T>`: формально O(1) на вставку _при уже найденном узле_, но на практике он почти всегда **медленнее** `List<T>` — каждый элемент это отдельный heap-объект-узел (аллокация + нагрузка на GC), и нет cache locality, которую даёт непрерывный массив. `HashSet` здесь — лишь чтобы показать, что есть и хеш-коллекции; их устройство — отдельная тема.

### Потокобезопасность

`List<T>` **не потокобезопасен**. Конкурентная модификация из нескольких потоков → порча данных или исключения. Прямого аналога `ConcurrentList<T>` в BCL **нет**: `ConcurrentBag`/`ConcurrentQueue` неупорядочены/FIFO и не дают доступа по индексу. Если нужна именно семантика списка с индексом под конкуренцией — `lock` вокруг `List<T>` или иммутабельный `ImmutableList<T>`.

### 🔬 Глубже / на синьора

**Инвалидация энумератора (`_version`).** Модификация списка внутри `foreach` по нему → `InvalidOperationException` «Collection was modified; enumeration operation may not execute.». Механизм: у списка есть поле `_version`, инкрементируемое при каждой структурной модификации; энумератор запоминает версию на старте и сверяет на каждом `MoveNext` — расхождение означает, что под ним поменяли коллекцию. (Базовый факт «нельзя менять список в `foreach`» — mid+, сам механизм `_version` — senior.)

**`List<T>.Enumerator` — это `struct`.** Поэтому `foreach` прямо по `List<T>` **не аллоцирует** энумератор в куче (он живёт на стеке) — заметная оптимизация на горячих путях. Но если список приведён к `IEnumerable<T>`/`IEnumerator<T>`, struct-энумератор **боксируется** в кучу, и преимущество теряется. Отсюда правило: на горячем пути не прячь `List<T>` за `IEnumerable<T>`, если важны аллокации.

**`CollectionsMarshal.AsSpan` (.NET 5+).** Даёт `Span<T>` поверх backing array **без копии** — и для value-типов это ref-доступ к элементам, обходящий копию-индектора из раздела про `ref vs value`:

csharp

```csharp
foreach (ref var p in CollectionsMarshal.AsSpan(list))
    p.X = 5;   // меняем элементы на месте, через ссылку
```

Это противоположность антипаттерну `list.ToArray().AsSpan()`, который сначала аллоцирует копию массива, а уже потом строит span. Опасность: span держит _текущий_ backing array — любой `Add`, вызвавший resize, делает span невалидным. Полное раскрытие `Span<T>`/`Memory<T>` — отдельная будущая тема.

**Backing array и LOH.** Внутренний массив — один непрерывный блок. Как только он перерастает **85 000 байт**, аллокация уезжает в Large Object Heap: он не компактится (фрагментация) и собирается вместе с gen2. Для `List<int>` это ~21 000+ элементов, для списка ссылок — ~10 000+ на 64-бит. Кросс-ссылка на тему GC/аллокаций.

### Interview layer

**Вопросы по частоте:**

**1. Как устроен `List<T>` под капотом?**  
→ follow-up: что происходит при переполнении?  
→ trap: какова амортизированная стоимость `Add` и почему именно такая?  
Суть: обёртка над `T[]`; при `Count==Capacity` — новый массив ×2 + копирование; amortized O(1) из-за геометрического ряда копирований.

**2. `Count` vs `Capacity`?**  
→ follow-up: как уменьшить `Capacity`?  
→ trap: ужмётся ли он сам после `RemoveAll`?  
Суть: `Count` — элементы, `Capacity` — вместимость массива; сама не сжимается, нужен `TrimExcess`.

**3. `List<T>` vs `Array` vs `LinkedList`?**  
→ follow-up: что быстрее при вставке в середину?  
→ trap: почему `LinkedList` на практике часто медленнее, хотя вставка O(1)?  
Суть: `List` — массив, O(1) индекс, O(n) вставка; `LinkedList` — узел на элемент, нет cache locality, аллокации.

**4. Можно ли менять `list[i].Field` для списка структур?**  
→ trap: а для массива? в чём разница?  
Суть: нет (`CS1612`) — индексатор `List` возвращает копию; у массива доступ к элементу даёт managed ref, поэтому работает.

**5. Что будет при изменении списка в `foreach`?**  
→ (senior) как это реализовано? → поле `_version`, `InvalidOperationException`.

**6. (senior)** Почему `foreach` по `List<T>` не аллоцирует, а по `IEnumerable<T>` — да? → struct-энумератор vs его boxing при касте.

**7. (senior)** Как получить `Span<T>` поверх списка без копии? → `CollectionsMarshal.AsSpan`.

**Шаблонно-неверные ответы:**

- _«`List` — это связный список»_ — нет, динамический массив (`T[]`).
- _«`Add` всегда O(1)»_ — amortized O(1); в момент resize O(n).
- _«`LinkedList` всегда быстрее для вставок»_ — на практике обычно медленнее из-за узлов и cache locality.
- _«`list[i].field = x` для структур просто работает»_ — не компилируется (`CS1612`).

**Model answer (mid+):**

> `List<T>` — это обёртка над массивом `T[]`. Есть `Count` (сколько элементов) и `Capacity` (вместимость массива); когда они сравниваются и приходит `Add`, список выделяет новый массив вдвое больше, копирует туда всё и меняет ссылку — поэтому `Add` амортизированно O(1), но в момент resize O(n). Индексатор — O(1), а `Insert`/`Remove` в середину — O(n) из-за сдвига. Если знаю размер заранее, задаю `Capacity` в конструкторе или через `EnsureCapacity`, чтобы избежать промежуточных удвоений. Помню, что для списка структур `list[i].Field = x` не компилируется — индексатор возвращает копию, в отличие от массива, где это managed reference.

> _(senior add-on):_ `List<T>.Enumerator` — структура, поэтому `foreach` по конкретному `List<T>` не аллоцирует, но при касте к `IEnumerable<T>` энумератор боксируется. Модификация в `foreach` ловится через поле `_version` и бросает `InvalidOperationException`. Для in-place доступа к элементам без копии беру `CollectionsMarshal.AsSpan`, помня, что resize инвалидирует span; а когда backing array перерастает 85 000 байт, он уходит в LOH.

### Linked List <-> List

Короткий ответ: да, в .NET тот же вывод, и для value-типов даже жёстче, чем в Java.

Причины те же самые, что в Java: cache locality и накладные расходы на каждый узел. Но в .NET есть специфический твист, которого в Java (до Valhalla) нет — и он делает разрыв ещё больше.

Покажу, в чём физическая суть разницы — именно раскладка в памяти решает всё:

![[Pasted image 20260624163938.png]]

Теперь по сути, почему в .NET разрыв даже **жёстче**, чем в Java:

В Java твой `ArrayList<Integer>` всё равно боксит каждый `int` в объект `Integer` на куче — массив хранит только ссылки, а сами числа разбросаны (до проекта Valhalla). То есть Java-шный «контейнер на массиве» уже частично страдает от того же разброса.

В .NET `List<int>` — это буквально `int[]`, числа лежат **инлайн**, плотно, без боксинга. Вот тот самый верхний ряд на картинке. Это даёт List в дотнете преимущество ещё больше, чем в Java. А `LinkedList<int>` каждый `int` заворачивает в `LinkedListNode<int>` — это `class`, ссылочный тип, отдельная аллокация на куче с полями `Value` + `Next` + `Prev` + ссылка на сам список. Тот самый нижний ряд.

Отсюда три удара по LinkedList:

Кэш — обход списка это прыжки по случайным адресам, промах кэша почти на каждом шаге. List идёт по памяти линейно, префетчер процессора заранее подтягивает следующие элементы.

GC и память — N элементов = N отдельных аллокаций + оверхед на заголовок объекта и два указателя на каждый узел. List это одна аллокация массива.

«O(1) вставка в середину» — миф на практике. Чтобы вставить в середину, надо сначала **дойти** до позиции, а это O(n) обхода с теми самыми промахами кэша. List при вставке делает `Array.Copy` — это `memmove`, который железо обожает. На реальных размерах List выигрывает почти всегда.

Где `LinkedList<T>` всё-таки оправдан — единственный честный кейс: когда у тебя **уже есть на руках ссылка на узел** (`LinkedListNode<T>`) и нужно O(1) удалить или переставить его без обхода. Канонический пример — **LRU-кэш**: держишь ноды в `Dictionary<key, LinkedListNode>`, и тогда «удалить» и «передвинуть в начало» реально O(1), обхода нет. Вот тут она к месту.

Ещё нюанс про очереди: в BCL нет аналога Java-шного `ArrayDeque`. `Queue<T>` — это кольцевой буфер на массиве, отличный FIFO. Иногда за двусторонней очередью тянутся к `LinkedList<T>`, но почти всегда array-backed решение быстрее.

Короче: дефолт в .NET — всегда `List<T>`, а `LinkedList<T>` доставай только когда держишь ссылки на узлы и нужен splice/remove без поиска.

### Промах кэша

Хорошо, давай разберём по-человечески. «Промах кэша» — это про то, что у процессора память не одна, а несколько уровней с разной скоростью, и иногда нужных данных не оказывается в быстрой.

Аналогия. Представь, что ты готовишь. У тебя есть:

Разделочная стола под рукой (это **кэш L1** — крошечный, но мгновенный доступ). Полка рядом (**L2/L3** — чуть дальше, чуть медленнее). И кладовка в другой комнате (**RAM** — огромная, но идти долго).

Когда тебе нужен ингредиент, ты сначала смотришь на стол. Если он там — это **попадание** (cache hit), берёшь мгновенно. Если его нет — это **промах** (cache miss), и приходится тащиться в кладовку. Готовка встаёт, пока ты ходишь.

Вот насколько разная цена «достать со стола» против «сходить в кладовку»:

![[Pasted image 20260624164538.png]]

Видишь, какая пропасть? Достать из L1 и сходить в RAM — это разница примерно в 50–100 раз. Если перевести в человеческий масштаб: будь доступ к L1 равен одной секунде, поход в RAM занял бы больше минуты. Процессор всё это время простаивает и ждёт.

Теперь важная деталь, без которой картинка неполная. Процессор **никогда не тащит из памяти один байт**. Он всегда забирает целый блок — **cache line**, обычно 64 байта. Это как: ты пошёл в кладовку за одним яйцом, но раз уж дошёл — хватаешь сразу весь лоток и ставишь на стол. Логика простая: «раз тебе понадобилось вот это, скорее всего следующим понадобится то, что лежало рядом».

Вот тут и сходятся все ниточки с прошлым вопросом про `List` и `LinkedList`.

`int` занимает 4 байта. Значит в одну cache line (64 байта) влезает **16 интов**. Когда ты в `List<int>` читаешь элемент `[0]`, процессор за один поход в память притащил тебе `[0]..[15]` сразу. Следующие 15 обращений — это попадания, мгновенные, из L1. Ты проходишь массив почти не спускаясь в медленную RAM.

А в `LinkedList<int>` каждый узел — отдельный объект, разбросанный по куче. Читаешь узел — процессор тащит cache line, но рядом с этим узлом лежит мусор (его же поля `Next`/`Prev`, заголовок объекта, чужие объекты), а вовсе не следующий по списку элемент. Следующий узел сидит по совсем другому адресу. Значит на каждый шаг по списку — **новый промах, новый поход в кладовку**. Префетчер процессора тут бессилен: он не умеет угадать случайный адрес из указателя, пока этот указатель не прочитает.

Отсюда и формулировка из прошлого ответа: обход списка — это «прыжки по случайным адресам с промахом почти на каждом шаге». Промах кэша — это и есть тот самый поход в кладовку вместо того, чтобы взять со стола. У `List` их мало, у `LinkedList` — на каждый элемент.
