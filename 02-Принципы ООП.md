## Принципы ООП

### 0. Что такое ООП

**Mental model.** ООП — парадигма, где программа моделируется как набор объектов; каждый объект объединяет состояние (данные) и поведение (методы), а взаимодействуют объекты через контракты, а не через прямой доступ к чужим внутренностям.

Четыре принципа — каждый одной фразой, с которой ведёшь устный ответ:

- **Encapsulation** — данные и операции над ними живут вместе; наружу торчит только то, что нужно, остальное защищено.
- **Abstraction** — работаем с тем, _что_ объект умеет, не зная, _как_ он это делает.
- **Inheritance** — новый тип переиспользует и специализирует существующий (отношение _is-a_).
- **Polymorphism** — один интерфейс, много реализаций; нужная выбирается за тебя в нужный момент.

В C# это собрано из `class` / `struct` / `interface` / `record`, модификаторов доступа и ключевых слов `virtual` / `override` / `abstract` / `sealed` / `new`. Порядок принципов держим сквозным: **E → A → I → P**.

### 1. Encapsulation

**Mental model.** Инкапсуляция — объект сам отвечает за свои инварианты: состояние спрятано, изменить его можно только через контролируемый интерфейс.

**How it works.**  
Два слоя: (1) _bundling_ — состояние и поведение в одном типе; (2) _access control_ — кто и как достаёт до состояния. В C# контроль доступа даёт 6 модификаторов:

|Modifier|Кто видит член|
|---|---|
|`private`|только сам тип|
|`public`|все|
|`protected`|тип + наследники|
|`internal`|всё внутри той же assembly|
|`protected internal`|наследники **ИЛИ** та же assembly (объединение)|
|`private protected`|наследники **И** та же assembly (пересечение) — с C# 7.2|

Ловушка прямо тут: `protected internal` — это **OR** (доступнее, чем каждый по отдельности), `private protected` — **AND** (строже). Их постоянно путают.

**Properties** — основной инструмент инкапсуляции: backing field + валидация в setter, computed-свойства, `get`/`set`/`init`. Почему свойство, а не публичное поле: можно поменять реализацию (добавить проверку, логику), не ломая публичный контракт, и сделать поле «только для чтения снаружи».

`data hiding` — это **механизм** реализации инкапсуляции (прячем данные), а **не отдельный принцип** и не синоним.

#### 🔬 Глубже / на синьора

- `init`-accessor (C# 9): сеттер работает только в конструкторе и object initializer → даёт неизменяемость без ручного readonly-поля.
- Смена `public field` → `public property` — **binary-breaking change**: меняется IL (доступ к полю vs вызов аксессора), ломаются уже скомпилированные потребители, поле нельзя передать по `ref`/`out`. Поэтому в публичных API свойства ставят сразу.
- Auto-property компилируется в скрытый backing field + пару аксессоров.

#### Interview layer

**Вопросы (по частоте):**

1. «Что такое инкапсуляция?» → FU: «чем отличается от abstraction?» → **trap:** «data hiding — это отдельный принцип?»
2. «Перечисли все access modifiers» → **trap:** «`protected internal` vs `private protected`?»

**Слабые ответы:**

- «Инкапсуляция — это `private`-поля» — путает механизм с принципом; не звучит про инварианты и контракт.
- «`protected internal` = и наследник, и та же сборка» — перепутаны OR/AND.

**Model answer (RU).** Инкапсуляция — объект держит своё состояние закрытым и отвечает за собственные инварианты; снаружи доступен только контролируемый интерфейс (обычно свойства/методы). Data hiding — частный механизм этого, не отдельный принцип. В C# контроль доступа реализуют 6 модификаторов; ключевая тонкость — `protected internal` это объединение (наследник _или_ сборка), а `private protected` — пересечение (наследник _и_ сборка).

**Model answer (EN).** Encapsulation means an object keeps its state private and owns its invariants; the outside world only touches a controlled interface, usually properties or methods. Data hiding is one mechanism behind it, not a separate pillar. C# enforces access with six modifiers; the subtle one is that `protected internal` is a union (derived _or_ same assembly) while `private protected` is an intersection (derived _and_ same assembly).

### 2. Abstraction

**Mental model.** Абстракция — выделяем существенное поведение и прячем несущественную сложность; думаем в терминах «что тип умеет», а не «как он устроен».

**How it works.**  
Показываем нужное поведение, скрываем сложность реализации. Инструменты в C# — `interface` и `abstract class`: они задают, _что_ можно делать, не фиксируя _как_.

**Abstraction vs Encapsulation** (главный trap-вопрос секции) — это разные уровни, не синонимы:

- **Abstraction** — про дизайн: какой набор операций ты выставляешь, что считаешь существенным. Решает «_что_».
- **Encapsulation** — про реализацию: как спрятано и защищено состояние за этим интерфейсом. Решает «_как спрятано_».

Конкретно: `List<T>` даёт тебе `Add` / `Count` / индексатор — это абстракция «динамический массив»; а то, что внутри `_items` и `_size` приватны и сам resize массива скрыт, — это инкапсуляция. Одно про витрину, другое про замок на складе.

#### 🔬 Глубже / на синьора

- Абстракция как **контракт**: код зависит от интерфейса, а не от конкретного класса — мост к DIP (глава SOLID).
- _Leaky abstraction_ — когда детали реализации «протекают» сквозь интерфейс (например, `IEnumerable`, который под капотом ходит в БД и бросает специфичные исключения).

#### Interview layer

**Вопросы:** «Abstraction vs encapsulation?» (классика, идёт почти всегда).

**Слабый ответ:** «Абстракция прячет данные, инкапсуляция прячет данные» — стирает разницу; интервьюер ждёт разведения уровней «что» vs «как спрятано».

**Model answer (RU).** Абстракция — про то, _что_ тип выставляет наружу как существенное поведение (через интерфейс/абстрактный класс), отбрасывая лишнее. Инкапсуляция — про то, _как_ за этим интерфейсом спрятано и защищено состояние. Первое — дизайн контракта, второе — защита реализации; обычно работают вместе.

**Model answer (EN).** Abstraction is about _what_ a type exposes as its essential behavior — modeling via interfaces or abstract classes and dropping the irrelevant. Encapsulation is about _how_ the state behind that surface is hidden and protected. One designs the contract, the other guards the implementation; they usually go together.

### 3. Inheritance

**Mental model.** Наследование — производный тип переиспользует и специализирует базовый, выражая _is-a_. Это самая жёсткая связь в ООП, поэтому применяй осознанно и предпочитай композицию, когда отношение не строго _is-a_.

**How it works.**

- **Single inheritance** для классов; **multiple** — только через интерфейсы.
- Корень всего — `System.Object`: даёт `ToString()`, `Equals()`, `GetHashCode()`, `GetType()` каждому типу. _(Контракт переопределения `Equals`/`GetHashCode` и records — отдельная глава Equality.)_
- **Конструкторы в иерархии.** Порядок при создании `Derived`:
    
    1. инициализаторы полей `Derived`;
    2. вызов `base(...)` → инициализаторы полей `Base` → тело конструктора `Base`;
    3. тело конструктора `Derived`.
    
    То есть тело базового конструктора отрабатывает **до** тела производного; `base(...)` явно или неявно вызывается всегда.
- **Виды**: multilevel (`A → B → C`) и hierarchical (несколько классов от одного базового) — поддерживаются, но для интервью второстепенны. Важны single (классы) и multiple-через-interfaces.

**Composition over inheritance.** Наследование жёстко связывает наследника с реализацией базы (_fragile base class_: изменение базы молча ломает наследников). Если отношение — _has-a_, а не _is-a_, бери композицию: вкладывай объект полем и делегируй.

> LSP (Liskov Substitution): наследник не должен ломать контракт базового типа — подробно в главе SOLID.

#### 🔬 Глубже / на синьора

- **Почему нет MI классов — diamond problem.** Если `D` наследует `B` и `C`, оба от `A`, возникает неоднозначность _состояния_ `A` (чья копия?) и разрешения членов. C# это запрещает для классов. **Default interface methods (DIM, C# 8 / .NET Core 3.0)** дают множественное наследование _поведения_ (но не _состояния_): интерфейс может нести реализацию по умолчанию, а конфликт двух default-методов резолвится явным `override` в реализаторе. _(Детали DIM — в секции 5.)_
- **Virtual-вызов из конструктора — gotcha.** В C# во время базового конструктора тип объекта уже самый производный, поэтому виртуальный вызов из тела `Base`-ctor уйдёт в `Derived.override`. Но тело `Derived`-ctor ещё не отработало (шаг 3 выше) — поля, которые он присваивает, ещё в default; инициализаторы полей `Derived` при этом уже сработали. Отличается от C++, где во время базового ctor «активна» базовая vtable.
- `sealed` на классе/методе включает JIT-девиртуализацию — см. секцию Polymorphism.

#### Interview layer

**Вопросы (по частоте):**

1. «Почему C# не поддерживает множественное наследование?» → FU: «как обойти?» → **trap:** «а DIM в C# 8 — разве не MI?»
2. «Inheritance vs composition — что выберешь и почему?»
3. (senior) «Что произойдёт при вызове virtual-метода из конструктора базы?»

**Слабые ответы:**

- «MI нет, потому что так решили / это плохо» — нет механизма (diamond, неоднозначность состояния).
- «Наследование = переиспользование кода» — переиспользование даёт и композиция; не звучит _is-a_ и цена жёсткой связи.

**Model answer — почему нет MI (RU).** Класс в C# наследует один базовый — это исключает diamond-неоднозначность состояния; множественность переиспользования достигается через интерфейсы, а с C# 8 интерфейсы могут нести и реализацию по умолчанию (DIM) — это множественное наследование поведения, но не состояния.

**Model answer — почему нет MI (EN).** A C# class inherits a single base, which rules out the diamond ambiguity over state; you get multiple reuse through interfaces, and since C# 8 interfaces can carry default implementations (DIM) — multiple inheritance of behavior, not of state.

**Model answer — inheritance vs composition (RU).** Наследование беру при истинном _is-a_, когда готов жёстко связать наследника с реализацией базы; композицию — при _has-a_ или когда нужно собирать поведение из независимых частей. По умолчанию предпочитаю композицию: она не страдает от fragile base class, легче тестируется и допускает подмену вложенного объекта в рантайме, тогда как наследование фиксирует связь на этапе компиляции.

**Model answer — inheritance vs composition (EN).** I use inheritance for a genuine _is-a_ when I'm willing to couple the subclass tightly to the base's implementation; composition for _has-a_ or when assembling behavior from independent parts. I default to composition: it avoids the fragile base class problem, is easier to test, and lets me swap the contained object at runtime, whereas inheritance fixes the relationship at compile time.

### 4. Polymorphism

**Mental model.** Полиморфизм — один интерфейс, много форм поведения; механизм расширяемости: код работает с базовым типом, а конкретная реализация выбирается за него.

**How it works.**  
Два вида:

- **Compile-time (static / early):** `method overloading`, `operator overloading`. Резолвит компилятор по сигнатуре на этапе компиляции.
- **Runtime (dynamic / late):** `overriding` через `virtual`/`override`. Резолвится по **runtime-типу объекта**.

**`virtual` / `override` / `new` — разница (ключевой trap):**

- `virtual` — разрешает переопределение; вызов идёт по типу **объекта** (runtime).
- `override` — переопределяет `virtual`-член; сам остаётся виртуальным дальше по цепочке.
- `new` — **method hiding**: не переопределяет, а _прячет_ член базы; вызов идёт по типу **ссылки** (compile-time).

csharp

```csharp
class Base { public virtual void M() => Console.WriteLine("Base"); }
class Over : Base { public override void M() => Console.WriteLine("Over"); }
class Hide : Base { public new      void M() => Console.WriteLine("Hide"); }

Base o = new Over();  o.M();      // "Over"  — override → по типу объекта
Base h = new Hide();  h.M();      // "Base"  — new → по типу ссылки
((Hide)h).M();                    // "Hide"  — каст меняет тип ссылки
```

`abstract` метод — `virtual` без тела; обязывает неабстрактного наследника дать `override`; класс с `abstract`-членом сам обязан быть `abstract`.  
`sealed` — на классе запрещает наследование; на `override`-методе запрещает дальнейшее переопределение.

#### 🔬 Глубже / на синьора

- **Таксономия полиморфизма:** _ad-hoc_ (overloading) / _subtype_ (overriding — «настоящий» ООП-полиморфизм) / _parametric_ (generics). Строго говоря, overloading многие не считают полиморфизмом — это разрешение перегрузки. На FU «overloading — это полиморфизм?» так и отвечай: формально ad-hoc, в отличие от subtype.
- **Parametric (generics):** один код работает с любым типом-параметром (`List<T>`, `Func<T>`), тип фиксируется статически в точке использования; CLR специализирует generic под value-типы (по одному коду на каждый struct-T) и шарит один код для всех reference-типов. В отличие от subtype, диспетчеризации по runtime-типу тут нет.
- **Covariant return types (C# 9):** `override` может вернуть более производный тип, чем базовый метод.
- **Механизм virtual dispatch:** у каждого reference-объекта в заголовке есть указатель на `MethodTable` его типа; виртуальный вызов идёт через слот в этой таблице (поэтому он чуть дороже невиртуального). Невиртуальные вызовы биндятся статически. `sealed`/отсутствие переопределений позволяют JIT сделать **девиртуализацию** и заинлайнить вызов.
- Спрятать член без `new` можно, но компилятор предупредит (CS0108) — `new` декларирует, что hiding намеренный.

#### Interview layer

**Вопросы (по частоте):**

1. «Compile-time vs runtime polymorphism?» → FU: «overloading — это полиморфизм?» → **trap:** «а generics — какой?»
2. «`virtual` / `override` / `new` — разница?» → FU: «что вызовется через ссылку базового типа?» → **trap:** «метод не `virtual`, а в наследнике `new` — что будет?»
3. (senior) «Как устроен virtual dispatch?»

**Слабые ответы:**

- «`new` и `override` — одно и то же» — фатально; разница в том, по какому типу резолвится вызов.
- «Полиморфизм — это перегрузка» — сводит всё к ad-hoc, теряет главное (subtype/overriding).

**Model answer (RU).** Полиморфизм бывает compile-time (overloading/operator overloading, резолвит компилятор) и runtime (overriding через `virtual`/`override`, резолвится по типу объекта). Главная тонкость: `override` диспетчеризуется по runtime-типу объекта, а `new` лишь прячет член и резолвится по compile-time-типу ссылки — поэтому через ссылку базового типа `new`-метод даст базовую реализацию, а `override` — производную.  
_(senior add-on)_ Под капотом виртуальный вызов идёт через слот MethodTable; `sealed`/девиртуализация позволяют JIT заинлайнить и убрать косвенность.

**Model answer (EN).** Polymorphism is compile-time (overloading/operator overloading, resolved by the compiler) or runtime (overriding via `virtual`/`override`, resolved by the object's type). The key subtlety: `override` dispatches on the runtime type, while `new` merely hides a member and resolves on the compile-time type of the reference — so through a base-typed reference a `new` method gives the base implementation, an `override` gives the derived one.  
_(senior add-on)_ Under the hood a virtual call goes through a MethodTable slot; `sealed`/devirtualization let the JIT inline it and drop the indirection.

### 5. `abstract class` vs `interface`

**Mental model.** `abstract class` — «частично готовый родитель» с общим состоянием и кодом (_is-a_, один родитель). `interface` — чистый контракт способностей (_can-do_, можно реализовать несколько).

**How it works.**

||`abstract class`|`interface`|
|---|---|---|
|Instance state (поля)|да|нет|
|Конструкторы|да|нет|
|Множественность|нет (1 base)|да (несколько)|
|Реализация по умолчанию|всегда|нет до C# 8; **DIM** с C# 8|
|Модификаторы доступа у членов|да|все `public` неявно до C# 8; явные с C# 8|
|Добавить член без слома реализаторов|да (с телом)|ломает до C# 8; не ломает через DIM (C# 8)|

csharp

```csharp
abstract class Shape                       // есть state + общий код
{
    public string Name { get; }            // instance state — интерфейс так не может
    protected Shape(string name) => Name = name;   // конструктор
    public abstract double Area();         // обязателен override в наследнике
}

interface IDrawable                        // контракт; можно реализовать несколько
{
    void Draw();
    void DrawTwice() { Draw(); Draw(); }   // DIM (C# 8): реализация по умолчанию
}

class Circle : Shape, IDrawable { /* один base-класс + любое число интерфейсов */ }
```

**Когда что:**

- `abstract class` — есть общее **состояние** + общий код, тесная _is-a_ иерархия, один родитель.
- `interface` — контракт для **несвязанных** типов, нужна множественность, абстракции для DI/плагинов.

#### 🔬 Глубже / на синьора

- **DIM как инструмент версионирования API:** можно добавить метод с дефолтной реализацией в опубликованный интерфейс, не ломая существующих реализаторов — раньше это умели только абстрактные классы.
- **Почему DIM ≠ abstract class:** у интерфейса по-прежнему нет instance state, и default-метод по умолчанию вызывается **только через ссылку интерфейсного типа**, а не через класс (если класс его не реализует явно). Это не полноценная замена базовому классу.

#### Interview layer

**Вопросы (топ главы):** «`abstract class` vs `interface`, когда что?» → FU: «может ли интерфейс иметь реализацию?» → **trap:** «тогда зачем вообще abstract class?»

**Слабый ответ:** «Интерфейс — это контракт, абстрактный класс — почти контракт» — после C# 8 разваливается на FU про DIM; не назван ключевой водораздел — **state**.

**Model answer (RU).** Абстрактный класс несёт общее состояние, конструкторы и реализацию для тесной _is-a_ иерархии с единственным родителем; интерфейс — контракт способностей, реализуемый множеством несвязанных типов. С C# 8 интерфейс умеет реализацию по умолчанию (DIM), но всё равно не имеет instance state — поэтому водораздел не «есть ли код», а «нужно ли общее состояние и один ли это родитель».

**Model answer (EN).** An abstract class carries shared state, constructors and implementation for a tight _is-a_ hierarchy with a single parent; an interface is a capability contract many unrelated types can implement. Since C# 8 an interface can have default implementations (DIM), but it still has no instance state — so the real dividing line isn't "does it have code" but "do you need shared state and is there a single parent."
