
`List<T>.Contains` на списке из миллиона элементов, вызванный в цикле, кладёт сервис. Профайлер показывает, что 90% времени — линейный поиск: каждый `Contains` пробегает список целиком, и это O(n) на вызов, O(n²) на цикл. Замена `List` на `HashSet` роняет то же время с секунд до миллисекунд — при том же коде поиска. В чём фокус? Список хранит элементы подряд и, чтобы найти нужный, сравнивает с каждым. Хеш-таблица сначала по `GetHashCode` вычисляет, _в какой ячейке_ элемент вообще может лежать, и сравнивает только с горсткой кандидатов в этой ячейке. Отсюда — среднее O(1).

Но у этого фокуса есть цена и есть способы его сломать. Плохой `GetHashCode` превращает O(1) обратно в O(n), тихо, без единой ошибки. Мутабельный ключ «теряется» в словаре. Ключ-структура без `IEquatable<T>` боксится на каждом сравнении. Проблема, которую решает эта глава: **понять, что именно происходит между вызовом `dict[key]` и возвратом значения** — какие массивы задействованы, когда дёргается `GetHashCode`, а когда `Equals`, почему размер таблицы — простое число и когда она пересобирается. Без этой картины перф-баги хеш-коллекций не отлаживаются, а на собесе follow-up про внутренности идёт почти всегда.

### Mental model

> **`Dictionary` — это не «магический массив по ключу», а два массива: `buckets` (хеш → номер цепочки) и `entries` (сами пары, связанные в цепочки через индексы).** Поиск: `GetHashCode` даёт бакет, внутри бакета короткая цепочка сравнивается через `Equals`. Среднее O(1) держится ровно до тех пор, пока хеш хорошо распределён и ключи иммутабельны.

### Как это работает

#### Анатомия: два массива, а не связный список

Внутри `Dictionary<TKey,TValue>` (и `HashSet<T>`) — два массива. `buckets` — массив `int`, где по индексу бакета лежит индекс _головы цепочки_ в массиве `entries` (пустой бакет помечен нулём — реально хранится индекс+1, чтобы 0 значил «пусто»). `entries` — массив структур `Entry`, каждая хранит `{ hashCode, next, key, value }`. Поле `next` — это индекс следующего элемента цепочки в том же массиве `entries`, а не ссылка на объект-узел.

![[Pasted image 20260701161808.png]]

Почему цепочка через индексы, а не `LinkedList` из объектов: оба массива лежат непрерывно в памяти (хорошая cache locality), на вставку элемента не аллоцируется отдельный узел, а сборщику мусора нечего обходить внутри массива значимых `Entry`. Это заметно быстрее классического «связного списка узлов» из учебников.

#### Путь поиска: `GetHashCode` — один раз, `Equals` — по совпадению хеша

Когда ты пишешь `dict.TryGetValue(key, out var v)`, происходит следующее:

![[Pasted image 20260701161821.png]]

Два важных вывода из этого пути. Первое: `GetHashCode` вызывается **один раз** на поиск, а `Equals` — только для тех кандидатов в бакете, у кого совпал закешированный `hashCode`. Хеш хранится прямо в `Entry`, поэтому сравнение int'ов отсекает почти всех до дорогого `Equals`. Второе: скорость всей структуры держится на **длине цепочек**. Пока они короткие — O(1). Если хеш плохой и всё валится в один бакет — цепочка растёт до n.

#### Что делает плохой `GetHashCode`

Классический анти-пример — `public override int GetHashCode() => 42;`. Контракт формально **не нарушен**: равные объекты дают равный хеш (все дают 42), `Equals` внутри бакета всё различит, ошибок не будет. Но все ключи попадают в один бакет, цепочка становится длиной n, и `Add`/`ContainsKey`/`TryGetValue` из O(1) деградируют в O(n):

![[Pasted image 20260701161833.png]]

#### Resize: когда и почему простое число

Массивы не бесконечны. Когда `entries` заполняется целиком (все слоты заняты), словарь **пересобирается**: выделяет новые массивы размером примерно вдвое больше (округлённым до следующего простого числа), переносит записи и пересчитывает бакеты. Важная деталь, которую часто путают: у `Dictionary<TKey,TValue>` фактический load factor ≈ 1.0 — resize происходит при **полном** заполнении, а не при 75%, как в Java `HashMap` или в старом `Hashtable`. Ещё две частые ошибки: размер таблицы — **простое** число, а не степень двойки (степень двойки — это Java); и словарь при удалении элементов **не уменьшается**.

Практический вывод: если знаешь примерный размер заранее — задавай ёмкость в конструкторе (`new Dictionary<K,V>(capacity)`) или через `EnsureCapacity`. Это разом выделяет массивы нужного размера и убирает серию дорогих resize с рехешированием.

#### Как писать `GetHashCode` правильно (и как не надо)

Контракт (`equal ⇒ равный hash`, стабильность, дешевизна) разобран в главе про `Equals`/`GetHashCode` — здесь только практика для ключей коллекций. Современный способ — `HashCode.Combine`, он корректно перемешивает поля без ручных костылей:

```csharp
public override int GetHashCode() => HashCode.Combine(Name, Age, Address);
```

Чего избегать:

- `unchecked { x * 397 ^ y }` — легаси-паттерн (его генерил старый ReSharper). Работает, но `HashCode.Combine` перемешивает лучше. Уточнение про `unchecked`: в C# арифметика по умолчанию _и так_ unchecked, переполнение int не бросает исключение; `unchecked` нужен, только если проект собран с `<CheckForOverflowUnderflow>true`. То есть это не «защита», а страховка на случай checked-сборки.
- `ToString().GetHashCode()` — плохо: лишняя аллокация строки на каждый вызов, а сам строковый хеш **нестабилен между запусками процесса** (рандомизирован, см. ниже), так что для ключа это ещё и ненадёжно.
- Нестабильные поля в хеше (`Guid.NewGuid()`, `DateTime.Now`, `Random`) — хеш обязан быть детерминированным. При этом сам по себе `Guid.GetHashCode()` — нормальный, надёжный хеш; проблема именно в _генерации нового_ значения.
- Мутабельные поля равенства у ключа — после изменения объект «теряется» в словаре (бакет уже не тот). Ключи хеш-коллекций должны быть иммутабельны по полям равенства.

Для структур-ключей всё то же плюс обязательный `IEquatable<T>` (иначе `EqualityComparer<T>.Default` уходит в `object.Equals` с boxing — см. главу про компараторы):

```csharp
public readonly struct Point : IEquatable<Point>
{
    public int X { get; }
    public int Y { get; }
    public bool Equals(Point other) => X == other.X && Y == other.Y;
    public override bool Equals(object? obj) => obj is Point p && Equals(p);
    public override int GetHashCode() => HashCode.Combine(X, Y);
}
```

`record` и `record struct` генерируют весь этот набор (`Equals`, `GetHashCode`, `IEquatable<T>`) сами — механику разбираем в главе про Records.

#### `HashSet<T>`: тот же движок без значения

`HashSet<T>` устроен идентично `Dictionary`, только в `Entry` нет поля `value` — хранится лишь ключ. Тот же `buckets` + `entries`, те же цепочки, тот же `EqualityComparer<T>.Default` (или переданный компаратор), те же O(1)-в-среднем и деградация при плохом хеше. Поэтому всё сказанное про `GetHashCode`, resize и коллизии переносится 1:1.

Разница — в назначении. `HashSet` отвечает на вопрос «есть ли элемент», и его сила — **операции над множествами**: `UnionWith`, `IntersectWith`, `ExceptWith`, `SymmetricExceptWith`, а также проверки `IsSubsetOf`, `IsSupersetOf`, `Overlaps`. Каждая из них — прогон по элементам с O(1)-проверками принадлежности, то есть в сумме O(n+m), тогда как «пересечь два списка» наивно — O(n·m).

Главный практический контраст — с `List<T>.Contains`:

||`List<T>.Contains`|`HashSet<T>.Contains`|
|---|---|---|
|Механизм|линейный проход + `Equals` на каждом|`GetHashCode` → бакет → `Equals` на горстке|
|Сложность|O(n)|O(1) в среднем|
|Когда|мало элементов, важен порядок/индексы|частые проверки принадлежности, дедупликация|

Если в коде есть `list.Contains(x)` внутри цикла по большому объёму — это первый кандидат на замену `HashSet`.

### 🔬 Глубже / на синьора

#### Механика самого хеша: `HashCode` = xxHash32, строки = Marvin, оба рандомизированы

`HashCode.Combine`/`HashCode.Add` под капотом — это **xxHash32** с глобальным случайным seed'ом, инициализируемым **один раз на процесс**. Отсюда важнейшее следствие: результат `HashCode.Combine(...)` **не стабилен между запусками** приложения (и тем более между версиями рантайма и платформами). Никогда не сохраняй такие хеши на диск, в БД или в протокол — на следующем запуске они другие.

Нюансы API `HashCode`: `Combine` принимает **до 8** аргументов; для большего числа полей (или для коллекции) — `Add` в цикле, затем `ToHashCode()`. Сам `HashCode` — **мутабельная структура**, у неё нельзя звать `GetHashCode()` (он помечен `[Obsolete]` и бросает) — только `ToHashCode()`:

```csharp
var hc = new HashCode();
foreach (var s in segments) hc.Add(s, StringComparer.OrdinalIgnoreCase);
return hc.ToHashCode();
```

Хеш **строк** (`string.GetHashCode()`) использует другой алгоритм — **Marvin32** — и тоже рандомизирован per-process (защита от hash-flooding). Это тот самый механизм, из-за которого `Dictionary<string,…>` на дефолтном компараторе стартует нерандомизированным и при длинной цепочке коллизий переключается на рандомизированный, форсируя resize с пересчётом хешей — разбирали в главе про компараторы.

#### `Combine` чувствителен к порядку

`HashCode.Combine(a, b) != HashCode.Combine(b, a)` — это by design (xxHash смешивает последовательно). Для типов, где равенство **не зависит от порядка** (множества, мультимножества), нельзя строить хеш через `Combine` по элементам в произвольном порядке обхода — иначе равные множества получат разные хеши. Нужна order-insensitive комбинация: XOR или сумма хешей элементов (ценой худшего распределения), либо сначала отсортировать.

#### Быстрое взятие по модулю и почему простое число

`bucket = hash % n` с делением — дорого. На 64-битных платформах `Dictionary` использует `HashHelpers.FastMod`: предвычисленный множитель, заменяющий деление на умножение+сдвиг. А простое `n` берётся потому, что `hash % prime` перемешивает биты равномернее, чем `hash % 2^k`: последнее фактически берёт только младшие `k` бит хеша, и если у хешей плохи именно младшие биты — идут массовые коллизии. Java обходит это дополнительным перемешиванием бит и живёт со степенью двойки; .NET выбрал простые числа. В `Entry` хеш хранится как `hash & 0x7FFFFFFF` (или `uint`) — чтобы `%` не давал отрицательный индекс. Побочный плюс кеширования хеша: при resize для ключей-**значимых** типов хеш не пересчитывается (пересчёт нужен только ссылочным, чей identity-hash в теории мог бы измениться при перемещении).

#### Free list: почему удаление не двигает массив

При `Remove` слот не сдвигается — он попадает в **free list** (`_freeList` + `_freeCount`), а поле `next` удалённой записи временно указывает на следующий свободный слот. Следующий `Add` берёт слот из free list, а не с конца массива. Поэтому удаления не вызывают рехеш и не оставляют «дыр», которые ломали перф старого `Hashtable`. Цена — словарь помнит пиковый размер и не ужимается.

#### `FrozenDictionary` / `FrozenSet` (.NET 8+)

Если коллекция строится один раз и потом только читается (конфиги, справочники, lookup'ы), есть `FrozenDictionary<K,V>` / `FrozenSet<T>` (`.ToFrozenDictionary()`). Построение дороже — рантайм анализирует ключи и подбирает оптимальную раскладку/хеш-стратегию, — но чтение заметно быстрее обычного `Dictionary`. Обычный `Dictionary` — по-прежнему дефолт для read-write; `Frozen` — для read-heavy после построения.

### Interview layer

#### Вопросы, по частоте/сеньорности

**1. Как `Dictionary` устроен внутри и как работает поиск? (спрашивают почти всегда)**

- → _likely follow-up:_ сколько раз за поиск вызывается `GetHashCode`, а сколько `Equals`?
- → _trap:_ зачем в `Entry` хранится `hashCode`, если есть `Equals`? (Дешёвое сравнение int отсекает кандидатов до дорогого `Equals`; при resize не надо пересчитывать.)

**2. Что происходит при коллизии? (часто)**

- → _likely follow-up:_ коллизия — это баг?
- → _trap:_ что будет, если `GetHashCode` вернёт константу? (Корректность цела, но всё в один бакет → O(n).)

**3. Почему `HashSet`/`Dictionary` быстрее `List<T>.Contains`? (часто)**

- → _trap:_ всегда ли? (Нет — на очень малых коллекциях линейный проход по массиву может выигрывать за счёт cache locality и отсутствия хеширования.)

**4. Когда происходит resize и почему размер — простое число? (mid+)**

- → _likely follow-up:_ при каком заполнении? (При полном, LF≈1, не 0.75.)
- → _trap:_ уменьшается ли словарь при `Remove`? (Нет.)

**5. Стабилен ли `string.GetHashCode()` / `HashCode.Combine` между запусками? (senior)** (Нет — рандомизированы per-process, Marvin для строк, xxHash32 для `HashCode`; нельзя персистить.)

**6. `HashCode.Combine(a,b)` и `Combine(b,a)` — одинаковы? (senior)** (Нет, чувствителен к порядку; для order-independent равенства нужен XOR/сумма.)

**7. `HashSet`-specific: сложность `IntersectWith`/`UnionWith`? (mid+)** (O(n+m) за счёт O(1)-проверок принадлежности, против наивного O(n·m).)

#### Типичные поверхностные ответы и почему они проваливаются

- **«`Dictionary` — это массив, индексируемый ключом».** Нет: это два массива (`buckets`+`entries`) плюс хеширование и цепочки. Ключ не индекс — из ключа _вычисляется_ бакет.
- **«Коллизии — это баг / их не бывает при хорошем хеше».** Коллизии нормальны и неизбежны (32 бита на бесконечное множество объектов); их разрешает цепочка + `Equals`. Проблема не в наличии коллизий, а в их количестве.
- **«Resize при 75% заполнения».** Это Java/`Hashtable`. Генерик `Dictionary` растёт при полном заполнении массива `entries`.
- **«Размер таблицы — степень двойки».** Это Java. .NET использует простые числа.
- **«`GetHashCode` возвращает адрес / он стабилен, можно сохранить».** Нет: дефолтный identity-hash — не адрес (см. главу 1), а `HashCode`/строковый хеш рандомизированы per-process.

#### Model answers

**Q: Как работает поиск в `Dictionary`?**

**RU:** Внутри два массива: `buckets` хранит для каждого бакета индекс головы цепочки в массиве `entries`, а `entries` хранит сами записи `{hashCode, next, key, value}`, связанные в цепочки через индекс `next`. При поиске `GetHashCode(key)` вызывается один раз, из него берётся номер бакета (`hash % n`, n — простое), дальше идём по цепочке: сначала сравниваем закешированный int-`hashCode` (дёшево), и только при совпадении зовём `Equals`. Среднее — O(1); при плохом хеше цепочки длинные и получается O(n).

**EN:** Internally there are two arrays: `buckets` holds, per bucket, the index of the chain head in `entries`, and `entries` holds the records `{hashCode, next, key, value}` linked into chains via the `next` index. On lookup, `GetHashCode(key)` is called once, the bucket comes from `hash % n` (n prime), then we walk the chain: first compare the cached int `hashCode` (cheap), and only on a match call `Equals`. Average O(1); with a poor hash the chains grow and it degrades to O(n).  
_(senior add-on:_ the hash is cached in the entry so `Equals` is skipped for hash-mismatches and value-type keys aren't rehashed on resize.)*

**Q: Что будет, если `GetHashCode` всегда возвращает одно значение?**

**RU:** Корректность не пострадает — контракт `equal ⇒ равный hash` формально соблюдён, а `Equals` внутри бакета всё различит. Но все ключи попадут в один бакет, цепочка станет длиной n, и `Add`/`ContainsKey`/`TryGetValue` деградируют из O(1) в O(n). Это тихий перф-баг: тесты зелёные, а под нагрузкой словарь работает как список.

**EN:** Correctness is fine — the `equal ⇒ equal hash` contract still holds and `Equals` disambiguates within the bucket. But every key lands in one bucket, the chain becomes length n, and `Add`/`ContainsKey`/`TryGetValue` degrade from O(1) to O(n). It's a silent performance bug: tests pass, but under load the dictionary behaves like a list.

**Q: Почему `HashSet.Contains` быстрее `List.Contains`?**

**RU:** `List.Contains` линейно проходит массив и зовёт `Equals` на каждом элементе — O(n). `HashSet` по `GetHashCode` сразу вычисляет бакет и сравнивает только с горсткой кандидатов в нём — O(1) в среднем. Оговорка: на очень маленьких коллекциях линейный проход по непрерывному массиву может выигрывать за счёт cache locality и отсутствия вычисления хеша.

**EN:** `List.Contains` scans the array linearly, calling `Equals` on each element — O(n). `HashSet` computes the bucket from `GetHashCode` and only compares against the few candidates there — O(1) average. Caveat: for very small collections a linear scan over a contiguous array can win thanks to cache locality and no hashing overhead.
