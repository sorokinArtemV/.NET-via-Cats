Анонимный тип — это immutable reference type, который компилятор генерирует на месте по литералу `new { … }`, без объявления именованного класса. Появились в C# 3 вместе с LINQ и ровно под него: когда внутри запроса нужно вытащить из объекта пару полей и тащить их дальше по цепочке, заводить ради этого DTO — церемония. `new { p.Name, p.Age }` даёт готовую типобезопасную «форму данных» прямо в выражении.

Проблема, которую они закрывают: **нужна структура из нескольких именованных полей с value-equality, живущая только внутри метода/запроса, и писать под неё класс не хочется.** Два главных сценария — проекции в LINQ (`Select`) и составные ключи в `GroupBy` (где сравнение по значению нужно «бесплатно»).

### Ментальная модель

`new { … }` — это «класс на один метод»: компилятор генерирует за тебя immutable `class` с read-only свойствами и переопределёнными `Equals`/`GetHashCode`/`ToString` по значениям. У него нет имени, которое можно написать, поэтому он не выходит за пределы метода (только как `object`/`dynamic`).

### Как это работает

**Синтаксис и вывод имён.** Имена свойств можно задать явно или дать компилятору вывести их из выражения — это называется _projection initializer_:

```csharp
var a = new { Name = "Ann", Age = 30 };   // имена заданы явно
var b = new { p.Name, p.Age };            // имена выведены: Name, Age
```

Вывести имя можно только из простого доступа к члену или переменной (`p.Name`, `name`). Для выражения (`p.Age + 1`) имя придётся задать руками.

**Что генерирует компилятор.** Под капотом — обычный `sealed class` с read-only-свойствами и тремя переопределёнными методами. Это ключевой момент: анонимный тип не имеет ничего общего с `dynamic` или словарём, доступ к свойствам полностью статически типизирован и проверяется на компиляции.

**Value-equality.** Поскольку `Equals`/`GetHashCode` переопределены по всем свойствам, два экземпляра равны, если совпали все значения. Это и делает анонимные типы пригодными как ключи словаря и `GroupBy`:

```csharp
var a = new { Name = "Ann", Age = 30 };
var b = new { Name = "Ann", Age = 30 };
a.Equals(b);            // True — по значениям
ReferenceEquals(a, b);  // False — это разные объекты в куче
```

(Механику `Equals`/`GetHashCode`/value-equality подробно разбирали в главе про равенство — здесь это применение того же контракта.)

**`var` обязателен, и тип заперт в методе.** Имя сгенерированного типа недоступно в исходниках, поэтому объявить переменную можно только через `var`. Из метода анонимный тип выходит лишь как `object` или `dynamic` — с потерей статической типизации:

```csharp
object MakePoint() => new { X = 1, Y = 2 };  // снаружи доступа к X/Y уже нет
```

Отсюда практическое правило: анонимный тип — инструмент _внутри_ метода/запроса. Нужно вернуть форму наружу — это сигнал брать `ValueTuple` или `record`.

**🔬 Глубже:** перетащить анонимный тип через границу метода всё же можно трюком «cast by example» — generic-метод выводит `T` из переданного образца той же формы (`TryCast<T>(object o, T shape)`). Это кунштюк для общего развития, в проде так не делают.

**Type unification.** Два литерала дают _один и тот же_ сгенерированный тип, если у них совпадают имена свойств, их типы **и порядок** — в пределах одной сборки. Значения не важны.

**Non-destructive mutation через `with` (C# 10).** Свойства read-only, присвоить нельзя. Но с C# 10 анонимные типы поддерживают `with`-выражение: оно не мутирует объект, а создаёт **новую** копию с изменёнными свойствами (вызывая конструктор сгенерированного типа). Добавить или убрать свойства `with` не может — только переопределить существующие.

```csharp
var p  = new { Name = "Ann", Age = 30 };
var p2 = p with { Age = 31 };   // C# 10+: новый объект { Name="Ann", Age=31 }, p не тронут
```

### Анонимный тип vs `ValueTuple` vs `record`

Это любимая развилка интервьюера: три способа собрать «структуру из именованных полей», и важно знать, чем они расходятся механически (а не на словах). `ValueTuple` разбирали в отдельной главе, `record` — впереди.

||Анонимный тип|`ValueTuple`|`record`|
|---|---|---|---|
|Категория|reference type (heap)|value type (struct)|reference (`record class`) или value (`record struct`)|
|Изменяемость|read-only (immutable)|**mutable** public-поля|immutable по умолчанию (`init`)|
|Equality|по значению|по значению|по значению|
|Имена членов в рантайме|реальные свойства|**стёрты** → `Item1`/`Item2` (имена только в compile-time)|реальные свойства|
|Можно вернуть / назвать тип|нет (только `object`/`dynamic`)|да: `(string Name, int Age)`|да|
|Свои методы|нет|нет|да|
|`with` (non-destructive)|да (C# 10)|нет (копия struct руками)|да (C# 9; `record struct` — C# 10)|
|Expression trees|**да**|**нет**|да|
|Аллокация|heap, на каждый экземпляр|нет (struct, inline/стек)|heap (`record class`) / нет (`record struct`)|
|Версия|C# 3|C# 7|C# 9 / 10|

Практический выбор: **анонимный тип** — форма «только внутри этого запроса/метода», с value-equality даром. **`ValueTuple`** — когда форму надо вернуть/передать/назвать и важна дешевизна (нет heap-аллокации). **`record`** — когда нужен настоящий именованный immutable-тип с методами, который живёт в доменной модели.

> 💡 С точки зрения MS, для большинства _нового_ кода предпочтителен `ValueTuple` (value type, deconstruction, дешевле). Анонимный тип остаётся правильным выбором там, где нужны reference-семантика или поддержка expression trees.

### 🔬 Глубже / на синьора

**Generic backing class.** Компилятор генерирует одно _обобщённое_ определение на каждую форму (набор имён + порядок), вида `internal sealed class <>f__AnonymousType0<TName, TAge>`, а конкретные типы свойств подставляются как type-параметры. Следствие: имена и порядок зашиты в _определение_ типа (разные имена/порядок → разные определения), а типы свойств — это аргументы дженерика (та же форма с другими типами свойств → то же определение, но другой constructed type). Именно эта обобщённость и даёт unification и переиспользование.

**Unification ограничена сборкой.** Сгенерированный класс `internal`, поэтому два идентичных по форме анонимных типа в _разных_ сборках — это разные типы. Внутри одной — один.

**Аллокации в горячих путях.** Анонимный тип — reference type, значит каждый экземпляр это аллокация в куче. Проекция миллиона элементов в `Select(x => new { … })` создаёт миллион объектов → давление на GC. Проекция в `ValueTuple` (`Select(x => (x.A, x.B))`) этого не делает: struct не аллоцируется в куче, пока остаётся в типизированной последовательности. Это и есть механизм за фразой «tuple быстрее» — он держится именно в hot LINQ-путях, а не «вообще всегда».

**Expression trees — почему EF любит анонимные типы.** Анонимные типы выражаются в expression trees, а кортежи — нет. Поэтому LINQ-провайдеры (EF и пр.) умеют транслировать `Select(x => new { x.Id, x.Name })` в SQL, а проекцию в `ValueTuple` — нет. Подробно про деревья выражений и `IQueryable` — в будущих главах; здесь достаточно знать сам факт и его причину.

### Интервью-слой

**Вопросы по частоте / seniority:**

1. _(очень часто)_ Что такое анонимный тип и где ты его применяешь?
2. _(часто)_ Он изменяемый или нет? Это reference или value type?
3. _(mid+)_ Чем анонимный тип отличается от `ValueTuple`? Когда что брать?
4. _(mid+, ловушка)_ Два `new { Name="x", Age=1 }` в разных местах метода — это один тип или разные? А если поменять порядок свойств?
5. _(mid+/senior)_ Можно ли вернуть анонимный тип из метода? Почему нет и как обойти?
6. _(senior)_ Почему EF умеет проецировать в анонимный тип, но не в кортеж?

**Дерево follow-up'ов:**

- Q: что такое анонимный тип? → likely: mutable или нет, reference или value? → **trap:** два литерала одинаковой формы — один тип? а что именно определяет тождество? _(ответ: имена + типы + порядок, в одной сборке)._
- Q: anon vs tuple? → likely: какой дешевле и почему? _(value type, нет heap-аллокации)_ → **trap:** тогда зачем вообще анонимные типы, если кортежи дешевле? _(reference-семантика и expression trees — кортежи в деревья выражений не ложатся, EF-проекция в них не транслируется)._

**Поверхностные ответы и почему они проваливаются:**

- «Это как `dynamic` / словарь» — нет: статически типизированный immutable-класс с реальными свойствами, доступ проверяется компилятором. Никакого позднего связывания.
- «Это просто кортеж» — нет: reference type vs value type; имена в рантайме реальные vs стёртые (`Item1`); разное поведение в expression trees; разные версии языка.
- «Они изменяемые, можно присвоить свойству» — нет: свойства read-only. «Изменение» — только через `with` (C# 10), который создаёт новый объект.
- «Порядок свойств не важен» — важен: другой порядок → другой сгенерированный тип.

**Модельный ответ (mid+):**

> Анонимный тип — это immutable reference type, который компилятор генерирует на месте по литералу `new { … }`. Свойства read-only, компилятор сам переопределяет `Equals`/`GetHashCode`/`ToString` по значениям всех свойств, поэтому экземпляры сравниваются по значению и годятся как ключи в `GroupBy`/словаре. Имени у типа нет — переменную объявляешь через `var`, и наружу он выходит только как `object`/`dynamic`, поэтому живёт внутри метода. Основной сценарий — проекции в LINQ: `Select(x => new { x.Id, x.Name })`. Два литерала с одинаковыми именами, типами и порядком свойств в одной сборке дают один и тот же сгенерированный тип.
> 
> 🔹 _Senior add-on:_ для нового кода обычно предпочитают `ValueTuple` — это value type без heap-аллокации и с deconstruction. Анонимный тип берут там, где нужны reference-семантика или поддержка expression trees: EF транслирует проекцию в анонимный тип в SQL, а в кортеж — нет.
