### Введение

Generics (обобщения) — это **параметризация типом**: тип становится параметром определения, а конкретный тип подставляется при использовании. Один класс / метод / интерфейс / делегат работает с любым `T`, не теряя строгой типизации.

Создавались, чтобы убрать боль обобщённого кода до C# 2.0 / .NET 2.0 (2005), когда «контейнер чего угодно» строили на `object` (`ArrayList`, `Hashtable`). Это давало три конкретные проблемы.

**1. Нет type safety.** `ArrayList` принимает `object`, поэтому компилятор не мешает положить рядом `int` и `string`. Ошибка всплывает только в runtime — на касте при чтении:

```csharp
var list = new ArrayList();
list.Add(42);
list.Add("oops");            // компилятор молчит
int x = (int)list[1];        // InvalidCastException в runtime
```

**2. Каст на каждом чтении.** Раз внутри `object`, обратно достаём через `(int)list[0]` — шумно и хрупко.

**3. Boxing для value-типов.** `ArrayList.Add(42)` упаковывает `int` в `object`: отдельная аллокация в куче на каждый элемент. 1000 чисел → 1000 boxed-объектов → давление на GC.

Альтернатива до generics — писать типизированный класс под каждый тип (`IntList`, `StringList`, …), то есть копипаст одного кода. Generics убирают всё три проблемы разом: один параметризованный тип, проверки на компиляции, без boxing, без дублирования.

### Mental model

**Тип как параметр: `T` подставляется при использовании и проверяется компилятором.** Один типобезопасный класс на все типы — без падения в `object`, без кастов, без boxing.

### Как это работает

**Generic-класс** — тип-параметр `T` объявляется в угловых скобках и используется внутри как обычный тип:

```csharp
public class Repository<T>
{
    private readonly List<T> _items = new();
    public void Add(T item) => _items.Add(item);
    public IEnumerable<T> GetAll() => _items;
}
```

**Generic-метод** — может жить и в обычном, не-обобщённом классе:

```csharp
public static void Swap<T>(ref T a, ref T b)
{
    (a, b) = (b, a);
}
```

**Generic-интерфейсы и -делегаты** — основа всей стандартной библиотеки: `IEnumerable<T>`, `IComparer<T>`, `Func<T, TResult>`, `Action<T>`, `Task<T>`. LINQ целиком построен на обобщённых методах. (Вариантность тип-параметров — `out T` / `in T` — вынесена в отдельную главу «Ковариантность и контравариантность».)

Ключевое про boxing — аккуратно. `List<int>` хранит элементы в backing-массиве `int[]`. Этот массив лежит **в куче**, но числа в нём — сырые `int`'ы подряд, **а не** boxed-объекты. То есть «нет boxing» означает не «значения на стеке», а «нет отдельной упаковки каждого элемента в `object` с отдельной аллокацией». Это формулировка, которую любят ломать фоллапом (см. interview-слой).

### Type inference

Для generic-**методов** компилятор обычно выводит `T` из аргументов — указывать явно не нужно:

```csharp
int x = 1, y = 2;
Swap(ref x, ref y);          // T выведен как int из аргументов
```

Вывод работает **только по аргументам**, не по возвращаемому типу. Если `T` фигурирует лишь в возвращаемом значении — вывести неоткуда, тип задаётся явно:

```csharp
public static T Create<T>() where T : new() => new T();

var a = Create();            // ❌ ошибка компиляции: T не вывести
var b = Create<Order>();     // ✅
```

> 🔬 Generic-**классы** вывод типов не поддерживают вовсе — `new Repository()` не скомпилируется, всегда `new Repository<Order>()`. Послабление есть только для конструкторов через target-typed `new` (C# 9): `Repository<Order> r = new();` — но это вывод из объявленного типа переменной, а не настоящий type inference по аргументам.

### Ограничения (constraints, `where`)

Без ограничения компилятор знает про `T` только то, что это `object` — доступны лишь `ToString()`, `Equals()`, `GetHashCode()`. Нельзя ни вызвать `T.CompareTo(...)`, ни сделать `new T()`, ни присвоить `null`. **Constraints расширяют «поверхность» `T`** — сообщают компилятору, что ещё с ним можно делать.

|Ограничение|Значение|Версия|
|---|---|---|
|`where T : class`|ссылочный тип|2.0|
|`where T : struct`|значимый тип, **исключая `Nullable<T>`**|2.0|
|`where T : new()`|есть public-конструктор без параметров|2.0|
|`where T : BaseClass`|наследник `BaseClass`|2.0|
|`where T : IInterface`|реализует интерфейс|2.0|
|`where T : U`|`T` наследует/реализует другой тип-параметр `U`|2.0|
|`where T : unmanaged`|unmanaged-тип (без ссылок внутри)|7.3|
|`where T : Enum` / `Delegate`|конкретно `System.Enum` / `System.Delegate`|7.3|
|`where T : notnull`|non-nullable value **или** reference|8.0|

Зачем механически: чтобы написать `if (a.CompareTo(b) > 0)`, нужно `where T : IComparable<T>` — иначе компилятор не видит метода `CompareTo` на `T`. Чтобы `new T()` — нужно `new()`.

**Порядок в одном `where` фиксированный:** сначала primary-ограничение (`class` / `struct` / `unmanaged` / базовый класс — что-то одно), затем интерфейсы и тип-параметры, в конце — `new()`:

csharp

```csharp
where T : BaseClass, IComparable<T>, new()
```

**Несколько тип-параметров — каждому свой `where`:**

csharp

```csharp
public class Cache<TKey, TValue>
    where TKey : notnull
    where TValue : class
{ }
```

> 🔬 Ловушка про `struct`: `Nullable<T>` (`int?`) сам по себе является struct'ом, но `where T : struct` его **исключает** — спец-правило компилятора, а не следствие «он не value-тип». До C# 7.3 не было `where T : Enum`, поэтому обобщения над enum'ами эмулировали через `where T : struct, IComparable, IConvertible, IFormattable` — встречается в старом коде.
> 
> 🔬 Нюансы nullability: `notnull` при нарушении даёт **warning, а не error**, и только в nullable-контексте. `class` в nullable-контексте означает **non-nullable** ссылку; `class?` разрешает `null`.

### `default(T)` и `default`

`default(T)` возвращает «нулевое» значение типа: `0` для числовых, `false` для `bool`, `null` для ссылочных, обнулённый экземпляр для struct'ов. Для `Nullable<T>` это `null` (`default(int?)` → `null`).

csharp

```csharp
public T GetOrDefault<T>(T? value) where T : struct
    => value ?? default;     // C# 7.1: литерал default, тип выводится
```

Литерал `default` без скобок (C# 7.1) пишется там, где тип ясен из контекста — присваивание, возврат, аргумент.

### 🔬 Глубже / на синьора: как generics живут в runtime

**Reified generics.** C# _не стирает_ типы. Информация о `T` сохраняется до рантайма: работают `typeof(T)`, `T` — настоящий тип, нет erasure-кастов. Контраст с Java, где **type erasure**: `List<String>` и `List<Integer>` в рантайме оба просто `List`, `T` стёрт до `Object`/границы. Из-за этого в Java нельзя `new T()`, `typeof(T)`, `is T`; в C# — можно. Reification — причина, по которой constraints, рефлексия по `T` и специализация под value-типы вообще возможны.

**JIT code-sharing** (диаграмма выше). CLR компилирует натив-код под обобщённый тип не одинаково для всех:

- Для **reference-типов** код **переиспользуется**: все ссылки одного размера (указатель) и ведут себя одинаково, поэтому `List<string>`, `List<object>`, `List<Customer>` делят один JIT-скомпилированный натив-код.
- Для **value-типов** генерируется **отдельная специализация** на каждый тип: `List<int>`, `List<double>`, `List<DateTime>` — у каждого свой код, потому что размеры и раскладка различаются. Отсюда **code bloat**: много value-инстанциаций раздувают объём натив-кода. Это и есть цена «нет boxing».

**Тип-идентичность ≠ общий код.** Даже когда reference-инстанциации делят натив-код, это _разные типы_: `typeof(List<string>) != typeof(List<object>)`. Делится тело кода, а не идентичность типа.

**Статика — на каждый closed-тип.** У каждого закрытого типа своя копия static-полей и свой запуск static-конструктора. `Counter<int>` и `Counter<string>` — независимые счётчики, даже несмотря на общий натив-код у reference-инстанциаций. Состояние привязано к закрытому типу, а не к телу кода. Классический trap (см. ниже).

**Open vs closed types.** `List<>` — открытый (unbound) тип с незаполненным параметром, `List<int>` — закрытый (constructed). Открытый нельзя инстанцировать; он живёт в рефлексии: `typeof(List<>)`, затем `MakeGenericType(typeof(int))`.

> За рамками главы: generic math — `static abstract` члены интерфейсов и `INumber<T>` (C# 11 / .NET 7) — отдельная тема.

### Interview-слой

#### Вопросы по частоте

**1. Чем generics лучше `object`? / Зачем они нужны? (топ)**  
→ likely follow-up: что именно происходит с `int` в `ArrayList` vs `List<int>`?  
→ trap: а где физически лежат элементы `List<int>` — на стеке?

**2. Что такое boxing и как generics его избегают?**  
→ follow-up: сколько аллокаций при добавлении 1000 `int` в `ArrayList`?  
→ trap: а в `List<int>` сколько boxing-аллокаций?

**3. Constraints — зачем и какие бывают?**  
→ follow-up: зачем нужен `new()`; как вызвать метод интерфейса на `T`?  
→ trap: `where T : struct` пропускает `int?`?

**4. Как generics работают в runtime, есть ли `T`? (reified vs erasure)**  
→ follow-up: чем отличается от Java?  
→ trap: `List<int>` и `List<string>` — один тип в рантайме или разные? Один код или разный?

**5. (trap) Статические поля на closed-типах.** `Counter<int>.Count` и `Counter<string>.Count` — общий счётчик?

**6. Type inference — когда `T` не выводится?**

#### Частые поверхностные ответы и почему валятся

- **«Generics = типобезопасность»** — верно, но неполно: без boxing/перформанса фоллап про производительность не закрыт.
- **«Value-типы остаются на стеке, поэтому нет boxing»** — ❌ неверно. Элементы `List<int>` лежат в массиве **в куче**. Правильно: они хранятся как сырые `int`, а не как boxed-`object` с отдельной аллокацией.
- **«В рантайме `T` стирается»** — ❌ это Java. В C# generics reified.
- **«Статические поля общие для `Foo<int>` и `Foo<string>`»** — ❌ раздельные, по комплекту на каждый closed-тип.

#### Модельные ответы

**Зачем generics (Q1).**

> Generics дают параметризацию типом с проверкой на компиляции. До них коллекции строились на `object`: не было type safety (ошибка типа всплывала в runtime на касте), приходилось кастовать на каждом чтении, и для value-типов был boxing — отдельная аллокация на элемент. Generics убирают всё это: один типобезопасный класс на все типы, без кастов и без boxing.  
> _(senior add-on)_ Альтернативой был бы отдельный типизированный класс под каждый тип — то есть дублирование, которое generics и устраняют.

**Boxing (Q2).**

> Boxing — упаковка value-типа в `object` с аллокацией в куче. В `ArrayList` добавление 1000 `int` даёт 1000 boxed-объектов и давление на GC. В `List<int>` boxing'а нет: значения лежат сырыми `int`'ами в backing-массиве `int[]`. Аллокаций — не на каждый элемент, а только при росте массива, когда удваивается capacity.

**Constraints (Q3).**

> Constraints расширяют то, что компилятор знает про `T`. Без них `T` виден только как `object` — доступны лишь методы `Object`. `where T : IComparable<T>` даёт право вызвать `CompareTo`, `where T : new()` — сделать `new T()`, `where T : BaseClass` — обращаться к членам базового типа. Основные: `class`, `struct`, `new()`, базовый класс, интерфейс; плюс `notnull` (8.0), `unmanaged` / `Enum` / `Delegate` (7.3). Порядок фиксированный: primary первым, `new()` последним.

**Runtime (Q4).**

> В C# generics reified: тип `T` сохраняется до рантайма, работают `typeof(T)`, рефлексия, `new T()`. В Java — type erasure, там `T` стирается до `Object`. `List<int>` и `List<string>` — разные closed-типы в рантайме. По натив-коду: reference-инстанциации делят один JIT-код, а под каждый value-тип CLR генерирует отдельную специализацию.

**Статика (Q5).**

> Раздельные. Static-поля и static-конструктор существуют по комплекту на каждый закрытый тип. `Counter<int>` и `Counter<string>` — независимые счётчики. Общий натив-код у reference-инстанциаций на это не влияет: состояние привязано к closed-типу, а не к телу кода.

**Type inference (Q6).**

> Для generic-методов `T` выводится из аргументов, поэтому `Swap(ref x, ref y)` пишется без `<T>`. Вывод идёт только по аргументам — если `T` встречается лишь в возвращаемом типе (`T Create<T>()`), вывести неоткуда, тип задаётся явно: `Create<Order>()`. Generic-классы вывод типов не поддерживают вовсе.

## Вариантность (ковариантность и контравариантность)

### Введение

Вариантность — это правила, по которым обобщённый тип с одним `T` можно присваивать такому же типу с _родственным_ (по наследованию) `T`. По умолчанию generic-типы в C# **инвариантны**: даже если `Cat : Animal`, типы `IEnumerable<Cat>` и `IEnumerable<Animal>` между собой никак не присваиваемы.

Это часто мешает. Передать список котов туда, где принимают «последовательность животных только на чтение» — безопасно (читать `Animal`'ов из коллекции `Cat`'ов нормально), но до C# 4.0 компилятор это запрещал:

csharp

```csharp
IEnumerable<Cat> cats = ...;
IEnumerable<Animal> animals = cats;   // до C# 4.0 — ошибка; с C# 4.0 — OK
```

Вариантность (C# 4.0 / .NET 4, 2010) снимает ограничение **там, где это статически безопасно** — и только там. Не «ослабление типобезопасности», а её аккуратное расширение.

(База по дженерикам — `T`, constraints, reified, boxing — в главе «Generics (механика)»; здесь только вариантность.)

### Mental model

**`out T` = producer:** тип только _отдаёт_ `T` (выходные позиции) → ковариантен, присваиваемость идёт **в ту же сторону**, что и наследование (`IEnumerable<Cat>` → `IEnumerable<Animal>`).  
**`in T` = consumer:** тип только _принимает_ `T` (входные позиции) → контравариантен, присваиваемость **разворачивается** (`Action<Animal>` → `Action<Cat>`).

### Как это работает

Вариантность задаётся модификатором `in`/`out` на тип-параметре и существует **только у интерфейсов и делегатов** — не у классов. Поэтому, например, `Task<T>` (класс) ковариантным быть не может в принципе, даже если `T` только возвращается; для ковариантного «результата» нужен интерфейс (`IEnumerable<out T>` и т.п.).

**Ковариантность (`out T`).** Если `Cat : Animal`, то `IEnumerable<Cat>` присваиваем переменной `IEnumerable<Animal>`. В BCL: `IEnumerable<out T>`, `IEnumerator<out T>`, `IReadOnlyList<out T>`, `IReadOnlyCollection<out T>`, `Func<…, out TResult>`.

csharp

```csharp
public interface IProducer<out T> { T Get(); }   // T только возвращается

IProducer<Cat> catSource = ...;
IProducer<Animal> animalSource = catSource;       // OK: каждый Cat — это Animal
```

**Контравариантность (`in T`).** Если `Cat : Animal`, то `Action<Animal>` присваиваем переменной `Action<Cat>`. В BCL: `Action<in T>`, `IComparer<in T>`, `IEqualityComparer<in T>`, `IComparable<in T>`.

csharp

```csharp
public interface IConsumer<in T> { void Handle(T item); }   // T только принимается

IConsumer<Animal> animalHandler = ...;
IConsumer<Cat> catHandler = animalHandler;        // OK: кто умеет с любым Animal, справится с Cat
```

**Инвариантность (дефолт).** `List<T>`, `IList<T>`, `Dictionary<TKey,TValue>` — и читают, и пишут `T`, поэтому `T` не помечен ни `in`, ни `out`. Ни одно направление присваивания не разрешено.

### Почему именно так

Это ядро темы — interviewer всегда копает сюда.

**Ковариантность безопасна на чтение.** Ковариантный тип `T` только _отдаёт_. Взяли `IEnumerable<Cat>` как `IEnumerable<Animal>` — всё, что вычитываем, это `Cat`, а `Cat` _is-a_ `Animal`. Положить ничего нельзя (нет входных позиций) — нарушить нечем.

**Контравариантность безопасна на запись.** Контравариантный тип `T` только _принимает_. `Action<Animal>` умеет обработать любого `Animal`; используем его как `Action<Cat>` — на вход придёт `Cat`, а это тоже `Animal`. Обработчик справится.

**Инвариантность — потому что мутабельный контейнер делает и то, и другое.** Будь `List<Cat>` присваиваем `List<Animal>` — через ссылку `List<Animal>` можно было бы записать `Dog` в список котов:

csharp

```csharp
List<Cat> cats = new();
List<Animal> animals = cats;   // ГИПОТЕТИЧЕСКИ, если бы было разрешено
animals.Add(new Dog());        // в списке котов лежит собака — взрыв на чтении
```

Раз контейнер и читает, и пишет `T`, безопасного направления нет — отсюда инвариантность по умолчанию.

### Value-типы не участвуют

Вариантность работает **только при reference-conversion** — когда представление в памяти не меняется (обе стороны — ссылки). Поэтому с value-типами она не действует:

csharp

```csharp
IEnumerable<int> ints = ...;
IEnumerable<object> objs = ints;   // ❌ ошибка компиляции
```

`int` → `object` — это **boxing**, конверсия со сменой представления, а не reference-conversion. Так что value-типовые аргументы всегда ведут себя инвариантно.

### 🔬 Глубже / на синьора

**Array covariance — «сломанная» ковариантность.** Массивы в C# ковариантны с версии 1.0: `string[]` присваиваем `object[]`. Но это **небезопасно**, и проверка переехала в runtime:

csharp

```csharp
string[] strings = { "a", "b" };
object[] objects = strings;        // компилируется: массивы ковариантны
objects[0] = new object();         // ArrayTypeMismatchException в runtime
```

CLR на _каждую запись_ в массив по ковариантной ссылке делает type-check (covariant array store check) — отсюда и исключение вместо порчи памяти, и небольшая цена на запись. Откуда это наследие: до generics (до .NET 2.0) ковариантность массивов позволяла писать обобщённые `Array.Sort(object[])` и подобное; ту же модель C# унаследовал от Java. **Generic-вариантность это и чинит:** она разрешена только в статически безопасных позициях и проверяется компилятором — runtime-сюрприза, как с массивами, тут быть не может.

**Правила позиций.** `out T` (covariant) можно ставить только в **выходных** позициях — возвращаемые типы, get-only-свойства; в параметрах метода — нельзя. `in T` (contravariant) — только во **входных** (параметры метода); в возвращаемом типе — нельзя. Параметр по значению считается входом, return — выходом.

`ref`- и `out`-параметры требуют **инвариантного** `T`: ни ковариантный, ни контравариантный `T` нельзя использовать как тип `ref`/`out`-параметра (ошибка компилятора **CS1961**). Для `ref` это очевидно — он и вход, и выход. Для `out` (хотя метод туда только пишет) C# тоже требует инвариантности; послабления нет (на момент .NET существует лишь языковое предложение его разрешить).

**`Func` совмещает обе вариантности.** `Func<in T1, …, in Tn, out TResult>` — контравариантен по всем аргументам и ковариантен по возвращаемому типу. Наглядный пример, что один тип может быть `in` по одним параметрам и `out` по другому одновременно.

**Declaration-site (C#) vs use-site (Java).** В C# вариантность объявляется **один раз** автором типа на тип-параметре (`IEnumerable<out T>`) — у потребителя всё «просто работает». В Java вариантности на типе нет; её указывает **потребитель на каждом использовании** через wildcards: `List<? extends Number>` (ковариантно), `List<? super Integer>` (контравариантно). Use-site гибче (можно выбирать на месте), но многословнее.

### Interview-слой

#### Вопросы по частоте

**1. Что такое co/contravariance? (definitional, топ)**  
→ follow-up: пример из BCL для каждого.  
→ trap: а `List<T>` ковариантен?

**2. `out` vs `in` — что где?**  
→ follow-up: почему `out T` нельзя в параметре метода?  
→ trap: где здесь producer/consumer?

**3. Почему covariance безопасна только на чтение / почему `List<T>` инвариантен?**  
→ follow-up: что сломается, если разрешить `List<Cat>` → `List<Animal>`?

**4. (трюк) Array covariance.** `string[]` → `object[]` компилируется — это нормально?  
→ trap: что произойдёт при записи и когда?

**5. Почему `IEnumerable<int>` нельзя присвоить `IEnumerable<object>`?**

**6. Declaration-site vs use-site (senior, vs Java).**

#### Частые поверхностные ответы и почему валятся

- **«Ковариантность — это про наследование дженериков»** — слишком размыто; на фоллапе «в какую сторону и почему безопасно» рассыпается.
- **«`out` для возврата, `in` для параметров»** — мнемоника верная, но без «producer/consumer» и «почему безопасно» не закрывает trap про `List<T>`.
- **«Массивы ковариантны, это удобно»** — ❌ упускает, что это небезопасно и кидает `ArrayTypeMismatchException` в runtime.
- **«`IEnumerable<int>` → `IEnumerable<object>` работает, ведь всё — `object`»** — ❌ нет: тут boxing, не reference-conversion; вариантность не применяется.

#### Модельные ответы

**Definitional (Q1).**

> Вариантность — совместимость generic-типов по наследованию их `T`. Ковариантность (`out`): если `Cat : Animal`, то `IEnumerable<Cat>` присваивается `IEnumerable<Animal>` — направление сохраняется. Контравариантность (`in`): `Action<Animal>` присваивается `Action<Cat>` — направление разворачивается. По умолчанию типы инвариантны (`List<T>`); вариантность задаётся `in`/`out` на тип-параметрах интерфейсов и делегатов (с C# 4.0).

**out vs in (Q2).**

> `out T` — тип только отдаёт `T` (producer): `T` в возвращаемых позициях, как у `IEnumerable<out T>`, `Func<out TResult>`. `in T` — тип только принимает `T` (consumer): `T` в параметрах, как у `Action<in T>`, `IComparer<in T>`. Поэтому `out T` нельзя в параметре метода, а `in T` — в возвращаемом типе.

**Почему так (Q3).**

> Ковариантность безопасна, только если `T` не пишется внутрь: при чтении любой `Cat` остаётся `Animal`. `List<T>` и читает, и пишет `T`, поэтому инвариантен: будь `List<Cat>` присваиваем `List<Animal>`, через эту ссылку можно было бы добавить `Dog` в список котов и взорваться на чтении.

**Array covariance (Q4).**

> Компилируется, потому что массивы ковариантны ещё с C# 1.0, но это небезопасно. При записи в массив по такой ссылке элемента не того типа CLR кидает `ArrayTypeMismatchException` в runtime — проверка идёт на каждую запись. Это историческое наследие до-generic эпохи. Generic-вариантность сделана иначе: разрешена только в статически безопасных позициях, без runtime-сюрприза.

**Value-типы (Q5).**

> Нельзя. Вариантность работает только для reference-conversion, где представление не меняется. `int` → `object` — это boxing, конверсия со сменой представления, поэтому `IEnumerable<int>` → `IEnumerable<object>` не проходит. Value-типовые аргументы всегда инвариантны.

**Declaration vs use site (Q6).**

> В C# вариантность — declaration-site: автор типа один раз пишет `in`/`out` на тип-параметре, и все использования вариантны. В Java — use-site: на типе вариантности нет, её указывает потребитель на каждом месте через wildcards (`? extends` / `? super`). Use-site гибче, но многословнее.
