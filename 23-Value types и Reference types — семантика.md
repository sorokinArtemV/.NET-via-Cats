## Глава: Value types и Reference types — семантика

Это самая фундаментальная развилка в системе типов C#. От того, value type перед тобой или reference type, зависит почти всё наблюдаемое поведение переменной: что произойдёт при присваивании, что увидит метод, в который ты её передал, может ли она быть `null` и как две переменные сравниваются на равенство. Бóльшая часть «необъяснимых» багов и почти все ловушки на собесе по этой теме растут из одного корня — из непонимания этой развилки.

И сразу договоримся о главном разделении, которое отличает эту главу от следующей. Здесь мы говорим про **семантику** — про контракт типа: как он копируется, как себя ведёт, как сравнивается. Про то, **где** он физически лежит (stack/heap), — отдельная глава дальше. Это разделение принципиально: фраза «value type живёт в стеке» — самый частый неверный ответ на собесе, и рождается он ровно потому, что «какой это тип» и «где он лежит» склеивают в одно. Мы разводим их намеренно: сначала — поведение, потом — память.

### Mental model

**Value type — переменная _владеет_ значением. Reference type — переменная _держит ссылку_ на значение, которым могут владеть многие.**

Отсюда вытекает всё остальное одной фразой: присваивание (и передача в метод) **копирует переменную**. У value type копия переменной — это копия самих данных, поэтому копии независимы. У reference type копия переменной — это копия ссылки, поэтому обе ссылки указывают на один объект.

### Как это работает

#### Два семейства

**Value types** — `struct` и `enum`, а значит все примитивы (`int`, `double`, `bool`, `char`, `byte`…), а также `decimal`, `DateTime`, `TimeSpan`, `Guid`, `Nullable<T>`, `ValueTuple`, `record struct`. Все примитивы — это struct'ы под капотом (`int` это `System.Int32`).

**Reference types** — `class`, `interface`, `delegate`, массив (любой, даже `int[]`), `string`, `object`, `dynamic`, `record` (по умолчанию это `record class`), анонимные типы.

Две ловушки в этом списке: массив `int[]` — **reference type**, хотя элементы внутри него value; а `record` без слова `struct` — тоже reference type.

#### Семантика присваивания

![[Pasted image 20260629115440.png]]


```csharp
int a = 5;
int b = a;     // копируются биты: b — независимая «5»
b = 9;         // a по-прежнему 5

var x = new StringBuilder("hi");
var y = x;     // копируется ссылка: x и y смотрят на один объект
y.Append("!"); // x.ToString() теперь тоже "hi!"
```

Reference type копирует **ссылку, а не объект**. Никакого глубокого копирования не происходит — это частая ошибка в голове новичка, который думает, что `y = x` создаёт второй объект.

#### Передача в метод

По умолчанию в C# **всё передаётся по значению** — включая reference types. Но «значение» reference type — это сама ссылка. Поэтому:

```csharp
void Mutate(StringBuilder sb) => sb.Append("!");  // меняем ОБЪЕКТ → видно снаружи
void Reassign(StringBuilder sb) => sb = new();     // меняем КОПИЮ ССЫЛКИ → снаружи НЕ видно
```

`Mutate` влияет на объект вызывающей стороны, потому что обе ссылки указывают на один объект. `Reassign` — нет: переприсваивание меняет только локальную копию ссылки. Чтобы переприсваивание было видно снаружи, нужен `ref` — это подробно в главе про `ref`/`out`/`in`.

Для value type метод всегда получает копию данных, и мутации внутри метода наружу не уходят (если опять же не `ref`/`out`).

#### `null` и `default(T)`

- **Reference type**: значение по умолчанию — `null`. Переменная может не указывать ни на какой объект.
- **Value type**: `null` невозможен без обёртки `Nullable<T>` (`int?`). `default(T)` — это **обнулённый экземпляр**: `default(int)` → `0`, `default(bool)` → `false`, `default(SomeStruct)` → struct со всеми полями в их default'ах.

Практический вывод: value type **никогда не выбросит `NullReferenceException` сам по себе**, потому что он не бывает null. А ещё — у struct всегда есть валидный «нулевой» экземпляр, даже если он бессмысленен по смыслу домена.

#### Identity vs equality (равенство по умолчанию)

Поведение «из коробки» различается, и это любимая зона follow-up'ов:

- **Reference type**: `Equals`/`==` по умолчанию сравнивают **по ссылке** (это один и тот же объект?), а не по содержимому. Два разных объекта с одинаковыми полями не равны, пока ты не переопределишь равенство.
- **Value type**: унаследованный `ValueType.Equals` сравнивает **по значению** — поле за полем. Две struct'ы с одинаковыми полями равны.

Важный нюанс, который проваливают: для **своей** struct оператор `==` по умолчанию **не компилируется** (`CS0019`) — его надо перегрузить вручную. Работает только `a.Equals(b)`. У примитивов `==` есть, потому что для них операторы определены. (`record` и `record struct` — исключение: компилятор генерирует и value-equality, и `==`.) Глубоко про корректную реализацию равенства — в отдельной главе про `Equals`/`GetHashCode`/`IEquatable<T>`.

#### 🔬 Глубже / на синьора

**Иерархия типов и где тут подвох.** Все value types наследуют `System.ValueType`, который наследует `System.Object`; enum'ы идут через `System.Enum : ValueType`. Сам `System.ValueType` — при этом **reference type** (абстрактный class). CLR специально обрабатывает наследников `ValueType` как value-семантику. Практическое следствие: раз value type — потомок `object`, его _можно_ присвоить в `object` или передать туда, где ждут `object`/интерфейс, — но это **boxing** (отдельная глава дальше). То есть «value type наследуется от object» — правда, но переход к object не бесплатен.

**Почему `ValueType.Equals` по умолчанию медленный.** Унаследованная реализация для struct, если в ней есть хоть одно поле-ссылка, сравнивает поля через **reflection** — отсюда заметный оверхед. Если struct «bit-blittable» (только value-поля без указателей), runtime может сравнить память напрямую, но полагаться на это нельзя. Вывод для синьора: для любой struct, которую кладёшь в коллекции или сравниваешь в горячем пути, реализуй `IEquatable<T>` и переопредели `Equals`/`GetHashCode` — это и корректнее, и убирает boxing с reflection. (Связка с главой про равенство.)

**`default(struct)` нельзя «починить» конструктором.** Начиная с C# 10 у struct можно объявить parameterless-конструктор и field initializers:

```csharp
struct Money { public decimal Amount = 0m; public string Currency = "USD"; public Money() {} }
```

`new Money()` вызовет этот конструктор. Но `default(Money)` и `new Money[10]` его **игнорируют** и просто обнуляют память (`Currency` станет `null`!). То есть гарантированно валидный дефолт у struct сделать невозможно — всегда есть путь получить обнулённый экземпляр в обход конструктора. Это аргумент, почему mutable-инварианты в struct ненадёжны (подробнее — в главе struct vs class).

**`string` — reference type, но ведёт себя «как значение».** Он immutable, а его `==` перегружен на value-equality (плюс интернирование литералов). Это сбивает с толку: люди записывают `string` в value types. Он reference type — просто с переопределённым равенством и неизменяемостью. Запомни как отдельный случай.

### Interview layer

#### Вопросы по частоте

1. **«В чём разница между value type и reference type?»** — почти гарантированный вопрос. Проверяют, скажешь ли ты про _семантику копирования_, а не только про «стек/куча».
2. **«Что произойдёт при присваивании / передаче в метод?»** — основной разбор.
3. **«Может ли value type быть `null`? А что такое `default`?»**
4. **«Как сравниваются на равенство по умолчанию?»** — переход к Equals/GetHashCode.
5. (синьор) **«`string` — value или reference? Почему он тогда ведёт себя как значение?»**

#### Follow-up tree

> **Q:** Чем отличаются value и reference types?  
> **→ likely:** Что будет при `b = a; b.X = 9;` для каждого?  
> **→ trap:** «А `var y = x` для класса — это копия объекта?» (Нет — копия ссылки. Глубокого копирования нет.)

> **Q:** Передаётся ли reference type по ссылке?  
> **→ likely:** Тогда почему `sb = new()` внутри метода не виден снаружи?  
> **→ trap:** «Значит, чтобы переприсваивание стало видно — что нужно?» (`ref`. Различить _мутацию объекта_ и _переприсваивание ссылки_.)

> **Q:** Может ли struct быть `null`?  
> **→ likely:** А `default(SomeStruct)` — это что?  
> **→ trap:** «Если я в C# 10 напишу parameterless-конструктор struct, `default` его вызовет?» (Нет — `default` и массивы обнуляют в обход конструктора.)

#### Частые поверхностные ответы и почему проваливаются

- **«Value types лежат в стеке, reference — в куче».** Неполно и в общем случае **неверно**: место зависит от контекста (поле в объекте, элемент массива, замыкание, boxing). Семантику копирования это вообще не объясняет. Подменяет «как ведёт себя» на «где лежит». Разбор — в следующей главе.
- **«Reference type передаётся по ссылке».** Неточно. По умолчанию всё передаётся **по значению**; для reference type значение — это ссылка. Поэтому _мутация объекта_ видна снаружи, а _переприсваивание_ — нет. Тот, кто говорит «по ссылке», обычно не объяснит разницу между этими двумя случаями и попадётся на `ref`.
- **«`var y = x` копирует объект».** Нет — копируется ссылка. Объект остаётся один.
- **«Две struct'ы с одинаковыми полями — это один объект».** Путает equality с identity. Они равны по значению, но это две независимые копии, а не один объект.

#### Model answer (mid+)

**RU.** В C# типы делятся на value и reference по семантике копирования. Value type (`struct`, `enum`, примитивы) при присваивании и передаче в метод копируется по значению — копии независимы, мутация одной не трогает другую. Reference type (`class`, массив, `string`, делегат) хранит ссылку; при присваивании копируется ссылка, а не объект, поэтому обе переменные указывают на один объект, и мутация видна через обе. Передаётся по умолчанию всё по значению: для reference type это значит, что мутация объекта видна снаружи, а переприсваивание ссылки — нет, для этого нужен `ref`. Value type не может быть `null` (только через `Nullable<T>`), его `default` — обнулённый экземпляр. По умолчанию reference types сравниваются по ссылке, value types — по значению поле за полем.

_Senior add-on._ Под капотом все value types наследуют `System.ValueType : object`, поэтому переход value type к `object`/интерфейсу возможен, но это boxing с аллокацией. И ещё: `default(T)` у struct обнуляет память в обход любого parameterless-конструктора (C# 10) — гарантированно валидный дефолт у struct недостижим.

**EN.** In C# types split into value and reference by copy semantics. A value type (`struct`, `enum`, primitives) is copied by value on assignment and when passed to a method — copies are independent, mutating one doesn't affect the other. A reference type (`class`, array, `string`, delegate) holds a reference; assignment copies the reference, not the object, so both variables point to the same object and a mutation is visible through both. Everything is passed by value by default: for a reference type that means mutating the object is visible to the caller, but reassigning the reference is not — you need `ref` for that. A value type can't be `null` (only via `Nullable<T>`); its `default` is a zeroed instance. By default reference types compare by reference, value types compare by value, field by field.

_Senior add-on._ Under the hood all value types derive from `System.ValueType : object`, so converting a value type to `object`/an interface is possible but is boxing with a heap allocation. Also, `default(T)` for a struct zeroes memory bypassing any parameterless constructor (C# 10) — you can't guarantee a valid struct default.
