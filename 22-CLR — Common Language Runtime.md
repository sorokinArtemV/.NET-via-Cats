CLR (Common Language Runtime) — managed runtime платформы .NET: виртуальная машина, которая грузит, исполняет и сопровождает код .NET-языков (C#, F#, VB). Знать её на mid+ нужно не ради вопроса «что такое CLR» в лоб — это junior-скрин, — а потому что на CLR висит пласт mid+ вопросов: что происходит при запуске программы, managed vs unmanaged, JIT vs AOT. Сама по себе она закрывает три проблемы: избавляет от ручного управления памятью (GC вместо `malloc`/`free`, утечек и use-after-free), даёт один переносимый формат кода (IL крутится на x64/arm64, Windows/Linux без перекомпиляции исходников) и держит type safety с контролируемым выполнением.

**Модельный ответ (одна фраза + перечень):** CLR — managed runtime .NET: грузит IL, джитит его в native и сопровождает выполнение кода. За что отвечает:
- **управление памятью и GC** — выделение объектов и автоматическое освобождение;
- **компиляция IL → native (JIT)** и оптимизация в рантайме;
- **type safety** — проверка типов и верификация IL;
- **обработка исключений** (структурная);
- **многопоточность** — thread pool, синхронизация;
- **безопасность и interop** с нативным кодом (P/Invoke, marshalling).

Если совсем коротко: _«CLR делает код managed — берёт на себя память, типобезопасность, JIT, потоки и исключения, чтобы я не делал это руками»._

### Mental model

**CLR — это managed runtime (виртуальная машина) .NET: она грузит IL, на лету компилирует его в машинный код через JIT и сопровождает выполнение — память (GC), проверка типов, исключения, потоки.** «Managed» значит, что код живёт под надзором runtime, а не общается с железом напрямую.

Фраза, с которой ведёшь ответ на собесе: _«CLR исполняет не мой C#, а IL, и превращает IL в native через JIT прямо на той машине, где код запускается»._

### Как работает: pipeline компиляции и исполнения

Путь от исходника до исполнения двухступенчатый, и это ключевое отличие от C/C++:

1. **Roslyn (csc)** компилирует `.cs` в **IL** (CIL/MSIL) + **metadata**, упакованные в **assembly** (`.dll`/`.exe`). IL — платформо-независимый байткод.
2. При запуске **CLR** грузит assembly, читает metadata и отдаёт IL **JIT**-компилятору.
3. **JIT** превращает IL в native — но не весь модуль сразу, а **по методам, лениво, в момент первого вызова**. Результат кешируется в памяти на время жизни процесса; повторные вызовы идут уже в native.

![[Pasted image 20260625162349.png]]

Ключевой нюанс, который путают: один и тот же IL крутится на x64 и arm64, под Windows и Linux — без перекомпиляции исходников. Платформо-зависимым код становится только в момент JIT.

**Metadata** — это не довесок, а вторая половина assembly: описание всех типов, членов, сигнатур и атрибутов. Именно отсюда работают reflection и проверка типов в рантайме.

> **🔬 Глубже:** JIT компилирует **по методам и лениво** — не весь модуль при старте. Первый вызов каждого метода оплачивается JIT-компиляцией; это заметная часть cold start приложения. Дальше метод исполняется из native-кеша. Отсюда и берётся весь зоопарк оптимизаций старта — R2R, tiered compilation, AOT (см. блок JIT vs AOT ниже).

### Managed vs unmanaged code

||Managed|Unmanaged|
|---|---|---|
|Где исполняется|под надзором CLR|напрямую как native|
|Память|GC, автоматически|вручную (C/C++, malloc/free)|
|Type safety|verifiable IL, проверки|нет гарантий от runtime|
|Примеры|твой C#-код|OS API, нативные C-библиотеки|

Граница между мирами — **P/Invoke** и interop с marshalling параметров. Самый частый трюк на собесе здесь — `unsafe`:

**`unsafe` ≠ unmanaged.** `unsafe`-код компилируется в тот же managed assembly и крутится под CLR. Просто внутри блока разрешены указатели и адресная арифметика — отключены _некоторые проверки безопасности_, но не сам managed-надзор. Unmanaged code — это вообще отдельный native-модуль за пределами CLR. `unsafe` остаётся managed.

> **🔬 Глубже:** managed heap чистит GC (→ глава **GC**). А вот unmanaged-ресурсы — file handles, sockets, native-память от P/Invoke — GC сам по себе не освобождает вовремя: он не знает их «веса» и не понимает их семантику. Поэтому их закрывают детерминированно через `IDisposable`/`using`, а finalizer оставляют как страховку (→ глава про **IDisposable/finalizers**).

### JIT vs AOT

Это самый «mid+» подраздел главы — здесь интервьюер реально копает.

**JIT (по умолчанию):** IL → native в рантайме, по требованию.

- Плюс: компилирует под _конкретный_ CPU машины запуска и адаптируется к реальному поведению программы (tiered compilation, Dynamic PGO).
- Минус: cold start (каждый метод джитится при первом вызове), в рантайме нужны и IL, и сам JIT.

**AOT (NativeAOT):** IL → native заранее, на этапе `publish`.

- Результат — самодостаточный native-бинарь: **без JIT и без необходимости установленного .NET runtime** на целевой машине.
- Плюс: почти мгновенный старт, меньше памяти, меньше размер, работает там, где JIT запрещён.
- Минус: нет рантайм-адаптивности (теряешь часть peak throughput, который даёт PGO), ограничения на reflection и runtime codegen (`Reflection.Emit`, `MakeGenericType` по неизвестному типу), бинарь привязан к конкретной платформе.

**Когда что:**

- **JIT** — долгоживущие серверные приложения, где важен steady-state throughput, а разовый cold start не критичен.
- **AOT** — serverless/CLI/контейнеры с тысячами инстансов, где cold start и память решают.

Версии (проверено по docs, июнь 2026):

- **NativeAOT**: console-приложения — с **.NET 7**; ASP.NET Core (minimal API) — с **.NET 8**.
- **ReadyToRun (R2R)** — промежуточный вариант: часть кода прекомпилируется в native заранее, но JIT остаётся доступен в рантайме и может переджитить горячие методы. Bias на быстрый старт; «полу-AOT».

> **🔬 Глубже — tiered compilation (default с .NET Core 3.0):**  
> JIT компилирует метод не один раз, а в несколько ступеней.
> 
> - **Tier 0 (Quick JIT)** — быстрая компиляция почти без оптимизаций. Цель — быстро стартовать.
> - **Tier 1** — фоновая перекомпиляция «горячих» методов (по счётчику вызовов) в полностью оптимизированный native, который подменяет Tier 0.
> - **OSR (on-stack replacement, .NET 7)** — позволяет заменить уже _выполняющийся_ долгий цикл на оптимизированную версию, не дожидаясь выхода из метода. До него длинные циклы могли застрять в Tier 0.
> - **Dynamic PGO (on by default с .NET 8)** — Tier 0 инструментируется и собирает профиль (какие типы и ветки реально горячие); Tier 1 использует его для devirtualization виртуальных вызовов и других оптимизаций под фактическое поведение. Это и есть тот «адаптивный» козырь JIT, которого AOT лишён.
> - R2R-код тоже участвует: tiered compilation может переджитить precompiled-методы в более качественный Tier 1.

### CTS / CLS / metadata

Короткий блок — на mid+ глубоко не спрашивают, достаточно знать назначение.

- **CTS (Common Type System)** — единая система типов рантайма: как описываются value/reference типы, что такое `class`, `struct`, `interface`, `delegate`. Благодаря ей типы из C#, F#, VB понимают друг друга.
- **CLS (Common Language Specification)** — подмножество правил для _публичного_ API библиотеки, чтобы он был совместим со всеми .NET-языками (например, не полагаться на различие имён только по регистру или на типы, которых нет в части языков).
- **Metadata** — самоописание сборки (типы, члены, сигнатуры, атрибуты); основа reflection и type safety.

### Interview layer

#### Q1. «Что происходит, когда запускается C#-программа? Как C# превращается в исполняемый код?» — _частый, junior–mid_

Дерево:

- → likely: «В машинный код компилирует C#-компилятор?» — **trap**: нет. `csc`/Roslyn даёт **IL**, а native делает **JIT** в рантайме.
- → trap: «JIT компилирует всю программу при старте?» — нет, **по методам, лениво, при первом вызове**.

**Поверхностный ответ и почему он плохой:** _«C# компилируется в машинный код»_ — путает C# с C/C++ и выкидывает весь слой IL + JIT, на котором держится переносимость и адаптивная оптимизация. Интервьюер сразу видит, что человек не понимает двухступенчатую модель.

**Модельный ответ:** Roslyn компилирует C# в IL + metadata, упакованные в assembly. При запуске CLR грузит assembly и отдаёт IL JIT-компилятору, который превращает IL в машинный код под конкретный CPU — лениво, по методам, в момент первого вызова; результат кешируется. То есть выполняется не C# напрямую, а native, полученный из IL уже на машине запуска. _(senior add-on: первый вызов метода джитится Tier 0 / Quick JIT, горячие методы фоном перекомпилируются в оптимизированный Tier 1, с .NET 8 ещё и с Dynamic PGO.)_

#### Q2. «Managed vs unmanaged code?» — _mid_

Дерево:

- → likely: «`unsafe`-код — это unmanaged?» — **trap**: нет, это managed assembly с отключёнными проверками, всё ещё под CLR.
- → trap: «Кто освобождает unmanaged-ресурсы?» — не GC напрямую; детерминированно через `IDisposable`/`using`, finalizer — страховка.

**Поверхностный ответ и почему он плохой:** _«managed — это C#, unmanaged — это C++»_ — формально не врёт, но это про языки, а вопрос про **runtime**. Не названо главное: managed = под надзором CLR (GC, type safety, verifiable IL).

**Модельный ответ:** Managed code исполняется под CLR — с GC, проверкой типов и verifiable IL; память освобождается автоматически. Unmanaged code — это native вне CLR (OS API, C/C++-библиотеки), где за память отвечаешь сам. Граница между ними — P/Invoke с marshalling. Важно: `unsafe` — это не unmanaged, а тот же managed-код под CLR, где локально разрешены указатели и адресная арифметика.

#### Q3. «JIT vs AOT? Когда выбрать NativeAOT?» — _mid+, актуальный_

Дерево:

- → likely: «Что теряешь с AOT?» — рантайм-оптимизации (Dynamic PGO), полноценный reflection/runtime codegen, часть peak throughput.
- → trap: «AOT всегда быстрее JIT?» — нет. **Старт** быстрее, но в **steady-state** JIT с PGO может обгонять AOT, потому что оптимизирует под фактическое поведение.

**Поверхностный ответ и почему он плохой:** _«AOT быстрее, потому что компилирует в native»_ — игнорирует, что JIT тоже выдаёт native, да ещё и адаптивный. Путает «быстрый старт» с «быстрее вообще».

**Модельный ответ:** JIT компилирует IL в native в рантайме, по методам — это даёт оптимизацию под конкретный CPU и адаптивность (tiered, Dynamic PGO с .NET 8), ценой cold start. NativeAOT (console — с .NET 7, ASP.NET Core — с .NET 8) компилирует в native заранее, на publish: получаем самодостаточный бинарь без JIT и без установленного runtime, с мгновенным стартом и меньшим потреблением памяти. Платой идут ограничения на reflection/runtime codegen и потеря части peak throughput. Поэтому JIT — для долгоживущих серверов, AOT — для serverless/CLI/контейнеров, где критичны cold start и память.

#### Q4. «Зачем нужна tiered compilation? Что такое Tier 0 и Tier 1?» — _mid+/senior_

**Модельный ответ:** Это компромисс между скоростью старта и пиковой производительностью. Tier 0 (Quick JIT) компилирует быстро и без оптимизаций — приложение стартует быстро. Горячие методы (по счётчику вызовов) рантайм фоном перекомпилирует в полностью оптимизированный Tier 1 и подменяет на него. Включено по умолчанию с .NET Core 3.0. _(senior add-on: OSR с .NET 7 заменяет даже выполняющийся длинный цикл; Dynamic PGO с .NET 8 инструментирует Tier 0 и кормит профилем Tier 1 для devirtualization.)_

#### Q5. «Что такое CLR / managed code?» — _junior-скрин, быстрый_

**Модельный ответ (одна фраза):** CLR — managed runtime .NET: грузит IL, джитит его в native и сопровождает выполнение — GC, type safety, исключения, потоки.
