## LINQ to Objects

У тебя в памяти коллекция — список заказов, массив пользователей — и нужно её отфильтровать, преобразовать, отсортировать, сгруппировать, посчитать сумму. Можно каждый раз писать `foreach` с аккумулятором и временными списками, но это шумно, плохо читается и плохо комбинируется: каждый новый вопрос к данным — новый цикл.

LINQ даёт единый, composable, типобезопасный словарь запросов над чем угодно, что реализует `IEnumerable<T>`. Один и тот же `Where`/`Select`/`GroupBy` работает поверх `List`, массива, результата другого запроса — источник не важен. **LINQ to Objects** — это конкретно та часть LINQ, что работает с in-memory коллекциями: набор extension-методов из `System.Linq.Enumerable` поверх `IEnumerable<T>`, выполняемых в процессе через обычные делегаты. (Вариант для удалённых источников — `IQueryable`, трансляция в SQL и т.п. — это отдельная история и следующая глава; здесь его не трогаем.)

### Ментальная модель

LINQ-запрос — это **рецепт, а не результат**: ленивая цепочка итераторов. Пока кто-то не начнёт перебор, не выполняется ничего; начал перебор дважды — выполнилось дважды. Источник при этом никогда не мутируется — операторы возвращают новые последовательности.

### Как это работает

**Два синтаксиса, один смысл.** Query-синтаксис (`from … where … select …`) — это синтаксический сахар, который компилятор разворачивает в те же вызовы методов. Method-синтаксис (цепочка `.Where().Select()`) полнее: не все операторы выразимы в query-форме (`ToList`, `Count`, `Skip`/`Take`, `Distinct` ключевых слов не имеют).

```csharp
// эквивалентны:
var q1 = from p in people where p.Age > 30 select p.Name;
var q2 = people.Where(p => p.Age > 30).Select(p => p.Name);
```

**Карта операторов** (мид+ должен держать в голове категории и то, _когда_ оператор выполняется):

|Категория|Операторы|Выполнение|
|---|---|---|
|Фильтрация|`Where`, `OfType`|deferred, streaming|
|Проекция|`Select`, `SelectMany`|deferred, streaming|
|Разбиение|`Take`, `Skip`, `TakeWhile`, `SkipWhile`, `Chunk`|deferred, streaming|
|Сортировка|`OrderBy(Descending)`, `ThenBy`, `Reverse`|deferred, **buffering**|
|Группировка|`GroupBy`|deferred, **buffering**|
||`ToLookup`|**immediate**|
|Соединение|`Join`, `GroupJoin`|deferred|
|Множества|`Distinct(By)`, `Union(By)`, `Intersect(By)`, `Except(By)`|deferred|
|Кванторы|`Any`, `All`, `Contains`|**immediate** (short-circuit)|
|Агрегация|`Count`, `Sum`, `Min/Max(By)`, `Average`, `Aggregate`|**immediate**|
|Элемент|`First/Single/Last(OrDefault)`, `ElementAt`|**immediate**|
|Материализация|`ToList`, `ToArray`, `ToDictionary`, `ToHashSet`|**immediate**|

**Deferred execution.** Операторы-конвейеры (`Where`, `Select`, `OrderBy`, …) при вызове не делают работу — они строят ленивую цепочку. Источник реально перебирается только при энумерации: `foreach` или материализатор (`ToList`, `Count`, `First`, …).

**Immediate execution (материализаторы).** Часть операторов форсирует перебор немедленно и возвращает готовый результат: `ToList`/`ToArray`/`ToDictionary`/`ToHashSet`, агрегаты (`Count`, `Sum`, `Any`, `First`…). Именно ими «фиксируют» результат запроса.

**Streaming vs buffering.** Тонкость, на которой валятся: _отложенный ≠ потоковый_. `Where`/`Select`/`Take` отдают элементы по одному — первый результат появляется сразу. А `OrderBy`/`GroupBy`/`Reverse` тоже отложенные, но при первом же переборе **буферизуют весь источник**, прежде чем отдать первый элемент (отсортировать нельзя, не прочитав всё).

**Multiple enumeration — главный footgun.** Раз запрос — рецепт, каждый новый перебор выполняет его заново: пайплайн прогоняется ещё раз, источник запрашивается ещё раз. Плюс захваченные переменные вычисляются в момент перебора, а не объявления:

```csharp
int threshold = 30;
var q = people.Where(p => p.Age > threshold);  // рецепт; threshold ещё не «вшит»
threshold = 40;
var a = q.ToList();   // фильтр по 40 — захвачена переменная, вычислена при переборе
var b = q.Count();    // источник перебран ЕЩЁ раз, с нуля
```

Если источник дорогой (запрос к БД, чтение файла, генератор с побочкой) — повторный перебор бьёт по производительности и может дать разные результаты. Лечение: материализовать один раз (`var list = q.ToList();`) и дальше работать со списком.

**Семейство element-операторов** — частый предмет вопросов, важны точные семантики исключений (тип — `InvalidOperationException`):

|Метод|Если пусто / нет совпадений|Если подходящих > 1|
|---|---|---|
|`First()`|бросает `InvalidOperationException`|берёт первый, не проверяет дальше|
|`FirstOrDefault()`|возвращает `default(T)`|берёт первый|
|`Single()`|бросает `InvalidOperationException`|**бросает** `InvalidOperationException`|
|`SingleOrDefault()`|возвращает `default(T)`|**бросает** `InvalidOperationException`|
|`Last()` / `LastOrDefault()`|бросает / `default(T)`|берёт последний|
|`ElementAt(i)`|`ArgumentOutOfRangeException`, если индекс вне диапазона|—|

Ключевая мысль не про синтаксис: **`Single` — это утверждение об инварианте «ровно один».** Если данных оказалось больше одного, это баг, и ты хочешь упасть, а не молча взять первый. `First` говорит «дай любой первый, остальное не волнует». Выбор между ними — заявление о данных.

> 💡 С .NET 6 у `FirstOrDefault`/`SingleOrDefault`/`LastOrDefault` появилась перегрузка с явным значением по умолчанию: `list.FirstOrDefault(x => x.Id == id, fallback)`.

### 🔬 Глубже / на синьора

**`Count()` vs `Any()` для «есть ли элементы».** `Any()` короткозамкнут — тянет ровно один элемент и останавливается. `Count() > 0` в общем случае перебирает всю последовательность. Исключение: если источник реализует `ICollection`/`ICollection<T>`, `Count()` берёт готовое свойство `.Count` за O(1). Для проверки «непусто» семантичнее и в худшем случае дешевле `Any()`.

> 🔬 С .NET 6 есть `TryGetNonEnumeratedCount(out int)` — пытается получить количество без перебора (если источник это позволяет) и возвращает `false`, если для подсчёта пришлось бы энумерировать.

**Под капотом — итератор.** Отложенность реализована через iterator-методы (`yield return`): компилятор генерирует state machine, которая отдаёт элементы по запросу `MoveNext()`. Это тот же механизм, что в пользовательских `IEnumerable<T>`-методах с `yield`. Детально — в будущей главе про итераторы/`yield`.

**`GroupBy` отложен, `ToLookup` — нет.** `GroupBy` строит группы лениво (при переборе), `ToLookup` выполняется немедленно и возвращает готовый `ILookup<TKey, TElement>`. Если нужна материализованная многоключевая «карта» для повторных обращений — `ToLookup`.

**Буферизация второго источника в set-операторах.** `Intersect`/`Except` сначала полностью читают вторую последовательность (строят `HashSet`), затем стримят первую. `Distinct`/`Union` стримят, попутно набивая set увиденного.

**Версии операторов (пины).** .NET 6: `Chunk`, `MinBy`/`MaxBy` (возвращают сам _элемент_ по ключу, а не значение ключа), `DistinctBy`/`ExceptBy`/`IntersectBy`/`UnionBy`, three-way `Zip`. .NET 9: `CountBy`, `AggregateBy` (группировка+агрегация без аллокации промежуточных `IGrouping`), `Index()` (отдаёт `(int Index, T Item)`). На собеседовании знание `MinBy` vs `Min` — хороший мид+-сигнал: `Min` даёт значение, `MinBy` — объект.

**Дерево follow-up'ов:**

- Q: что такое deferred execution? → likely: какие методы форсируют выполнение? → **trap:** что будет, если перебрать запрос дважды? а если источник — запрос к БД или чтение файла?
- Q: `First` vs `Single`? → likely: какой и когда бросает? → **trap:** что возьмёшь для «найти пользователя по Id» и почему? что дешевле по перебору? _(`Single` кодирует уникальность и читает ≥2 элемента ради проверки; `First` — «любой первый», читает 1)._
- Q: `Count` vs `Any`? → likely: какой короткозамкнут? → **trap:** а если источник — `List<T>`? _(`Count()` тогда O(1) через `ICollection`, но для «непусто» `Any()` семантичнее и не хуже)._

**Поверхностные ответы и почему проваливаются:**

- «deferred execution улучшает производительность» — лозунг, наполовину неверный. Сам по себе он не «ускоряет»: он откладывает работу и позволяет короткое замыкание (`First`/`Any`) и композицию без промежуточных коллекций — но та же ленивость порождает multiple enumeration, который замедляет. Нужно назвать механизм, а не эффект.
- «`ToList()` нужен, чтобы запрос выполнился» — неполно. Суть не «выполнить», а зафиксировать результат один раз: оторваться от источника и не платить за повторные переборы.
- «`Single` и `First` — почти одно и то же» — нет: `Single` падает на «больше одного», `First` — нет. Это разные утверждения о данных.
- «`OrderBy` ленивый, значит дёшев» — нет: при первом переборе он буферизует и сортирует весь источник, O(n log n) и полный проход никуда не делись.

**Модельный ответ (deferred execution):**

> Deferred execution значит, что оператор вроде `Where` или `Select` при вызове не выполняет работу, а строит ленивую цепочку итераторов. Реальный перебор источника происходит только при энумерации: `foreach` или материализатор (`ToList`/`Count`/`First`…). Практическое следствие — запрос это рецепт, а не результат: перебрал дважды — выполнился дважды и заново обратился к источнику; изменил между объявлением и перебором захваченную переменную или сам источник — изменился результат. Поэтому результат, который нужно переиспользовать, материализуют через `ToList()`. И не все отложенные операторы потоковые: `Where`/`Select` отдают по одному элементу, а `OrderBy`/`GroupBy`/`Reverse` при первом переборе буферизуют весь источник.
> 
> 🔹 _Senior add-on:_ `First()`/`Any()` короткозамкнуты и прекращают перебор рано; `Count()` на источнике-`ICollection` — O(1), иначе перебирает всё. Главный антипаттерн — multiple enumeration источника с побочкой или дорогим доступом (генератор, IO): материализуй один раз.
