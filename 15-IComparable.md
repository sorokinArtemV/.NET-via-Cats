## IComparable — упорядочивание в C#

### Зачем эта тема

Сортировка, бинарный поиск и упорядоченные коллекции (`SortedSet`, `SortedDictionary`) держатся на одном вопросе: «кто из двух объектов больше?». В C# есть два источника ответа — порядок, _вшитый в сам тип_ (естественный: числа по величине, строки по алфавиту), и порядок, _заданный снаружи_ (по конкретному полю, в обратную сторону, по своей логике). Глава про оба и про места, где порядок расходится с равенством — это парная тема к Equality, и интервьюер любит спросить, что будет, если `CompareTo` говорит «равны», а `Equals` — «нет».

### Mental model

- `IComparable<T>` — **естественный порядок, вшитый в тип**. Тип сам знает, как сравнить себя с другим: `int`, `string`, `DateTime` его реализуют.
- `IComparer<T>` — **порядок, заданный снаружи**, отдельным объектом-компаратором. Один тип → сколько угодно разных порядков.
- `CompareTo` возвращает **знак**, а не `-1/0/1`: `< 0` — текущий меньше, `0` — равны по порядку, `> 0` — больше.

### CompareTo и его контракт

```csharp
public interface IComparable<in T> { int CompareTo(T? other); }
```

Возвращается **знак**, не конкретное значение. `5.CompareTo(3)` может вернуть любое положительное число, не обязательно `1`. Опираться на точное значение — ошибка.

**Ловушка наивной реализации через вычитание.** Соблазн написать:

```csharp
public int CompareTo(MyInt other) => this.value - other.value; // БАГ
```

Для знака это «работает»… пока не переполнится. `int.MinValue - int.MaxValue` математически ≈ −4.3 млрд, не влезает в `int`, и в unchecked-контексте **переполняется в положительное число** — `CompareTo` вернёт «больше» там, где должно быть «меньше». Порядок ломается на краевых значениях. Правильно — делегировать встроенному компаратору:

```csharp
public int CompareTo(MyInt other) => this.value.CompareTo(other.value);
```

**Поведение с null.** По контракту любой объект **больше** `null`. Для ссылочного `T`: `x.CompareTo(null)` должен вернуть положительное число, а не бросать. `Comparer<T>.Default` это соблюдает: `null` «меньше» всего. И отдельное правило: из `CompareTo` вообще не принято бросать исключения — неспособность сравнить означает, что порядок не тотальный.

### Generic vs non-generic

Есть старый `IComparable` (метод `CompareTo(object)`) и `IComparable<T>` (метод `CompareTo(T)`). Разница не в «старое/новое», а в механике:

- `IComparable<T>` — **без boxing** для value-типов и **без каста**: компаратор получает `T` напрямую.
- `IComparable` (non-generic) принимает `object` → для структур boxing, плюс внутри нужна ручная проверка типа и бросок `ArgumentException` на чужой тип.

Non-generic не «мёртв»: он живёт в négeneric API и иногда нужен для совместимости. Но для своего типа реализуют `IComparable<T>` (часто оба, делегируя non-generic к generic).

### IComparer<T> — порядок снаружи

Когда нужен порядок, отличный от естественного (по другому полю, в обратную сторону, по своей логике), или порядок для типа, который ты не контролируешь:

```csharp
var cmp   = Comparer<int>.Default;                          // берёт IComparable<int> у int
var byLen = Comparer<string>.Create((a, b) =>               // инлайн-стратегия
                a.Length.CompareTo(b.Length));              // не a.Length - b.Length (overflow!)
people.Sort((a, b) => a.Age.CompareTo(b.Age));              // Comparison<T> делегат
```

Потребители: `List<T>.Sort`, `Array.Sort`, `OrderBy`/`OrderByDescending`, `List<T>.BinarySearch`, `SortedSet<T>`, `SortedDictionary<TKey,_>`, `SortedList<TKey,_>`.

### Согласованность с Equality — центр темы

Контракт: если `CompareTo` возвращает `0`, объекты считаются **равными по порядку**, и это _должно_ согласовываться с `Equals`. Рассогласование — источник тонких багов, потому что **упорядоченные коллекции определяют равенство через `CompareTo`, а не через `Equals`/`GetHashCode`**.

`SortedSet<T>` и `SortedDictionary<TKey,_>` используют **только компаратор**. Если `CompareTo` (или переданный `IComparer`) возвращает `0` для двух объектов, различных по `Equals`, — **второй не добавится**: для коллекции это «дубликат».

```csharp
// компаратор по Age; два разных человека одного возраста
var set = new SortedSet<Person>(
    Comparer<Person>.Create((a, b) => a.Age.CompareTo(b.Age)));
set.Add(new Person("Anna", 30));
set.Add(new Person("Bob",  30)); // НЕ добавится: CompareTo == 0 → «уже есть»
// set.Count == 1
```

Это парная ловушка к mutable-ключу из главы Equality: там `Dictionary` теряет ключ из-за хеша, тут `SortedSet` схлопывает «равные по порядку» элементы.

### Стабильность сортировки

- `List<T>.Sort` и `Array.Sort` — **introsort** (quicksort + heapsort + insertion sort), **нестабильны**: относительный порядок равных элементов _не_ сохраняется.
- LINQ `OrderBy`/`ThenBy` — **стабильны**: равные элементы сохраняют исходный порядок (поэтому `ThenBy` вообще имеет смысл).

Классический баг: отсортировал список по `LastName` через `List.Sort`, рассчитывая, что внутри одной фамилии сохранится прежний порядок по имени — а он съехал. Нужна стабильность — `OrderBy(...).ThenBy(...)` или вторичный ключ в компараторе.

||Что определяет|Где живёт|Метод|
|---|---|---|---|
|`IEquatable<T>`|равенство|внутри типа|`Equals(T)`|
|`IComparable<T>`|естественный порядок|внутри типа|`CompareTo(T)`|
|`IComparer<T>`|порядок-стратегия|снаружи|`Compare(T,T)`|

### 🔬 Глубже / на синьора

**Законы порядка.** `CompareTo` обязан задавать total order: антисимметрия (`a.CompareTo(b)` и `b.CompareTo(a)` — противоположные знаки), транзитивность (`a<b && b<c ⇒ a<c`), согласованность с нулём-равенством. Нарушишь (непоследовательный компаратор из-за гонки данных или кривой логики) — `Array.Sort`/`List.Sort` бросает `ArgumentException` «Unable to sort because the IComparer.Compare() method returns inconsistent results…» либо даёт мусорный порядок. Коварный нюанс: на коллекциях **≤16 элементов** баг часто не проявляется — на таком размере introsort идёт через insertion sort, который не запускает проверку согласованности; исключение всплывает только когда данные перерастают порог и включается основной алгоритм. «Работало на маленьких, упало на больших».

**`string.CompareTo` культурозависим — и расходится с `==`.** `string.CompareTo` (и `string.Compare` без `StringComparison`) делает **культурозависимое** сравнение по текущей культуре (`CurrentCulture`, через ICU), тогда как `==` и `string.Equals` без параметра — **ordinal** (побайтово). Следствия:

- Порядок сортировки строк зависит от локали машины и даже от ОС. Сортировка, «работавшая» на dev-машине, может дать другой порядок в проде — реальный источник багов (турецкая `i`, немецкая `ß`=`ss`).
- `string.CompareTo` нельзя сделать ordinal — для детерминированного порядка `string.Compare(a, b, StringComparison.Ordinal)` или `StringComparer.Ordinal`.

Полная картина сравнения строк — в будущей теме про строки.

**Как `Comparer<T>.Default` находит порядок.** Проверяет `IComparable<T>` → если есть, использует его; иначе non-generic `IComparable`; если ни одного — сравнивать нечем. Прямой вызов `Comparer<T>.Default.Compare(a, b)` бросает `ArgumentException` «At least one object must implement IComparable», а через `List.Sort`/`Array.Sort` это оборачивается в `InvalidOperationException` «Failed to compare two elements in the array» (исходный — во `InnerException`). Исключение возникает при первом сравнении, не при создании компаратора.

### Interview layer

**Вопросы по частоте:**

**1. Чем `IComparable<T>` отличается от `IComparer<T>`?**  
→ follow-up: когда какой?  
→ trap: можно ли задать несколько порядков для одного типа? (да — несколько `IComparer<T>`).  
Суть: `IComparable<T>` — естественный порядок внутри типа; `IComparer<T>` — стратегия снаружи, их может быть много.

**2. Что возвращает `CompareTo`?**  
→ follow-up: точно `-1/0/1`?  
→ trap: реализуй через `a - b` — что не так?  
Суть: знак, не конкретное значение; `a - b` переполняется на краевых int.

**3. Как связаны `CompareTo` и `Equals`?**  
→ follow-up: обязаны ли совпадать?  
→ trap: что будет в `SortedSet`, если `CompareTo == 0`, но `Equals == false`?  
Суть: `CompareTo==0` должно означать «равны»; `SortedSet`/`SortedDictionary` считают равенство через компаратор → «равные по порядку» схлопываются.

**4. Стабильна ли `List.Sort`?**  
→ follow-up: а `OrderBy`?  
→ trap: как получить стабильность на `List`?  
Суть: `List.Sort`/`Array.Sort` — introsort, нестабильны; `OrderBy` стабилен; для стабильности — `OrderBy().ThenBy()` или вторичный ключ.

**5. (senior)** Что будет, если компаратор непоследователен (нарушает транзитивность)? → `ArgumentException` «…inconsistent results…» или мусорный порядок; причём на ≤16 элементов часто не воспроизводится.

**6. (senior)** Почему сортировка строк отличается на разных машинах? → `CompareTo` культурозависим; для стабильного порядка — ordinal.

**Шаблонно-неверные ответы:**

- _«`CompareTo` возвращает −1, 0 или 1»_ — нет, знак; конкретное значение не специфицировано. Валит на реализации через вычитание.
- _«`IComparable` устарел, всегда generic»_ — generic предпочтителен, но дело в boxing/касте, а non-generic жив в négeneric API.
- _«`CompareTo == 0` и `Equals` — одно и то же»_ — не гарантировано; `SortedSet` показывает разницу на практике.
- _«Сортировка строк всегда одинакова»_ — нет, культурозависима по умолчанию.

**Model answer (mid+):**

> `IComparable<T>` задаёт естественный порядок, вшитый в сам тип, через `CompareTo`, возвращающий знак: меньше/равно/больше. Нужен другой порядок или порядок для чужого типа — реализую `IComparer<T>` или передаю `Comparer<T>.Create(...)`; одному типу так можно дать сколько угодно порядков. `CompareTo` никогда не пишу через вычитание `a - b` — оно переполняется на краевых значениях; делегирую встроенному `CompareTo`. Держу `CompareTo` согласованным с `Equals`: `CompareTo == 0` должно означать равенство, иначе `SortedSet` и `SortedDictionary`, которые определяют равенство через компаратор, молча схлопнут разные объекты. И помню, что `List.Sort`/`Array.Sort` нестабильны (introsort), а `OrderBy` стабилен — если важен порядок равных, беру `OrderBy().ThenBy()`.

> _(senior add-on):_ `string.CompareTo` культурозависим (`CurrentCulture`/ICU), в отличие от ordinal-семантики `==`, поэтому порядок строк может различаться между машинами и ОС; для детерминированного порядка использую `StringComparer.Ordinal`. Непоследовательный компаратор приводит к `ArgumentException` («inconsistent results») в `Array.Sort`, причём на маленьких коллекциях может не проявляться.
