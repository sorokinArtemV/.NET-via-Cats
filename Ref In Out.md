# Глава: ref / out / in

## Mental model
Все три модификатора передают в метод не значение и не объект, а **managed pointer на ячейку переменной** — алиас на её storage location. Различаются только контрактом доступа: `ref` — читать и писать, `out` — callee **обязан** записать, `in` — только читать (копии нет). По умолчанию C# передаёт аргументы **by-value**; эти три слова — три способа отказаться от копии.

## 1. Зачем это вообще нужно

По умолчанию аргумент передаётся **by-value** — копированием. Что именно копируется, зависит от типа:

- **value type** (`int`, `struct`) — копируются сами байты значения. Метод правит копию, оригинал caller не виден.
- **reference type** (`class`) — копируется **ссылка**, а не объект. Это источник вечной путаницы «в C# объекты передаются по ссылке» — **нет**. Передаётся *копия ссылки*. Через неё ты указываешь на тот же объект в куче (поэтому мутация полей видна обоим), но если внутри метода присвоить параметру новый объект — caller этого **не увидит**: переназначилась локальная копия ссылки, а не переменная caller.

```csharp
void Mutate(Person p)   => p.Name = "Bob";    // видно caller'у — тот же объект
void Reassign(Person p) => p = new Person();  // НЕ видно — переназначилась копия ссылки
```

> 🖼️ **SVG #1** — by-value vs by-ref (вставить рендер).
> 🖼️ **SVG #1b** — reference type by-value (вставить рендер).

Отсюда три вещи, которые by-value **не умеет**, и три модификатора как ответ на них:

| Боль | Решение |
|---|---|
| переназначить **саму переменную** caller (а не объект) / мутировать его value-type аргумент | `ref` |
| вернуть из метода **несколько результатов** через параметры | `out` |
| передать **большую структуру без копии**, но только на чтение | `in` |

---

## 2. ref — read+write alias

Контракт: переменная **должна быть инициализирована до вызова**, внутри метода её можно **читать и писать**, ключевое слово `ref` обязательно и на объявлении, и на вызове.

```csharp
void AddTen(ref int x) => x += 10;   // читает И пишет

int a = 5;
AddTen(ref a);   // a == 15
```

Ключевое, чего не видно на `int`: с `ref` для reference type ты можешь **переназначить переменную caller** на другой объект — не просто мутировать существующий:

```csharp
void Swap(ref string a, ref string b) => (a, b) = (b, a);

string x = "first", y = "second";
Swap(ref x, ref y);   // x == "second", y == "first" — переменные caller переназначены
```

Без `ref` `Swap` не работал бы: поменялись бы местами локальные копии ссылок, а `x`/`y` у caller остались бы прежними.

---

## 3. out — callee обязан заполнить

Контракт: инициализировать **до вызова не нужно**; внутри метода параметр считается неприсвоенным, поэтому **читать его до записи нельзя**; метод **обязан присвоить** значение на каждом пути нормального возврата (если метод бросает исключение — присваивать не обязан).

```csharp
void Init(out int x) => x = 42;   // должен записать до return

int a;            // не инициализирована — ок
Init(out a);      // a == 42
```

**Канонический сценарий — Try-pattern:**

```csharp
if (int.TryParse(s, out int value))
    Use(value);

dict.TryGetValue(key, out var found);   // out var — inline declaration, C# 7.0
int.TryParse(s, out _);                  // discard, если результат не нужен
```

«Зачем `out`, если есть кортежи?» — `out` старше `ValueTuple` (он с C# 1.0, value tuples — с C# 7.0) и закрепился в Try-API; `out var` встраивает declaration прямо в `if`. Важно не переврать аргумент: и `out`, и `ValueTuple` — **heap-free** (`ValueTuple` это `struct`), поэтому выбор между ними сегодня — в основном стилистика и совместимость с существующим API, а **не** экономия аллокаций. Без-аллокационное преимущество `out` держалось только против старого `Tuple<>` (reference type, C# 4.0).

---

## 4. in — read-only по ссылке (C# 7.2)

Контракт: передаётся **по ссылке, но только на чтение** — присвоить параметру внутри нельзя. Мотив — **большие `readonly struct`**, чтобы не копировать их при каждом вызове.

```csharp
readonly struct Matrix4x4 { /* 64 байта */ }

double Determinant(in Matrix4x4 m) { /* читает m, не копируя 64 байта */ }
```

Две особенности call site, которых нет у `ref`/`out`:
- ключевое слово `in` на вызове **необязательно** — `Determinant(m)` валиден;
- можно передать **rvalue** (литерал, результат выражения) — компилятор создаст временную переменную.

> ⚠️ На `int` `in` обычно бессмысленен: на x64 managed pointer — 8 байт, не дешевле копии 4-байтного `int` (размер указателя платформозависим). Выигрыш только на структурах заметного размера — и только при условии из 🔬-блока ниже.

## 🔬 Глубже / на синьора

Нить блока: **один механизм** (managed pointer) → **одна ловушка** (`in` defensive copy) → **три запрета** (property / overload / async).

**Единый механизм — managed pointer (byref).** В IL `ref`/`out`/`in` компилируются в один тип — managed pointer (`int32&`). Разница между ними чисто контрактная (правила definite assignment и read-only); на уровне CLR это один механизм. Этот же managed pointer лежит в основе `ref local`, `ref return`, `ref struct` и `Span<T>`.

**Managed vs unmanaged pointer.** `ref`/`out`/`in` — это *managed* pointer: он GC-aware (GC обновляет его, когда перемещает объект в куче), не поддерживает арифметику и ограничен по времени жизни (нельзя свободно сохранять на куче). Это не то же, что unmanaged `int*` из `unsafe`-кода, где за валидность адреса отвечаешь ты сам, а GC про него не знает.

**`in` и defensive copy — главная ловушка.** `in`-параметр — это *readonly* reference. Если тип — **не** `readonly struct`, а вызываемый внутри член (метод/property) **не** помечен `readonly`, компилятор не может гарантировать, что член не мутирует структуру, и вставляет **скрытую защитную копию** — на *каждый* такой вызов члена. Итог: `in` на не-readonly структуре с N вызовами членов делает N копий против одной копии у by-value — то есть может быть **медленнее** by-value.

| Тип структуры | Вызываемый член | Копия при вызове члена через `in`? |
|---|---|---|
| `readonly struct` | любой | нет |
| обычная `struct` | `readonly`-член | нет |
| обычная `struct` | не-`readonly`-член | **да, defensive copy** |

Лечение: делать структуру `readonly struct` либо помечать члены `readonly`.

**Property нельзя передать по `ref`/`out`/`in`.** Передавать можно только **variable** — local, field, array element, параметр, ref-возврат. Property — это пара методов get/set, у неё нет адресуемой ячейки, поэтому `Modify(ref obj.Prop)` не компилируется.

**Перегрузка только по `ref` vs `out` запрещена.** `M(ref int)` и `M(out int)` для системы сигнатур одинаковы → ошибка **CS0663**. При этом `M(int)` и `M(ref int)` различимы и сосуществуют.

**Нельзя в `async` и в итераторах (`yield`).** ref/out/in параметры там недопустимы. Причина: их аргументы переживают точку приостановки (`await`/`yield`), а состояние сохраняется в heap-объекте state machine; managed pointer может указывать на стек, которого к моменту возобновления уже нет, — хранить его на куче небезопасно.

**`ref local` / `ref return` (C# 7.0)** — расширение той же идеи на локали и возвраты (`ref int slot = ref array[i];`). См. главу _VERIFY: уточнить номер/название в chapter map_.

**Version pinning:** `ref`/`out` — C# 1.0 · `out var` / discard `out _` — C# 7.0 · `in` + `readonly struct` + `ref readonly` returns — C# 7.2 · `ref readonly` **параметры** — C# 12.

---
**Диаграмма 2 — reference type by-value** (закрывает главную ловушку «объекты передаются по ссылке»): обе переменные `p` — копии одной ссылки, поэтому указывают на тот же объект в куче; мутация поля видна обоим, а reseat наружу не пробивается.

## Interview layer

### Вопросы по частоте / seniority

1. **(junior/mid, очень часто)** В чём разница `ref`/`out`/`in`?
   - → когда `out` вместо `return`/кортежа?
   - **trap:** что значит «передать reference type by value»?
2. **(mid)** Передал объект в метод — можно его изменить? переназначить?
   - → как переназначить именно переменную caller? (`ref`)
   - **trap:** разница между мутацией объекта и reseat переменной.
3. **(mid)** Зачем `out`, если есть кортежи?
   - **trap:** в каком scope живёт `out var`?
4. **(senior)** `in` всегда избегает копирования?
   - → как гарантировать отсутствие копии?
   - **trap:** почему `in` на не-readonly структуре может быть **медленнее** by-value?
5. **(mid)** Почему нельзя передать property по `ref`?
6. **(mid)** Можно перегрузить метод только по `ref` vs `out`?
7. **(senior)** Почему ref/out нельзя в `async`/итераторе?
8. **(senior)** `ref` — это указатель? Чем managed pointer отличается от `int*`?

### Частые поверхностные/неверные ответы
- **«Объекты в C# передаются по ссылке.»** — Неверно. By-value передаётся *копия ссылки*; reseat переменной caller без `ref` невозможен.
- **«`in` — как `ref`, только read-only, и всегда быстрее для структур.»** — Игнорирует defensive copy: на не-`readonly struct` `in` может быть медленнее by-value.
- **«`out` нужен, потому что не аллоцирует в отличие от кортежа.»** — Неверно для `ValueTuple` (struct, heap-free). Реальные причины — возраст API и эргономика `out var`.

### Model-ответы

**Q1. Разница ref/out/in.**
Все три — это managed pointer на ячейку переменной, то есть передача по ссылке, а не копия. `ref` требует инициализации до вызова и разрешает чтение и запись. `out` инициализации не требует, но метод обязан присвоить значение перед возвратом — типичный кейс Try-pattern. `in` (с C# 7.2) передаёт по ссылке только для чтения, чтобы не копировать большие структуры.
*Senior add-on:* на уровне IL это один byref-pointer; разница — только контракт definite assignment и read-only, а у `in` ещё и риск скрытой defensive copy.

**Q2. Можно ли изменить переданный объект.**
Изменить *поля* объекта — да: by-value копируется ссылка, но указывает она на тот же объект в куче, поэтому мутация видна caller. А вот *переназначить* саму переменную caller на новый объект by-value нельзя — переназначится локальная копия ссылки. Для reseat нужен `ref`.
*Senior add-on:* отсюда классический миф «объекты передаются по ссылке» — путают copy-of-reference с pass-by-reference.

**Q3. Зачем out, если есть кортежи.**
`out` старше `ValueTuple` (C# 1.0 против 7.0) и закрепился в Try-API; `out var` позволяет inline-declaration прямо в условии (`if (int.TryParse(s, out var v))`). По аллокациям разницы нет — `ValueTuple` тоже struct, оба heap-free; так что в новом коде выбор скорее стилистический. Trap: `out var v`, объявленная в `if`, видна и во **внешнем** блоке после `if`, но definitely assigned там только на успешной ветке.

**Q4. in всегда избегает копий.**
Нет. `in` — это readonly reference. Если тип — не `readonly struct`, а вызываемый внутри член не помечен `readonly`, компилятор вставляет защитную копию на каждый вызов члена, чтобы не нарушить read-only контракт. В результате на не-readonly структуре `in` может быть медленнее by-value. Гарантия отсутствия копий — `readonly struct` или `readonly`-члены.

**Q5. Почему нельзя ref на property.**
По ссылке передаётся только variable — local, field, array element. Property — это методы get/set, у неё нет адресуемой ячейки, поэтому managed pointer на неё построить нельзя.

**Q6. Перегрузка ref vs out.**
Нельзя: `M(ref int)` и `M(out int)` для системы сигнатур идентичны — ошибка CS0663. А `M(int)` и `M(ref int)` различимы.

**Q7. Почему нельзя в async/итераторе.**
Аргумент по ссылке переживает точку приостановки, а состояние метода сохраняется в heap-объекте state machine. Managed pointer может указывать на стековую ячейку, которой к моменту возобновления уже нет, — хранить его на куче небезопасно, поэтому компилятор запрещает ref/out/in в таких методах.

**Q8. ref vs указатель.**
`ref` — это managed pointer: GC знает о нём и обновляет, когда перемещает объект; арифметики нет, время жизни ограничено. Unmanaged `int*` из `unsafe` GC не отслеживает — за валидность адреса (и за пиннинг при необходимости) отвечает разработчик.
