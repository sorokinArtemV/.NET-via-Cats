Ты наверняка уже писал лямбды — короткие функции вроде `u => u.Age > 18`. Привычная картина в голове: лямбда — это маленькая функция, которую можно вызвать. Но есть деталь, мимо которой почти все проходят, пока однажды не упрутся.

Возьми один и тот же фрагмент `.Where(u => u.Age > 18)` и напиши его сначала над `List<User>`, а потом над таблицей в базе через EF. Поведёт он себя совершенно по-разному. В первом случае C# честно перебирает список в памяти. Во втором — каким-то образом превращает твою лямбду в SQL `WHERE Age > 18` и фильтрует уже на стороне сервера базы. Вопрос, который тут стоит себе задать: **откуда база узнала, что у тебя внутри лямбды?** Ты ведь не писал никакого SQL — ты написал C#.

Ответ — expression trees, и это, пожалуй, самый «магический» механизм языка, ровно до того момента, пока не разберёшься, как он устроен. Идея вот в чём. Обычно лямбда компилируется в **исполняемый код**: получается функция, которую можно запустить, но нельзя «вскрыть» и посмотреть, что у неё внутри — это чёрный ящик из IL. Expression tree — это другой режим компиляции той же лямбды. Вместо исполняемой функции компилятор строит её **описание**: структуру данных, которая буквально говорит «здесь операция „больше“, слева — обращение к свойству `Age`, справа — число `18`». А описание, в отличие от чёрного ящика, можно прочитать, разобрать по узлам и перевести во что угодно — в том числе в тот самый SQL.

Если совсем коротко: **expression tree превращает код в данные.** И как только код становится данными, открываются ровно две возможности, ради которых всё и затевалось.

Первая — **отдать код как данные другому исполнителю**. Именно это происходит в EF: он получает не выполнимую функцию, а дерево, обходит его и собирает SQL-запрос. Если бы ему достался обычный делегат (как в случае с `List`), внутри был бы непрозрачный IL, и перевести его в SQL было бы невозможно. Поэтому `IQueryable` принимает `Expression<Func<…>>`, а `IEnumerable` — обычный `Func<…>`; это и есть техническая причина «магии» (подробный разбор — в главе **IEnumerable vs IQueryable**).

Вторая — **сгенерировать код прямо во время выполнения**. Дерево можно собрать руками, а потом одной командой превратить в настоящий, быстрый делегат. Так делают высокопроизводительные мапперы и сериализаторы, которым нужно обойти рефлексию по скорости: они один раз строят дерево «как достать свойство», компилируют его и дальше зовут как обычную функцию.

Эта глава завершает кластер **Reflection → Attributes → Expression Trees** и заодно расставляет по местам сравнение Reflection vs Expression Trees, которое постоянно путают (про это — в конце).

### Mental model

**Expression tree — это код в виде данных: объектный граф, который описывает выражение, но сам не исполняется.** Один и тот же текст лямбды компилируется в две принципиально разные вещи в зависимости от целевого типа: `Func<…>` — исполняемый делегат (чёрный ящик), `Expression<Func<…>>` — дерево, которое можно либо обойти и перевести во что-то (SQL), либо скомпилировать в делегат.

### Как это работает (mid+ core)

#### Делегат vs `Expression<TDelegate>` — что именно генерирует компилятор

Сравним две строчки, которые отличаются только типом переменной слева:

```csharp
Func<Person, bool> f = p => p.Age > 18;             // обычная функция: f(person) работает
Expression<Person, bool> e = p => p.Age > 18;        // ⚠ так не пишут — см. ниже
Expression<Func<Person, bool>> e = p => p.Age > 18; // дерево: e(person) НЕ работает
bool ok = e.Compile()(person);                       // сначала Compile() в делегат, потом вызов
```

Для первой строки компилятор генерирует тело метода — и ты можешь вызвать `f` сразу. Для строки с `Expression<…>` он генерирует **код, который во время выполнения соберёт дерево** из узлов. То есть `e` — это не функция, а данные о функции; вызвать её напрямую нельзя, нужно сначала «оживить» через `Compile()` либо отдать тому, кто умеет дерево читать.

#### Из чего состоит дерево

Дерево — это вложенные объекты-узлы. У каждого есть `NodeType` (что это за узел: `GreaterThan`, `Constant`, `Call`…) и `Type` (тип результата). Узлы **неизменяемы**. Корень лямбды — `LambdaExpression`, у него есть список `Parameters` и `Body` (само выражение). Дальше тело раскладывается на под-узлы. Чтобы это перестало быть абстракцией, разберём по косточкам нашу `p => p.Age > 18`:

![[Pasted image 20260630175529.png]]

Прочитать диаграмму стоит снизу вверх: «параметр `p` типа `Person` → берём у него свойство `Age` → сравниваем „больше“ с константой `18`». Ровно это EF и делает, только переводит каждый узел в кусочек SQL: `Age > 18`.

#### Как дерево читают — `ExpressionVisitor`

«Обойти дерево» — значит пройтись по всем узлам и что-то с каждым сделать (перевести в SQL, посчитать, переписать). Чтобы не писать ручной рекурсивный обход, в .NET есть готовый помощник — `ExpressionVisitor`. Ты наследуешься от него и переопределяешь только те `Visit*`-методы, которые тебе интересны; остальное он обойдёт сам. Поскольку узлы неизменяемы, визитор ничего не правит на месте, а **строит новое дерево**:

```csharp
// Заменяет в дереве все обращения к константе 18 на параметр t
sealed class ConstToParam : ExpressionVisitor
{
    public ParameterExpression Threshold { get; } = Expression.Parameter(typeof(int), "t");
    protected override Expression VisitConstant(ConstantExpression node)
        => node.Value is 18 ? Threshold : base.VisitConstant(node);
}
```

Под капотом любой LINQ-провайдер — это, по сути, большой специализированный визитор, который вместо замены узлов выписывает SQL.

#### Собрать дерево руками

До сих пор дерево нам строил компилятор из лямбды. Но его можно собрать и **вручную** — это второй сценарий из вступления, генерация кода в рантайме. Узлы создаются фабриками `Expression.*`, а в конце дерево превращается в настоящий делегат:

```csharp
var p = Expression.Parameter(typeof(Person), "p");           // p
Expression body = Expression.GreaterThan(                     // ... > ...
    Expression.Property(p, nameof(Person.Age)),               // p.Age
    Expression.Constant(18));                                 // 18
var lambda = Expression.Lambda<Func<Person, bool>>(body, p);  // p => p.Age > 18

Func<Person, bool> isAdult = lambda.Compile();  // компилируем в делегат
bool ok = isAdult(person);                       // вызываем как обычную функцию
```

Этот код собирает ровно то дерево, что на диаграмме выше, и компилирует его. Так строят, например, динамический getter свойства один раз — и дальше зовут как обычную функцию.

#### `Compile()` и его цена

`Compile()` «оживляет» дерево — превращает его в исполняемый делегат. Здесь важно сразу усвоить экономику, иначе легко сделать хуже, чем без деревьев: **компиляция дорогая**. Она строит IL и прогоняет его через JIT — это на порядки дороже одного вызова функции. Вывод простой: Expression Trees выгодны только по схеме «собрал и скомпилировал **один раз** → вызываешь делегат **много раз**». Если компилировать на каждый вызов, выйдет медленнее рефлексии. Поэтому скомпилированный делегат всегда **кэшируют** (обычно по ключу — типу или члену). Внутренности компиляции — в 🔬.

#### Почему ORM нужно дерево, а не делегат

Вернёмся к загадке из вступления — теперь на неё есть точный ответ:

```csharp
IEnumerable<User> a = users.Where(u => u.Age > 18);    // Func<>: фильтр крутится в памяти
IQueryable<User>  b = db.Users.Where(u => u.Age > 18); // Expression<>: дерево → SQL WHERE
```

`IEnumerable.Where` принимает делегат и выполняет его в процессе, по каждому элементу. `IQueryable.Where` принимает `Expression<Func<…>>`, и провайдер **читает дерево** и собирает SQL. Получи он делегат — внутри был бы непрозрачный IL, который в SQL не переведёшь, и пришлось бы тащить все строки в память и фильтровать там (это и есть пугающая «client-side evaluation»). Полный разбор — в главе **IEnumerable vs IQueryable**.

#### Чего компилятор в дерево не положит

Тут кроется источник классических «почему мой код в EF не компилируется». Из лямбды компилятор умеет делать дерево только если она — **одно выражение** (expression-bodied). Запрещены, в частности:

- **statement body** — `p => { ... }` с фигурными скобками и `return` (ошибка CS0834);
- **`async`/`await`** (CS1989);
- **null-conditional `?.`** (CS8072), **кортежи** `(a, b)` (CS8143), **`is`-паттерны** (CS8122);
- **локальные функции** (CS8110), **collection expressions** `[1, 2]` (CS9175), **`dynamic`**, именованные/опциональные аргументы.

Причина историческая: почти всё это — синтаксис, появившийся после C# 3.0, и его добавление в деревья сломало бы все библиотеки, которые деревья разбирают. Практический перевод: «в `Where` у EF нельзя многострочную лямбду или `?.`» — это прямое следствие. И важная асимметрия, которую любят на собеседовании: то, что **компилятор** не строит из statement-лямбды, ты всё ещё можешь собрать **руками** через `Expression.Block` (см. 🔬).

### 🔬 Глубже / на синьора

**Что делает `Compile()` под капотом.** В обычном рантайме `Compile()` генерирует IL через `Reflection.Emit` (`DynamicMethod`), который затем компилирует JIT — отсюда делегат, близкий к написанному руками. Есть нюанс прогрева: для первых вызовов рантайм может использовать **интерпретатор** деревьев (быстрее «скомпилировать», медленнее исполнять), а затем подняться до настоящей компиляции. Управлять можно явно: `lambda.Compile(preferInterpretation: true)` — выгодно, когда делегат вызовут считаные разы. Под **Native AOT** `Reflection.Emit` недоступен, поэтому `System.Linq.Expressions` **всегда работают в интерпретируемом режиме** — это медленнее скомпилированного, и привычный приём «`Expression.Compile` вместо рефлексии ради скорости» под AOT теряет смысл (там предпочтителен кэшированный `CreateDelegate`, см. главу **Reflection**).

**Кэширование скомпилированных делегатов.** Поскольку компиляция доминирует над вызовом по стоимости, в проде компилируют один раз и кладут делегат в `ConcurrentDictionary<TKey, TDelegate>`. EF Core для горячих запросов даёт явный `EF.CompileQuery` / `EF.CompileAsyncQuery` — тот же принцип: вынести построение и трансляцию дерева из горячего пути.

**Захваченные переменные (closures) в дереве.** Когда лямбда захватывает локальную переменную, в дереве она оказывается не как `ConstantExpression` со значением, а как **доступ к полю closure-объекта** (`MemberExpression` над `ConstantExpression`, хранящим экземпляр класса-замыкания). Для EF это принципиально: он распознаёт такие узлы и **выносит их в SQL-параметры**, поэтому два вызова запроса с разными значениями переменной дают один параметризованный SQL и переиспользуют план — а не плодят новый план на каждое значение.

**Деревья неизменяемы; визитор — это трансформация.** Любое «изменение» дерева — это построение нового через `ExpressionVisitor`. Это даёт безопасную композицию (комбинирование предикатов, подстановка параметров), но означает аллокации при каждой трансформации — ещё одна причина кэшировать результат.

**Statement trees руками.** Хотя компилятор не конвертирует statement-лямбду, фабрики позволяют собрать полноценное «процедурное» дерево: `Expression.Block` (последовательность + локальные переменные), `Expression.Assign`, `Expression.Condition`, `Expression.Loop` + `Expression.Break`/`Continue`, `Expression.Try`. На этом построен DLR и многие кодогенераторы. То есть ограничение «только одно выражение» — свойство **C#-конвертации лямбды**, а не самой модели деревьев.

**Reflection vs Expression Trees — сведём окончательно.** Их путают именно потому, что оба «делают что-то динамически», но задачи разные:

||Reflection|Expression Trees|
|---|---|---|
|Задача|**читать/инспектировать** метаданные (типы, члены, атрибуты)|**представлять и генерировать** код|
|Перечислить методы/атрибуты типа|да|нет (не его задача)|
|Скорость повторного вызова|медленный per-call lookup; ускоряется кэшем + делегатом|после `Compile()` ≈ прямой вызов|
|Перевод в другой язык (SQL)|нет|да (обход дерева)|
|Под Native AOT|работает, но враждебен trimming|`Compile` → интерпретатор (медленно)|

На практике их **комбинируют**: рефлексией находишь нужный член один раз, строишь по нему дерево, компилируешь в делегат и кэшируешь — так устроены быстрые мапперы и сериализаторы (рефлексия для «что», дерево для «быстро и много раз»).

#### Частые поверхностные ответы и почему проваливаются

- **«Expression tree — это просто лямбда».** Стирает суть: лямбда в делегат исполняема и непрозрачна, дерево — данные, описывающие код, которые надо либо скомпилировать, либо обойти.
- **«Деревья всегда быстрее рефлексии».** Игнорирует стоимость `Compile()`: без кэширования делегата это проигрыш; под AOT — интерпретатор.
- **«В EF любую лямбду можно».** Не знает про ограничение «только expression-body»: statement body, `?.`, кортежи, `is`-паттерны не конвертируются.
- **«Дерево можно изменить на месте».** Узлы неизменяемы; трансформация — это новый граф через `ExpressionVisitor`.

#### Модельные ответы

**«Что такое expression tree и чем отличается от делегата?»**

> _RU:_ Expression tree — это представление кода в виде объектного графа: узлы вроде `BinaryExpression`, `MemberExpression`, `ConstantExpression` описывают выражение, но сами не исполняются. Один и тот же текст лямбды компилируется по-разному в зависимости от целевого типа: `Func<…>` — это исполняемый делегат (IL, чёрный ящик), `Expression<Func<…>>` — дерево-данные. Делегат вызываешь сразу; дерево либо обходишь и транслируешь (так EF строит SQL), либо компилируешь в делегат через `Compile()`. Поэтому дерево — это «код как данные», а делегат — «код как функция».

> _EN:_ An expression tree is code represented as an object graph: nodes like `BinaryExpression`, `MemberExpression`, `ConstantExpression` describe an expression but don’t execute on their own. The same lambda text compiles differently depending on the target type: `Func<…>` is an executable delegate (IL, a black box), `Expression<Func<…>>` is tree data. A delegate you just call; a tree you either traverse and translate (how EF builds SQL) or compile into a delegate via `Compile()`. So a tree is “code as data,” a delegate is “code as a function.”

**«Почему `IQueryable` принимает `Expression`, а не `Func`?»**

> _RU:_ Потому что провайдеру (EF) нужно перевести запрос в SQL, а для этого он должен прочитать структуру предиката, а не выполнить его. `Expression<Func<T,bool>>` — это дерево, которое EF обходит визитором и транслирует в `WHERE`. Если передать `Func<T,bool>` (как в `IEnumerable.Where`), внутри будет непрозрачный делегат, который нельзя разобрать, — фильтр выполнится client-side, уже после загрузки всех строк. Ограничение «только expression-body» в LINQ-запросах — прямое следствие: statement-лямбду компилятор в дерево не превращает.

> _EN:_ Because the provider (EF) needs to translate the query into SQL, which requires reading the predicate’s structure rather than executing it. `Expression<Func<T,bool>>` is a tree EF traverses with a visitor and translates into a `WHERE`. Passing a `Func<T,bool>` (as in `IEnumerable.Where`) gives an opaque delegate it can’t inspect — the filter runs client-side, after all rows are loaded. The “expression-body only” restriction in LINQ queries follows directly: the compiler won’t convert a statement-bodied lambda into a tree.

**«Expression Trees vs Reflection?»**

> _RU:_ Это разные инструменты. Reflection читает и инспектирует метаданные — перечисляет члены, читает атрибуты, создаёт объекты по `Type`; её вызов интерпретируемый и медленный per-call, лечится кэшем и делегатом. Expression Trees представляют и генерируют код: ты строишь дерево, и `Compile()` даёт делегат со скоростью почти прямого вызова, но компиляция дорогая, поэтому делегат кэшируют. Перечислить методы или прочитать атрибуты дерево не умеет — это не его задача. На практике их комбинируют: рефлексией находишь член один раз, по нему строишь и кэшируешь скомпилированное дерево для быстрого повторного доступа.

> _EN:_ They’re different tools. Reflection reads and inspects metadata — enumerates members, reads attributes, creates objects from a `Type`; its invocation is interpreted and slow per call, mitigated by caching plus a delegate. Expression Trees represent and generate code: you build a tree and `Compile()` yields a delegate that runs near a direct call, but compilation is expensive, so the delegate is cached. A tree can’t enumerate methods or read attributes — not its job. In practice you combine them: use reflection once to find a member, then build and cache a compiled tree for fast repeated access.
