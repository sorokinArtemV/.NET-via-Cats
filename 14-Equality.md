### Зачем эта тема

«Равны ли эти два объекта» звучит как один вопрос, но в C# это **несколько независимых механизмов**, которые легко спутать: сравнение по ссылке против сравнения по значению, оператор `==` против виртуального `Equals`, плюс `GetHashCode`, который обязан быть согласован с `Equals`, иначе ломаются `Dictionary` и `HashSet`. Интервьюер почти всегда бьёт по стыкам: «для строк `==` и `Equals` — одно и то же?», «почему у структуры `==` не компилируется?», «что будет, если переопределить `Equals`, но забыть `GetHashCode`?». Глава разбирает каждый механизм и места, где они расходятся.

### Mental model

Равенство в C# — это **четыре разных механизма**, а не один:

- `==` — статический оператор, который **компилятор** выбирает на этапе компиляции по _статическому_ типу переменных.
- `Equals(object)` — виртуальный метод, который **рантайм** диспетчеризует по _фактическому_ типу объекта.
- `ReferenceEquals` — всегда сравнение идентичности ссылки, переопределить нельзя.
- `IEquatable<T>.Equals(T)` — типизированная версия `Equals` без boxing и каста.

Ключевой раскол, из которого растут почти все ловушки: **`==` решается компилятором по типу переменной, `Equals` — рантаймом по самому объекту.**

### Reference equality vs value equality

Два смысла «равны»: это _один и тот же объект_ (identity) или _два объекта с одинаковым содержимым_ (value). Дефолты зависят от того, class это или struct:

- **`class`** по умолчанию: и `==`, и `Equals` дают **идентичность ссылки**. Два разных объекта с одинаковыми полями не равны, пока ты сам не переопределишь.
- **`struct`** по умолчанию: `==` **не определён вовсе** — `a == b` для структуры без перегрузки оператора это ошибка компиляции `CS0019` («оператор `==` нельзя применить к структуре, пока тип явно не перегрузит его»). А вот `Equals` унаследован от `ValueType` и делает **value-сравнение по полям**, но медленно (см. senior-блок).
- **`string`** — особый случай и любимый источник вопросов: это reference type, но и `==`, и `Equals` переопределены на сравнение **по значению** (посимвольно). Поэтому `"ab" == "ab"` → `true`, хотя формально это могут быть разные объекты.

### Четыре механизма по отдельности

```csharp
object.ReferenceEquals(a, b); // static, всегда идентичность ссылки, не переопределяется
a == b;                       // static-оператор, выбор компилятором по статическому типу
a.Equals(b);                  // virtual, диспетч рантаймом по фактическому типу
object.Equals(a, b);          // static null-safe хелпер (семантика — ниже)
```

- **`ReferenceEquals`** всегда сравнивает ссылки и игнорирует любые перегрузки. Для value-типов бессмыслен: аргументы боксируются в два разных объекта, результат всегда `false`.
- **`==`** — статический оператор. Он **не виртуальный**: компилятор смотрит на _объявленные_ типы операндов и подставляет нужную перегрузку прямо в IL. Если перегрузки для этого типа нет — берётся ссылочное сравнение (для классов) или ошибка (для структур).
- **`Equals(object)`** — виртуальный, диспетчеризуется по фактическому типу в рантайме. Каст переменной к другому типу его _не_ меняет.
- **`object.Equals(a, b)`** — статический хелпер: возвращает `true`, если оба аргумента `null`; `false`, если `null` ровно один; иначе вызывает виртуальный `a.Equals(b)`. Безопасен, когда `a` может быть `null`.

Отсюда — классическая ловушка. Один и тот же объект сравнивается по-разному в зависимости от того, к какому типу приведена переменная:

Каст к `object` не меняет объект — он меняет только то, **какую перегрузку `==` видит компилятор**. `Equals` же диспетчеризуется по фактическому типу и остаётся `string`-овым, поэтому возвращает `true` независимо от каста. Тот же эффект ловит generic-код: внутри `Compare<T>(T a, T b)`, где `T` известен только как `object`-подобный, `a == b` даст ссылочное сравнение, даже если `T` на самом деле `string` — поэтому в обобщённом коде используют `EqualityComparer<T>.Default.Equals`, а не `==`.

**Важная оговорка про строки.** Ловушка проявляется только на строках, созданных **в рантайме** — `new string(...)`, конкатенация, ввод, чтение из БД. Строковые **литералы интернируются**: `(object)"hello" == (object)"hello"` вернёт `true`, потому что обе ссылки указывают на один интернированный объект. Если проверяешь пример на литералах — увидишь `true` и сделаешь неверный вывод. Чтобы воспроизвести `false`, нужны заведомо разные ссылки: `var s1 = new string("hello".ToCharArray()); var s2 = new string("hello".ToCharArray());`.

### Контракт GetHashCode

`GetHashCode` существует ради одной задачи — раскладывать объекты по бакетам в hash-коллекциях (`Dictionary`, `HashSet`). Контракт:

1. **Равные объекты обязаны иметь равный хеш.** Если `a.Equals(b)`, то `a.GetHashCode() == b.GetHashCode()`. Это жёсткое требование — нарушишь, и объект «потеряется» в `Dictionary`.
2. **Обратное не требуется**: разные объекты _могут_ иметь одинаковый хеш (коллизия — норма, неизбежна по принципу Дирихле).
3. **Хеш стабилен**, пока объект лежит ключом в коллекции. Менять поля, влияющие на хеш, у объекта-ключа нельзя.
4. **Переопределил `Equals` → обязан переопределить `GetHashCode`** (и наоборот). Иначе два «равных» объекта попадут в разные бакеты.

**Ловушка mutable-ключа.** Положил объект в `HashSet`, потом изменил поле, которое участвует в `GetHashCode` — объект остаётся в старом бакете, но `Contains` ищет его в новом. Объект физически в коллекции, но «не находится». Это реальный продакшен-баг; вывод — ключи hash-коллекций должны быть иммутабельны по тем полям, что входят в равенство. Парная ловушка к рассогласованию `CompareTo`/`Equals` из главы про `IComparable`.

### IEquatable<T>

`IEquatable<T>` — это типизированный `Equals(T other)` вместо `Equals(object other)`. Зачем:

- **Нет boxing** для value-типов. `Equals(object)` у структуры боксирует аргумент в кучу; `Equals(T)` принимает структуру напрямую.
- **Нет каста** и проверки типа внутри.

Кто его вызывает: `EqualityComparer<T>.Default` сначала проверяет, реализует ли `T` интерфейс `IEquatable<T>`, и если да — зовёт типизированный `Equals(T)`. А через `EqualityComparer<T>.Default` работают `Dictionary<TKey,_>`, `HashSet<T>`, `List<T>.Contains/IndexOf`, LINQ. Поэтому для типа, который пойдёт ключом или в `Contains`, реализация `IEquatable<T>` — это и корректность, и производительность.

### Чеклист корректной реализации (для класса)

```csharp
public sealed class Point : IEquatable<Point>   // sealed: см. примечание ниже
{
    public int X { get; }
    public int Y { get; }

    public bool Equals(Point? other) =>           // типизированный, основная логика тут
        other is not null && X == other.X && Y == other.Y;

    public override bool Equals(object? obj) =>   // override от object → делегирует
        Equals(obj as Point);

    public override int GetHashCode() =>          // согласован с Equals: те же поля
        HashCode.Combine(X, Y);

    public static bool operator ==(Point? a, Point? b) =>  // пара == / !=
        a is null ? b is null : a.Equals(b);
    public static bool operator !=(Point? a, Point? b) => !(a == b);
}
```

Правила: вся логика — в `Equals(T)`; остальные методы делегируют; `GetHashCode` использует **те же поля**, что и `Equals`.

- **`sealed`** взят не случайно: он снимает проблему симметрии равенства при наследовании (наследник с дополнительным полем легко ломает `a.Equals(b) == b.Equals(a)`; подробнее в senior-блоке). Если тип не sealed — сравнивай `GetType()`, а не `is`.
- **`HashCode.Combine`** (.NET Core 2.1+ / .NET Standard 2.1) избавляет от ручного `x * 397 ^ y`, но берёт **максимум 8 аргументов**. Для типов с бóльшим числом полей используй мутабельную структуру `HashCode`: `var h = new HashCode(); h.Add(a); h.Add(b); … return h.ToHashCode();`.
- Для иммутабельных типов с value-семантикой почти всегда лучше взять `record` — он сгенерирует `Equals`/`GetHashCode`/`==`/`IEquatable<T>` сам (кросс-ссылка на главу про records).

### 🔬 Глубже / на синьора

**Почему дефолтный `ValueType.Equals` медленный.** Для структуры без перегрузки `Equals` идёт в `ValueType.Equals`, который работает в два пути. Если структура «bitwise-сравнима» — нет ссылочных полей (GC-указателей) и нет «дыр» padding'а между полями — рантайм вызывает `CanCompareBits` → `FastEqualsCheck`, то есть быстрый `memcmp` по байтам. Иначе — **рефлексия**: перебор полей через `FieldInfo` с боксированием значений, медленно. Важный нюанс: **поля с плавающей точкой ломают bitwise-путь**, потому что `+0.0` и `−0.0` имеют разные биты, но равны, а `NaN` — наоборот. Поэтому структура с `double` всегда идёт по медленному рефлексивному пути.

**Почему дефолтный `ValueType.GetHashCode` даёт плохое распределение.** Та же развилка. Быстрый путь (нет ссылок + плотная упаковка) XOR'ит 4-байтовые блоки — учитывает все поля. Но рефлексивный fallback ради скорости берёт хеш **только первого нестатического поля** (munged с id типа). Следствие: структура `(int Id, string Name)`, у которой `Id` всегда `0`, даст одинаковый хеш для всех экземпляров → все в один бакет → `Dictionary` деградирует до связного списка с O(n)-поиском. Вывод тот же: для struct-ключей переопределяй `Equals`/`GetHashCode` руками или объявляй `record struct`, где компилятор генерирует нормальные реализации.

**Дефолтный `object.GetHashCode` (reference type).** Это **identity hash**: уникальное число, привязанное к объекту. Вычисляется лениво при первом вызове и сохраняется в заголовке объекта (в области sync-block index). Это **не адрес** — GC двигает объекты в памяти, а хеш обязан быть стабильным на всю жизнь объекта, поэтому адрес использовать нельзя.

**Equality и наследование.** Легко сломать симметрию: если базовый `Equals` проверяет `obj is Base`, а производный добавляет поле, то `b.Equals(d)` и `d.Equals(b)` могут разойтись — нарушение контракта (равенство обязано быть рефлексивным, симметричным и транзитивным). Поэтому в `Equals` для не-sealed иерархий обычно сравнивают `GetType() == other.GetType()` (строгое совпадение типа), а не `is` (который пропускает наследников).

**`IEqualityComparer<T>` — равенство снаружи.** Когда нужно несколько стратегий равенства для одного типа или равенство для типа, который ты не контролируешь — реализуешь `IEqualityComparer<T>` (`Equals(T,T)` + `GetHashCode(T)`) и передаёшь в `Dictionary`/`HashSet`/`Distinct`/`GroupBy`. Это параллель к `IComparer<T>` из главы про `IComparable`. Готовые: `StringComparer.OrdinalIgnoreCase`, `StringComparer.Ordinal` и т.д.

**NaN — где `==` и `Equals` расходятся.** `NaN == NaN` → `false` (так требует IEEE 754), но `NaN.Equals(NaN)` → `true`. Сделано намеренно: иначе `NaN` нельзя было бы найти ключом в коллекции или в массиве (рефлексивность `Equals` обязательна). А вот `+0.0` и `−0.0` равны под **обоими** — и `==`, и `Equals`. Это та же причина, по которой float ломает bitwise-путь `ValueType.Equals`: побитовое равенство и семантическое равенство для float не совпадают.

### Interview layer

**Вопросы по частоте:**

**1. В чём разница `==` и `Equals`?**  
→ follow-up: для строк они дают одно и то же?  
→ trap: что выведет `(object)s1 == (object)s2` для двух одинаковых строк, созданных в рантайме?  
Суть: `==` статичен (компилятор, по типу переменной), `Equals` виртуален (рантайм, по объекту). Для `string` оба — value, но каст к `object` переключает `==` на ссылочное сравнение (на литералах не воспроизводится из-за интернирования).

**2. Зачем переопределять `GetHashCode` вместе с `Equals`?**  
→ follow-up: что будет, если переопределить только `Equals`?  
→ trap: можно ли менять поля у объекта, лежащего ключом в `Dictionary`?  
Суть: hash-коллекции сначала ищут бакет по хешу. Рассогласование → равные объекты в разных бакетах → ключ не находится.

**3. Что вернёт дефолтный `Equals` у класса и у структуры?**  
→ follow-up: почему `a == b` для структуры не компилируется?  
→ trap (senior): чем плох дефолтный `ValueType.Equals` по производительности?  
Суть: class → идентичность ссылки; struct → value-сравнение по полям (но `==` не определён без перегрузки). Дефолт у структуры медленный из-за рефлексии (или ломается на float).

**4. Зачем нужен `IEquatable<T>`?**  
→ follow-up: кто его реально вызывает?  
→ trap: какова цена его отсутствия для структуры-ключа?  
Суть: типизированный `Equals(T)` без boxing/каста; вызывается через `EqualityComparer<T>.Default` из `Dictionary`/`HashSet`/LINQ.

**5. Сформулируй контракт `GetHashCode`.**  
→ trap: обязаны ли два неравных объекта иметь разные хеши?  
Суть: равные → равный хеш (обязательно); неравные — не обязаны различаться (коллизии нормальны).

**6. (senior) `NaN == NaN`?** → почему `Equals` отвечает иначе?

**7. (senior) Как задать равенство, не трогая сам тип?** → `IEqualityComparer<T>`, `StringComparer`.

**Шаблонно-неверные ответы:**

- _«`==` сравнивает значения, `Equals` — ссылки»_ — наоборот для классов по дефолту, и для строк зависит от статического типа. Поверхностно и валит на follow-up про `(object)`.
- _«`GetHashCode` нужен для уникальности объектов»_ — нет, не уникальность, а согласованность с `Equals`; коллизии допустимы.
- _«У структур равенство работает само, можно не думать»_ — работает, но через медленную рефлексию и с плохим хешем; для ключей это баг по производительности.
- _«Переопределил `Equals` — оператор `==` тоже поменялся»_ — нет, `==` не виртуален; его надо перегружать отдельно.

**Model answer (mid+):**

> Равенство в C# — это несколько механизмов. `==` — статический оператор: компилятор выбирает перегрузку по типу переменной на этапе компиляции, он не виртуальный. `Equals` — виртуальный метод от `object`: рантайм диспетчеризует его по фактическому типу. Поэтому для строк `s1 == s2` и `s1.Equals(s2)` дают `true` по значению, но `(object)s1 == (object)s2` для рантайм-строк даст `false` — компилятор увидел тип `object` и подставил ссылочное сравнение, тогда как `Equals` остался строковым. При переопределении `Equals` обязательно переопределяю `GetHashCode` теми же полями: hash-коллекции ищут по хешу, и рассогласование приводит к тому, что равный ключ не находится. Для типов, идущих в `Dictionary`/`HashSet`, реализую `IEquatable<T>` — это убирает boxing для структур, и именно его зовёт `EqualityComparer<T>.Default`.

> _(senior add-on):_ Дефолтный `ValueType.Equals` идёт либо по быстрому `memcmp` (если нет ссылок и padding'а), либо по рефлексии; поля `double` ломают быстрый путь из-за `NaN` и знакового нуля. А дефолтный `ValueType.GetHashCode` в рефлексивном пути берёт хеш только первого поля — отсюда плохое распределение, поэтому для struct-ключей переопределяю руками или беру `record struct`.
