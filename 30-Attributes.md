Атрибут — это кусочек метаданных, который ты навешиваешь на элемент кода (тип, метод, свойство, параметр, сборку), чтобы прикрепить к нему структурированную информацию. Ключевое и контринтуитивное: **сам по себе атрибут не делает ничего**. `[Obsolete]` не запрещает вызывать метод, `[Required]` не валидирует, `[JsonIgnore]` не прячет свойство. Это инертные данные, запечённые в сборку. «Поведение» появляется только тогда, когда кто-то — компилятор, рантайм через рефлексию или генератор кода — эти данные прочитает и решит на их основе что-то сделать.

Зачем такой механизм нужен? Он даёт **декларативный** способ расширять смысл кода, не меняя его логику. Вместо того чтобы писать императивную проверку «если это свойство — ключ, то…», ты помечаешь `[Key]`, а ORM сам разберётся при чтении. Атрибуты — это контракт между твоим кодом и инструментом, который его обрабатывает: валидаторами, сериализаторами, DI, тест-раннерами, роутингом ASP.NET, а в последние годы — и source generators. Эта глава продолжает **Reflection**: там мы научились читать метаданные, здесь — записывать их декларативно и грамотно потреблять.

### Mental model

**Атрибут — это данные в метаданных, а не код.** Компилятор сериализует аргументы атрибута в blob внутри сборки; до тех пор пока кто-то не прочитает этот blob (рефлексией в рантайме, или это не сделает сам компилятор/генератор на этапе сборки), атрибут полностью пассивен. «Кто читает и когда» — главный вопрос, который надо держать в голове про любой атрибут.

### Как это работает (mid+ core)

#### Что можно положить в атрибут

Аргументы атрибута сериализуются в метаданные, поэтому они обязаны быть **compile-time-константами**. Разрешены: примитивные типы (`bool`, числовые, `char`, `string`), `enum`, `Type` (через `typeof`) и одномерные массивы из перечисленного. Нельзя: произвольные объекты, неконстантные выражения, `new List<…>()`.

```csharp
[MyAttr(42, Tags = new[] { "a", "b" }, Target = typeof(Person))] // ок
[MyAttr(DateTime.Now)]   // ❌ не константа
[MyAttr(3.14m)]          // ❌ decimal не допустим как аргумент атрибута (ловушка)
```

`decimal` в списке аргументов запрещён — он не «примитив» на уровне метаданных. Это любимый каверзный вопрос.

#### Позиционные vs именованные параметры

```csharp
[AttributeUsage(AttributeTargets.Property)]
public sealed class ColumnAttribute : Attribute
{
    public ColumnAttribute(string name) => Name = name; // позиционный (ctor)
    public string Name { get; }
    public bool IsKey { get; set; }                     // именованный (свойство/поле)
}

[Column("user_id", IsKey = true)] // позиционный + именованный
public int Id { get; set; }
```

Позиционные параметры — это параметры конструктора (обязательные, по порядку). Именованные — публичные свойства/поля, задаются синтаксисом `Name = value` (необязательные). Конвенция: класс называют с суффиксом `Attribute`, а применяют без него (`[Column]` ↔ `ColumnAttribute`); компилятор сопоставляет сам.

#### `AttributeUsage` — три параметра, которые надо знать наизусть

```csharp
[AttributeUsage(AttributeTargets.Method | AttributeTargets.Property,
                AllowMultiple = false,   // дефолт: false
                Inherited = true)]        // дефолт: true
public sealed class AuditAttribute : Attribute { }
```

- **`AttributeTargets`** (флаговый enum) — куда можно вешать. Нарушение — ошибка компиляции.
- **`AllowMultiple`** (дефолт `false`) — можно ли повесить атрибут несколько раз на один элемент.
- **`Inherited`** (дефолт `true`) — увидит ли `GetCustomAttributes(inherit: true)` атрибут с базового класса/метода на наследнике. Тонкость: наследование работает для классов и переопределённых членов, но **не** тянется с интерфейсов на реализующий класс — это частая ошибка ожиданий.

#### Generic attributes (C# 11+)

Вместо передачи `Type` через `typeof` можно типизировать атрибут параметром и наложить ограничения — компилятор проверит тип на этапе компиляции:

```csharp
public sealed class ValidatorAttribute<T> : Attribute where T : IValidator { }

[Validator<PersonValidator>]  // ок, если PersonValidator : IValidator
[Validator<string>]           // ❌ ошибка компиляции — не проходит constraint
public class Person { }
```

Главный выигрыш — типобезопасность вместо рантайм-проверки `typeof`-аргумента. Ограничения этой фичи — в 🔬 (они неочевидны и любимы интервьюерами).

#### Как читать атрибуты — три инструмента с разной ценой

|Способ|Что делает|Создаёт экземпляр атрибута?|Когда брать|
|---|---|---|---|
|`IsDefined(type, inherit)`|проверяет только наличие, возвращает `bool`|**нет**|нужен факт «есть/нет» — самый дешёвый|
|`GetCustomAttributes(...)` / `GetCustomAttribute<T>()`|возвращает готовые объекты атрибутов|**да** (вызывает ctor)|нужны значения из атрибута|
|`CustomAttributeData.GetCustomAttributes(...)`|читает blob метаданных как данные|**нет**|инспекция без исполнения ctor|

```csharp
if (member.IsDefined(typeof(ObsoleteAttribute), inherit: true)) { /* дёшево */ }

var col = prop.GetCustomAttribute<ColumnAttribute>(); // инстанцирует
if (col is { IsKey: true }) { /* ... */ }
```

Ключевой нюанс: `GetCustomAttributes` **конструирует новый экземпляр** атрибута при каждом вызове (об этом в 🔬), поэтому в горячем пути её кэшируют, как и любую рефлексию.

#### Важные встроенные атрибуты

- **`[Obsolete("msg")]`** — предупреждение при использовании; с `[Obsolete("msg", error: true)]` — ошибка компиляции. Читается компилятором.
- **`[Conditional("DEBUG")]`** — компилятор **вырезает сами вызовы** метода, если символ не определён (так устроен `Debug.Assert`). Метод обязан возвращать `void`. Это не то же самое, что `#if` — см. 🔬.
- **`[Flags]`** — меняет `ToString()`/парсинг enum как битовой маски (разбор в главе **Enums**).
- **`[CallerMemberName]` / `[CallerLineNumber]` / `[CallerFilePath]` / `[CallerArgumentExpression("x")]`** — компилятор подставляет значения на стороне вызова. Особый случай: их «поведение» обеспечивает компилятор, а не рефлексия.

### 🔬 Глубже / на синьора

**Когда атрибут инстанцируется и почему это не бесплатно.** `GetCustomAttributes` каждый раз **заново вызывает конструктор** атрибута и создаёт новые объекты — они не кэшируются рантаймом и не равны между вызовами по ссылке. Для часто читаемых атрибутов (валидация, маппинг в цикле) это сочетание lookup + аллокация; правильный паттерн — прочитать один раз на тип и положить результат в `ConcurrentDictionary<Type, …>`. Если нужен только факт наличия — `IsDefined` дешевле, потому что не конструирует объект; если нужно проинспектировать аргументы без побочных эффектов ctor — `CustomAttributeData` читает их как сырые данные.

**Pseudo-custom attributes — то, чего `GetCustomAttributes` не покажет.** Часть атрибутов компилятор не кладёт как обычный CA-blob, а превращает в **флаги прямо в таблицах метаданных**: `[Serializable]`, `[StructLayout]`, `[DllImport]`, `[MarshalAs]`, `[FieldOffset]`, `[In]`/`[Out]`. Их **не вернёт** ни `GetCustomAttributes`, ни `CustomAttributeData` — читать нужно через специализированные API (`type.IsSerializable`, `type.Attributes & TypeAttributes.Serializable`, `type.StructLayoutAttribute`, `method.GetMethodImplementationFlags()`). Это прямое продолжение того, о чём шла речь в главе **Reflection**, и сильный senior-差ентиатор.

**Ограничения generic attributes (C# 11+), которые ломают наивные ожидания.**

- Применять можно только **полностью закрытый** generic-атрибут. `[Foo<T>]`, где `T` — параметр объемлющего типа, не компилируется (CS8968): все типы-аргументы должны быть конкретными.
- Типы-аргументы подчиняются тем же правилам, что и `typeof`: **нельзя** `dynamic` (CS8970), nullable reference types (`string?`), и кортежи C#-синтаксисом `(int, int)` — потому что они требуют сопутствующих метаданных-аннотаций, которые некуда «прицепить». Обходы: `object` вместо `dynamic`, `string` вместо `string?`, `ValueTuple<int,int>` вместо `(int, int)`, `IntPtr` вместо `nint`.
- Семантика `AllowMultiple = false`: `[Attr<string>]` и `[Attr<object>]` на одном элементе считаются «тем же» атрибутом (компилятор смотрит на исходное открытое определение) — вместе они запрещены.
- Читать из рефлексии: `GetCustomAttributes(typeof(Attr<>), …)`, затем у найденного `attr.GetType().GetGenericArguments()`.

**`[Conditional]` vs `#if`.** `#if DEBUG` вырезает **код целиком** на этапе препроцессинга — если символа нет, тела метода в сборке вообще нет. `[Conditional("DEBUG")]` оставляет метод в сборке, но **компилятор у вызывающего** опускает вызовы, если символ не определён в _его_ проекте. То есть условность переезжает на сторону вызова — поэтому `[Conditional]` и работает для библиотечных диагностических методов, а аргументы вычеркнутых вызовов не вычисляются.

**Атрибуты как драйвер source generators и AOT-дружелюбие.** Современный паттерн: маркер-атрибут (`[GeneratedRegex]`, `[JsonSerializable]`, `[LoggerMessage]`) читается генератором **на этапе компиляции** и разворачивается в обычный код — без рантайм-рефлексии. Это и есть стратегический ответ платформы на враждебность Reflection к trimming/Native AOT (см. 🔬 в **Reflection**): тот же декларативный синтаксис атрибутов, но «потребитель» переехал из рантайма в компилятор, и метаданные больше не нужны живыми во время выполнения.

### Interview layer

#### Вопросы по частоте

1. **Что такое атрибут и делает ли он что-нибудь сам?** (очень часто, mid)
2. **`AttributeUsage`: что значат `AllowMultiple`, `Inherited`, `AttributeTargets`?** (часто, mid+)
3. **Как прочитать атрибуты типа/члена через рефлексию?** (часто, mid)
4. **Что можно передавать в атрибут как аргумент?** (mid, ловушки про `decimal`/неконстанты)
5. **`IsDefined` vs `GetCustomAttributes` — когда что?** (mid+)
6. **Generic attributes: зачем и какие ограничения?** (senior)
7. **Чем `[Conditional]` отличается от `#if`?** (senior trap)

#### Деревья follow-up

**«Что такое атрибут?»**  
→ _likely:_ делает ли он что-то сам по себе? → _(нет, это инертные метаданные; смысл придаёт читатель)_  
→ _trap:_ кто и когда его читает на примере `[Obsolete]`? → _(компилятор на этапе компиляции; рефлексия — в рантайме; это разные потребители одних данных)_

**«Как читаешь атрибуты?»**  
→ _likely:_ разница `IsDefined` и `GetCustomAttribute`? → _(`IsDefined` не создаёт экземпляр, дешевле; `GetCustomAttribute` вызывает ctor)_  
→ _trap:_ `GetCustomAttributes` кэширует экземпляры? → _(нет — конструирует заново при каждом вызове, поэтому кэшируешь сам)_

**«Сделай атрибут с типом-параметром»**  
→ _likely:_ зачем generic-атрибут вместо `typeof`-параметра? → _(типобезопасность через constraint на этапе компиляции)_  
→ _trap:_ почему `[Validator<string?>]` не компилируется? → _(типы-аргументы подчиняются правилам `typeof`: nullable ref types / `dynamic` / C#-кортежи запрещены)_

#### Частые поверхностные ответы и почему проваливаются

- **«Атрибут навешивает поведение».** Главная ошибка: атрибут инертен; `[Required]` ничего не валидирует, пока валидатор не прочитает его рефлексией.
- **«`Inherited` — про наследование классов вообще».** Не учитывает, что с интерфейсов на класс атрибут не наследуется и что флаг работает в связке с `inherit: true` в `GetCustomAttributes`.
- **«В атрибут можно передать что угодно».** Игнорирует требование compile-time-констант и ловушку с `decimal`/`DateTime`.
- **«Generic-атрибут можно применить с `T` из объемлющего типа».** Нет: атрибут должен быть полностью закрытым (CS8968).

#### Модельные ответы

**«Что такое атрибут и делает ли он что-то сам?»**

> _RU:_ Атрибут — это декларативные метаданные, которые компилятор сериализует в сборку и навешивает на элемент кода. Сам по себе он пассивен: не меняет поведение и не исполняется. Смысл появляется, когда его читает потребитель — компилятор на этапе компиляции (`[Obsolete]`, `[Conditional]`), рантайм через рефлексию (`GetCustomAttributes`) или генератор кода на сборке (`[GeneratedRegex]`). Поэтому правильный вопрос про любой атрибут — кто и когда его читает.

> _EN:_ An attribute is declarative metadata the compiler serializes into the assembly and attaches to a code element. By itself it’s inert — it doesn’t change behavior or run. Meaning comes from a consumer reading it: the compiler at compile time (`[Obsolete]`, `[Conditional]`), the runtime via reflection (`GetCustomAttributes`), or a source generator at build time (`[GeneratedRegex]`). So the right question about any attribute is who reads it, and when.

**«`IsDefined` vs `GetCustomAttributes`?»**

> _RU:_ `IsDefined` только проверяет наличие атрибута и возвращает `bool`, не создавая его экземпляр — это дешевле. `GetCustomAttributes`/`GetCustomAttribute<T>` конструирует объект атрибута (вызывает его ctor), чтобы дать доступ к значениям, и делает это заново при каждом вызове, ничего не кэшируя. Поэтому: нужен факт — `IsDefined`; нужны данные — `GetCustomAttribute`, но в горячем пути результат кэшируешь сам. Если нужно прочитать аргументы без побочных эффектов конструктора — `CustomAttributeData`.

> _EN:_ `IsDefined` only checks presence and returns a `bool` without instantiating the attribute, so it’s cheaper. `GetCustomAttributes`/`GetCustomAttribute<T>` constructs the attribute object (runs its ctor) to expose its values, and does so afresh on every call with no caching. So: need presence — `IsDefined`; need data — `GetCustomAttribute`, but cache it yourself on a hot path. To read the arguments without the ctor’s side effects, use `CustomAttributeData`.
