Эта глава — не «чем отличаются value и reference type» (это была Глава 1), а **дизайн-решение**: когда осознанно писать `struct`, а когда `class`. Цена ошибки тут не академическая. Выбрал struct там, где нужен class, — и получаешь баги мутации копий, которые молча «ничего не делают», и регрессии производительности от копирования больших значений. Выбрал class там, где напрашивался маленький immutable-value, — лишние аллокации и давление на GC в горячем пути. Интервьюеры любят эту тему именно потому, что она проверяет инженерное суждение, а не знание определений: «когда бы ты выбрал struct?» и сразу «почему mutable struct — это плохо?».

### Mental model

**По умолчанию — `class`. `struct` — это оптимизация под узкий профиль: маленький, immutable, логически одно значение. И железное правило: не делай mutable struct.**

Class — для всего, что имеет **идентичность** и живёт во времени (меняющееся состояние, lifetime). Struct — для того, что _является значением_, как `int` или `Point`: два экземпляра с одинаковыми полями неразличимы и взаимозаменяемы.

### Как это работает

#### Когда struct (официальный guideline)

Microsoft формулирует через «избегай, если не выполнены **все** условия». Бери `struct`, только когда тип:

- **логически одно значение**, как примитив (координата, деньги, цвет, strongly-typed id);
- **меньше ~16 байт** по размеру экземпляра;
- **immutable** (после создания не меняется);
- **редко боксится** (иначе аллокации съедают выгоду — см. Главу 3).

Не выполнено хоть одно — пиши `class`. Хорошие кандидаты в struct: `Point(X, Y)`, `Money(Amount, Currency)`, `UserId(Guid)`. Плохие: `Order`, `User`, всё с идентичностью и изменяемым состоянием.

#### Почему mutable struct — зло

Это главный практический пункт темы. Раз struct копируется по значению (Глава 1), то любое чтение, которое возвращает не переменную, а **значение**, отдаёт **копию** — и мутация уходит в копию, а оригинал не трогается:

![[Pasted image 20260629115758.png]]

У этой ловушки два лица. Первое — **молчаливое**: `var p = points[0]; p.X = 99;` компилируется и «работает», но меняет копию `p`, а `points[0]` остаётся прежним (схема выше). Второе — **компилятор спасает**: `points[0].X = 99` на `List<T>` даёт ошибку `CS1612` («нельзя изменить возвращаемое значение, это не переменная»), потому что индексатор возвращает копию. Коварный нюанс: у **массива** `arr[0].X = 99` **компилируется и работает** — индексатор массива возвращает ссылку на элемент, а не копию. То есть одна и та же строка ведёт себя по-разному на `array` и на `List<T>`. Mutable struct превращает это в минное поле — поэтому делай struct **immutable**, и проблема исчезает по построению.

#### Ограничения struct

- **Нет наследования.** Struct неявно sealed и не может наследоваться от другого struct/class — все value types и так наследуют `System.ValueType` (Глава 1). Полиморфизм через базовый тип недоступен.
- **Может реализовывать интерфейсы** — но присвоение struct в переменную-интерфейс боксит (Глава 3). Без boxing — только через generic-constraint.
- **Нельзя `static`** (для статического — `static class`) и нельзя положить `this` под наследование.
- **Дефолт всегда доступен** — `default(T)` обнуляет в обход конструктора (Глава 1), поэтому гарантировать «всегда валидный» экземпляр struct нельзя.

#### 🔬 Глубже / на синьора

**16 байт — это про стоимость копии, а не магический потолок.** Правило существует потому, что примерно до этого размера побитовая копия на x64 «бесплатна». Технически JIT умеет передавать и более крупные struct'ы в регистрах по calling convention, так что жёсткого предела нет. Но суть остаётся: каждое присваивание, каждая передача по значению (без `in`), каждый шаг LINQ копирует **весь** struct. Struct на 64 байта, прошедший по значению через 5 кадров, — это 320 байт копирования. Для больших struct'ов передавай через `in`/`ref` (отдельная глава про модификаторы) — получаешь reference-семантику без аллокации.

**Defensive copy: тихий перформанс-killer.** Если non-readonly struct лежит в `readonly`-поле или передан как `in`-параметр, то при вызове **не-readonly** члена компилятор делает **защитную копию** — иначе член мог бы мутировать то, что обязано быть неизменным. В горячем пути это незаметные лишние копии. Лечится двумя способами: пометить тип `readonly struct` (C# 7.2) — тогда компилятор знает, что мутаций нет; или пометить отдельные члены `readonly` (C# 8). Полное раскрытие `readonly struct` и `in` — в соответствующих главах; здесь важно знать _механизм_ и _симптом_.

**Современный дефолт для value-типа — `readonly record struct` (C# 10).** Даёт value-equality, immutability, `ToString`, деконструкцию и `with` почти без бойлерплейта — это замена ручному «напишу struct и не забуду сделать его immutable». Полный разбор records отложен (отдельная тема); здесь — как ориентир: если нужен маленький immutable value type, в современном C# тянись за `readonly record struct`, а не за голым struct.

**`ref struct` / `Span<T>` (teaser).** Особый вид struct'а, который обязан жить на стеке (нельзя в куче, в полях класса, в async, в обобщениях старых версий). Это инструмент для zero-alloc работы с памятью — отдельная глава позже.

### Interview layer

#### Вопросы по частоте

1. **«Когда выбрать struct вместо class?»** — почти гарантированно. Ждут guideline: маленький, immutable, одно значение, редко боксится — иначе class.
2. **«Почему mutable struct — плохо?»** — главная проверка. Копия при чтении → мутация уходит в копию.
3. **«Может ли struct наследоваться? Реализовывать интерфейсы?»** — нет наследования; интерфейсы да, но через переменную-интерфейс боксит.
4. (синьор) **«Что такое defensive copy и как его убрать?»** / **«Зачем `readonly struct`?»**
5. (синьор) **«Чем `record struct` отличается от обычного struct?»**

#### Follow-up tree

> **Q:** Когда бы ты выбрал struct?  
> **→ likely:** А какие минусы у большого struct?  
> **→ trap:** «Если struct 64 байта и часто передаётся — что не так?» (Дорогие побитовые копии; нужен `in`/`ref`.)

> **Q:** Почему mutable struct — зло?  
> **→ likely:** Что выведет `var p = list[0]; p.X = 99; → list[0].X`?  
> **→ trap:** «А `arr[0].X = 99` на массиве — тоже копия?» (Нет — массив возвращает ссылку на элемент, мутация проходит.)

> **Q:** Зачем `readonly struct`?  
> **→ likely:** Что такое defensive copy?  
> **→ trap:** «Когда именно компилятор делает защитную копию?» (Non-readonly член у struct в `readonly`-поле или `in`-параметре.)

#### Частые поверхностные ответы и почему проваливаются

- **«Struct быстрее, поэтому используй struct».** Не безусловно: большой struct по значению копируется целиком и может оказаться медленнее ссылки на class. Выгода — только для маленьких типов и/или массивов значений.
- **«Mutable struct — норм, просто аккуратнее».** Нет: копия при чтении из коллекции/свойства порождает баги, которые молча «ничего не делают». Решение системное — immutability.
- **«Struct не может иметь методы/интерфейсы».** Может и методы, и интерфейсы; не может — наследоваться.
- **«`list[0].X = 5` и `arr[0].X = 5` ведут себя одинаково».** Нет — на `List<T>` это ошибка компиляции, на массиве работает.
- **«`readonly struct` — это просто про то, что поля нельзя менять».** Это ещё и про производительность: убирает defensive copies при `in`/readonly-полях.

#### Model answer (mid+)

**RU.** По умолчанию я пишу class; struct беру только для маленького (≈до 16 байт) immutable-типа, который логически представляет одно значение и редко боксится, — например `Point`, `Money`, strongly-typed id. Причина — value-семантика: меньше аллокаций, лучше локальность в массивах. Главное правило — не делать mutable struct: раз struct копируется по значению, чтение элемента из коллекции или свойства возвращает копию, и мутация уходит в копию, а оригинал не меняется; на `List<T>` это вообще ошибка компиляции, а на массиве молча меняет элемент — источник тонких багов. Struct не наследуется (неявно sealed, все наследуют `ValueType`), но может реализовывать интерфейсы — правда, через переменную-интерфейс это боксит. Если нужен class — это всё, что имеет идентичность и изменяемое состояние.

_Senior add-on._ 16 байт — эвристика про стоимость побитовой копии, не жёсткий предел; большие struct'ы передавай через `in`/`ref`. Non-readonly член у struct в `readonly`-поле или `in`-параметре вызывает defensive copy — лечится `readonly struct` (7.2) или `readonly`-членами (8). Современный дефолт для immutable value type — `readonly record struct` (C# 10).

**EN.** By default I write a class; I reach for a struct only for a small (~16 bytes or less) immutable type that logically represents a single value and is rarely boxed — like `Point`, `Money`, a strongly-typed id. The reason is value semantics: fewer allocations, better locality in arrays. The key rule is never make a mutable struct: since a struct is copied by value, reading an element from a collection or a property returns a copy, so a mutation hits the copy and the original is unchanged; on a `List<T>` it's even a compile error, while on an array it silently mutates the element — a source of subtle bugs. A struct can't inherit (it's implicitly sealed, all derive from `ValueType`) but can implement interfaces — though assigning it to an interface variable boxes it. Use a class for anything with identity and mutable state.

_Senior add-on._ The 16-byte figure is a heuristic about bitwise-copy cost, not a hard cap; pass large structs by `in`/`ref`. A non-readonly member on a struct held in a `readonly` field or an `in` parameter triggers a defensive copy — fixed by `readonly struct` (7.2) or `readonly` members (8). The modern default for an immutable value type is a `readonly record struct` (C# 10).
