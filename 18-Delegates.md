## Делегаты, Func/Actions, лямбды и замыкания

### Зачем это нужно

В C# легко передать в метод _данные_ — число, строку, объект. А как передать _поведение_ — кусок логики, который метод выполнит, но который заранее не знает? Классический пример: сортировка. `List<T>.Sort()` умеет сравнивать элементы, но _как именно_ сравнивать «больше/меньше» — знает только вызывающий код. Сортировке нужно отдать не данные, а **способ сравнения**.

До делегатов на каждый такой случай заводили интерфейс и класс: `IComparer`, `IFilter`, `IHandler` — церемония из десятка строк ради одной строчки логики. Делегат убирает обёртку: **метод сам становится значением**. Его можно положить в переменную, передать параметром, вернуть из метода, сложить в список, вызвать позже. Это фундамент, на котором стоят коллбэки, события, LINQ, стратегии без классов-наследников, вся «функциональная» часть C#.

Лямбды — это синтаксис, чтобы писать такие кусочки поведения прямо на месте, не объявляя отдельный метод. Замыкания — то, что происходит, когда такой кусочек «утаскивает» с собой переменные из окружения. Дальше — по слоям.

### 1. Делегат как тип

**Mental model:** делегат — это типобезопасная, multicast-способная ссылка на «метод + его объект»; по сути объект, который можно вызвать как метод.

Объявление задаёт _сигнатуру_ — какие методы делегат вообще может держать:

csharp

```csharp
public delegate int MathOperation(int x, int y);

static int Add(int a, int b) => a + b;

MathOperation op = Add;     // метод как значение
int result = op(2, 3);      // 5 — вызов через делегат
```

Ключевой факт: **делегат — это reference type**, а не голый указатель на функцию в стиле C. Когда компилятор видит `delegate`-объявление, он генерирует `sealed`-класс, унаследованный от `MulticastDelegate` (тот, в свою очередь, от `Delegate`). Инстанс делегата хранит **два** поля: `Target` — объект, на котором вызывать метод (для instance-метода), и `Method` — сам метод. Для `Add`, который статический, `Target` будет `null`; для instance-метода делегат **захватывает ссылку на объект** и держит его живым.

Поэтому на вопрос «делегат — value или reference type?» правильный ответ — reference type, и follow-up «а что внутри?» закрывается парой `Target` + `Method`.

🔬 **Глубже / на синьора.** Сгенерённый класс несёт методы `Invoke` (синхронный вызов), а также `BeginInvoke`/`EndInvoke` — наследие старой APM-модели асинхронности. На .NET Framework они работали (через Remoting), но на **.NET Core / .NET 5+** вызов `BeginInvoke`/`EndInvoke` бросает `System.PlatformNotSupportedException: Operation is not supported on this platform.` — реализация зависела от Remoting, которого в .NET Core нет. Современная замена — `Task.Run(() => del(...))`. Ловушка миграции: компилятор это не ловит (методы emit'ятся, а не лежат в библиотеке), падает только в рантайме.

Та же анатомия объясняет и multicast — один делегат может держать список методов:

### 2. Func, Action, Predicate — готовые делегаты

**Mental model:** чтобы не объявлять свой `delegate`-тип под каждую сигнатуру, BCL даёт готовые generic-делегаты. `Action` — ничего не возвращает, `Func` — возвращает, последний параметр типа всегда результат.

csharp

```csharp
Action<string> say = msg => Console.WriteLine(msg);   // void, до 16 параметров
Func<int, int, int> sum = (a, b) => a + b;            // последний T — результат
Predicate<int> isEven = x => x % 2 == 0;              // T → bool
```

**«Func vs собственный delegate — функционально ничем» — неверно.** Разница не только в удобстве. `Func`/`Action` физически **не выражают** `ref`/`out`/`in`-параметры, `params` и атрибуты на параметрах. (`in`/`out` в `Func<in T, out R>` — это аннотации вариантности, а не модификаторы передачи параметра.) Если сигнатуре нужен `out` — готовый делегат не подойдёт, custom обязателен:

csharp

```csharp
// так — только своим делегатом, Func<…> это не выразит:
public delegate bool TryParse(string s, out int result);
```

Плюс свой делегат читаемее (`MathOperation` против `Func<int,int,int>`), несёт осмысленное имя и может документироваться. Вывод для собеседования: в 90% случаев бери `Func`/`Action`; объявляй свой делегат, когда нужна сигнатура с `ref`/`out`/`in`/`params` или когда именованный тип улучшает читаемость API.

**`Predicate<T>` — это НЕ «то же самое, что `Func<T,bool>`».** Форма одинаковая, но это **два разных, невзаимозаменяемых типа**: `Predicate<int>` нельзя передать туда, где параметр объявлен как `Func<int,bool>`, без явной конверсии. На практике: LINQ и весь современный API стандартизировались на `Func<T,bool>`; `Predicate<T>` остался в старых методах — `Array.Find`, `List<T>.FindAll` / `RemoveAll` / `Exists`. Рядом живут ещё два специализированных делегата: `Comparison<T>` (для `List<T>.Sort`) и `Converter<TIn,TOut>`.

|Сценарий|Бери|
|---|---|
|Возвращает значение|`Func<…, TResult>`|
|Ничего не возвращает (void)|`Action<…>`|
|Условие `T → bool` в LINQ/новом API|`Func<T, bool>`|
|Условие для `Array.Find` / `List.FindAll`|`Predicate<T>`|
|`ref`/`out`/`in`/`params` в сигнатуре|свой `delegate` (готовые не умеют)|

### 3. Лямбды и анонимные методы

**Mental model:** лямбда — это метод, который за тебя пишет компилятор: имя, модификаторы и часто типы он выводит сам, тебе остаётся тело.

Историческая ступенька — анонимные методы (`delegate { }`, C# 2); сегодня их вытеснил `=>`. Лямбды бывают expression (`x => x * 2`) и statement (`x => { ...; return ...; }`). Метод не обязательно писать инлайн: если уже есть подходящий именованный метод, его имя само конвертируется в делегат — это **method group conversion**:

csharp

```csharp
Func<int,int,int> f = (a, b) => Add(a, b);  // лямбда-обёртка
Func<int,int,int> g = Add;                  // method group — без обёртки
```

🔬 **Глубже / на синьора.**

**Лямбда → delegate vs лямбда → `Expression`.** В зависимости от _целевого типа_ одна и та же лямбда компилируется по-разному. Если цель — делегат (`Func<…>`), компилятор генерирует **исполняемый метод**. Если цель — `Expression<Func<…>>`, он строит **дерево выражения — данные, описывающие код**, которые можно разобрать в рантайме. Это и есть фундамент различия `IEnumerable` (лямбда = код, выполняется в памяти) и `IQueryable` (лямбда = дерево, транслируется, например, в SQL). Сам LINQ/EF вне scope этой главы, но механизм — языковой, и его спрашивают.

csharp

```csharp
Func<int,int>             code = x => x * 2;   // скомпилированный метод
Expression<Func<int,int>> tree = x => x * 2;   // дерево: данные, не код
```

**Версии.** `static`-лямбды (**C# 9**) — `static x => x * 2` — запрещают захват переменных окружения: попытка захватить даёт ошибку компиляции; внутри можно ссылаться только на static-члены и константы. Это инструмент, чтобы гарантированно убрать аллокацию замыкания (см. §4). Natural type (**C# 10**) — `var f = (int x) => x * x;` — компилятор сам выводит `Func<int,int>` (или синтезирует делегат-тип, если готового нет, например при `ref`-параметре). Дефолтные значения параметров лямбд — **C# 12**.

**Кэширование делегата.** Non-capturing лямбда кэшируется компилятором в статическом поле — создаётся один раз. А вот method group conversion **от static-метода** стал кэшироваться только в **C# 11** (.NET 7); до этого `SomeMethod` в качестве аргумента аллоцировал новый делегат на каждый вызов — частый сюрприз в hot path. Method group от instance-метода по-прежнему аллоцирует (нужно захватить таргет).

> Local functions (C# 7) — близкий конкурент лямбд: когда логика не утекает наружу, они не требуют аллокации делегата и могут быть `static`. Подробно — отдельная тема (cross-ref); здесь достаточно знать, что они есть.

### 4. Замыкания (closures) — главное

**Mental model:** замыкание — это лямбда плюс захваченные ею внешние переменные, которые компилятор «поднимает» с локального стека на heap, чтобы они пережили метод.

Лямбда может ссылаться на переменные из окружающего метода — это **captured variables**. И вот ключевой, контринтуитивный факт: **захват идёт by reference, а не by value**. Лямбда держит не _снимок значения_, а саму переменную — если её изменить после создания лямбды, лямбда увидит новое значение.

Механизм: когда лямбда что-то захватывает, компилятор генерирует скрытый класс (**display class**), делает захваченные переменные его **полями**, а сама лямбда становится методом этого класса. Локальная переменная физически переезжает с стека на heap — в инстанс display class. И исходный код, и лямбда после этого работают с **одним и тем же полем одного объекта**:

![[Pasted image 20260624173317.png]]

**Ловушка переменной цикла** — самый частый трюк-вопрос на эту тему. Что выведет код?

csharp

```csharp
var actions = new List<Action>();
for (int i = 0; i < 3; i++)
    actions.Add(() => Console.Write(i));
foreach (var a in actions) a();   // ???
```

Выведет **`333`**, а не `012`. Все три лямбды захватили **одну и ту же** переменную `i` (захват by reference), к моменту вызова цикл закончился и `i == 3`. Чтобы получить `012`, нужна новая переменная на каждой итерации:

csharp

```csharp
for (int i = 0; i < 3; i++) { int copy = i; actions.Add(() => Console.Write(copy)); }
```

А вот тонкость, которую и проверяют follow-up'ом: с `foreach` поведение **изменилось в C# 5**. До C# 5 переменная итерации `foreach` была общей (тот же баг `333`); начиная с C# 5 она стала **пер-итерационной** — каждый виток захватывается отдельно, и аналогичный цикл выводит ожидаемое. Но `for` это **не затронуло**: его переменная по-прежнему одна на весь цикл. Итог: `foreach` после C# 5 «безопасен», `for` — нет.

🔬 **Глубже / на синьора.**

**Цена.** Захват — не бесплатный: каждый «вход» в область с захватывающей лямбдой аллоцирует инстанс display class на heap (+ нагрузка на GC). На холодном пути незаметно, в hot path (вызывается миллионы раз) — ощутимо. Инструменты: `static`-лямбда (C# 9) гарантирует отсутствие захвата (и компилятор кэширует делегат), либо local function со `static`, либо передача состояния параметром вместо захвата.

**Захват `this`.** Если лямбда обращается к полю или методу экземпляра, она захватывает **`this`** целиком — а значит держит весь объект живым. Классическая утечка: объект подписывает свою лямбду на долгоживущий event чужого объекта; пока не отписался, GC не соберёт ни лямбду, ни `this`. Lifetime событий — почему важно отписываться — см. кросс-ссылку на тему _events_.

### 5. Multicast delegates

**Mental model:** один делегат может держать список методов; `+=` добавляет, `-=` убирает, вызов прогоняет весь список по порядку.

Это и есть основа событий. Два неочевидных, но обязательных для mid+ факта:

**Делегаты immutable.** `+=` и `-=` **не мутируют** делегат, а создают **новый** инстанс с обновлённым invocation list и переприсваивают переменную. То есть `d += x` — это `d = d + x`.

**Семантика вызова** (то, что диаграмма выше резюмировала):

- методы вызываются **в порядке добавления**;
- если делегат возвращает значение, у multicast **наблюдается только `return` последнего** метода — результаты остальных молча отбрасываются (вот почему multicast практичен только для `void`);
- **исключение в любом обработчике прерывает остаток списка** — следующие методы не выполнятся.

Оба ограничения обходятся вручную через `GetInvocationList()` — он отдаёт массив одиночных делегатов, который можно прогнать в цикле, собирая все return'ы и/или изолируя исключения каждого:

csharp

```csharp
foreach (Func<int> h in combined.GetInvocationList())
{
    try { results.Add(h()); }      // собираем КАЖДЫЙ результат
    catch (Exception ex) { /* не роняем остальные */ }
}
```

🔬 **Ловушка `-=` с лямбдой.** Снять подписку можно только **равным** делегатом. Equality делегатов — по паре `Target` + `Method`. Две _отдельно написанные_ одинаковые лямбды компилируются в два разных метода → они не равны → `-=` ничего не снимет:

csharp

```csharp
source.Event += x => Handle(x);   // подписались
source.Event -= x => Handle(x);   // НЕ отписались: это другая лямбда
```

Чтобы отписаться — сохрани делегат в переменную и используй её и для `+=`, и для `-=` (либо подписывайся именованным методом).

### 6. 🔬 Вариантность делегатов (на синьора)

Generic-делегаты поддерживают вариантность через аннотации в их объявлении. `Func<out TResult>` **ковариантен** по возвращаемому типу — `Func<string>` присваивается в `Func<object>` (возвращать `string` там, где ждут `object`, безопасно). `Action<in T>` **контравариантен** по параметру — `Action<object>` присваивается в `Action<string>` (обработчик `object` примет и `string`). Мнемоника: `out`/ковариантность — на «выходе», `in`/контравариантность — на «входе». Помимо этого, ещё со времён C# 2 у method group → delegate действует своя вариантность: возвращаемый тип ковариантен, типы параметров контравариантны при сопоставлении метода делегату.

### 7. Interview-слой

#### Q1. Что такое делегат? Value или reference type? _(базовый, спрашивают почти всегда)_

- **Follow-up:** что он хранит внутри?
- **Trap:** делегат на instance-метод — он держит ссылку на объект? (Да — `Target`, и держит его живым для GC.)
- **Shallow-ответ и почему плох:** «указатель на метод» — проваливается на «value или reference?» и «что внутри?». Делегат — reference type (`sealed : MulticastDelegate`), хранит пару `Target` + `Method`.
- **Model answer:** «Делегат — это reference type, типобезопасная ссылка на метод. Внутри хранит `Target` (объект, на котором вызывать, `null` для static) и `Method`. Может быть multicast — держать список методов. На instance-метод захватывает и удерживает объект.»

#### Q2. Чем `Func<…>` отличается от собственного `delegate`? _(очень частый, mid)_

- **Follow-up:** когда нельзя обойтись `Func`/`Action`?
- **Trap:** а `ref`/`out`-параметр?
- **Shallow-ответ:** «функционально ничем, только удобнее» — **неверно**, и follow-up это вскроет.
- **Model answer:** «В большинстве случаев — только удобство, свой делегат объявлять не нужно. Но `Func`/`Action` не выражают `ref`/`out`/`in`-параметры и `params` — для такой сигнатуры custom-делегат обязателен. Плюс именованный делегат читаемее и документируется.»

#### Q3. Что компилятор делает с лямбдой, захватывающей переменную? _(mid+, отсекающий)_

- **Follow-up:** by value или by reference?
- **Trap:** покажут цикл `for` с `i` — что выведет?
- **Shallow-ответ:** «лямбда копирует переменную» — неверно (by reference), и цикл это докажет.
- **Model answer:** «Компилятор генерирует скрытый класс (display class), захваченные переменные становятся его полями, переменная переезжает на heap. Захват by reference — лямбда и внешний код делят одну переменную. Поэтому цикл `for` с захватом `i` выведет последнее значение для всех лямбд; чтобы избежать — копия в теле цикла. У `foreach` с C# 5 переменная пер-итерационная, баг не воспроизводится; у `for` — по-прежнему да.»

#### Q4. Multicast-делегат с non-void: какой результат? А если один бросит исключение? _(senior follow-up)_

- **Shallow-ответ:** «вернёт все результаты» — нет, отбросит все кроме последнего.
- **Model answer:** «Методы вызываются по порядку добавления. У не-void наблюдается только `return` последнего, остальные отбрасываются — поэтому multicast практичен для `void`. Исключение в обработчике прерывает остаток списка. Чтобы собрать все результаты или изолировать исключения — вызывать вручную через `GetInvocationList()`.»

#### Q5. Подписался лямбдой — как отписаться? _(mid+, практический)_

- **Trap:** `event -= x => Handle(x)` — сработает?
- **Shallow-ответ:** «сделать `-=` с такой же лямбдой» — не сработает: другая лямбда, другой метод, не равны.
- **Model answer:** «Не сработает: equality делегата по `Target` + `Method`, а две одинаковые по тексту лямбды — это разные методы. Чтобы отписаться, нужно сохранить делегат в переменную и использовать её для `+=` и `-=`, или подписываться именованным методом.»

#### Q6. 🔬 _(senior)_ Чем лямбда → `Func` отличается от лямбды → `Expression`?

- **Model answer:** «При цели-делегате компилятор генерирует исполняемый метод. При цели `Expression<Func<…>>` — дерево выражения, то есть данные, описывающие код, разбираемые в рантайме. Это фундамент различия `IEnumerable` (код в памяти) и `IQueryable` (дерево → трансляция, например в SQL).»
