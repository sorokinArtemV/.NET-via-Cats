### Зачем эта глава

«Conversion» / «приведение» — слово-ловушка. В обиходе им зовут что угодно: `(Dog)animal`, `object o = 42`, `int.Parse("5")`, `x as IDisposable`. На собесе это разворачивается в серию follow-up именно потому, что под одним словом сидят операции с **разной ценой и разными правилами отказа**: одна бесплатна, другая аллоцирует на куче, третья парсит строку — и каждая падает по-своему. Кандидат, который отвечает «ну, приведение типов» одним мешком, проваливает первый же уточняющий вопрос («а что при этом происходит с объектом?» / «а это аллоцирует?»). Глава раскладывает мешок на три ящика и показывает, какой инструмент к какому относится и как каждый ломается.

### Mental model

Под словом «conversion» прячутся три разные операции:

|Операция|Что делает|Меняет объект?|Как падает|
|---|---|---|---|
|① Реинтерпретация ссылки (up/down)|смотрит на тот же объект как на другой тип|нет|`InvalidCastException` (downcast)|
|② Смена представления (boxing, numeric, `Convert`)|делает новые биты / новый объект|да, новый|`InvalidCastException` / `Overflow` / `Format`|
|③ Проверка типа (`is` / `as`)|спрашивает «а что это на самом деле»|нет|не падает (`false` / `null`)|

**Ведущая фраза:** «"Приведение" в C# — это три разные вещи: реинтерпретация ссылки, смена представления и runtime-проверка типа. У них разная цена и разные правила отказа, поэтому важно, о какой именно идёт речь.»

### 1. Reference conversions: up/down casting

**Upcasting** — приведение от наследника к базовому типу (`Dog` → `Animal`). Неявное, безопасное, разрешается на компиляции: компилятор уже знает, что `Dog` _является_ `Animal`, поэтому runtime-проверка не нужна. Физически не происходит ничего — тот же указатель на тот же объект; просто переменная теперь имеет static type `Animal`, и через неё видна только часть API. Нужен для полиморфизма: работать с разнородными объектами через общий базовый тип/интерфейс (коллекции, фабрики, DI).

**Downcasting** — обратно, от базового к наследнику (`Animal` → `Dog`). Требует явного каста `(Dog)a` и runtime-проверки: CLR смотрит на реальный тип объекта (через method table pointer в его заголовке) и, если это не `Dog` и не его наследник, бросает `InvalidCastException`. То есть downcast — это не «переделать объект», а «проверить гипотезу о его реальном типе и начать смотреть на него шире».

Ключ к follow-up «что происходит с объектом при касте?»: **объект на куче неизменен**. Cast трогает только static type переменной — то, _как_ на объект смотрят. Реальный тип объект несёт в себе всегда (method table pointer в object header), и именно по нему работает виртуальная диспетчеризация — независимо от того, через какую переменную его дёргаешь.

> **🔬 Глубже / на синьора.** На уровне IL upcast не порождает проверочной инструкции (объект просто типизируется шире). Downcast компилируется в `castclass` — инструкцию с runtime type-check, бросающую исключение при несовпадении. `as`/`is` компилируются в `isinst` — тоже проверка типа, но без исключения. Поэтому «лишний» downcast в тесном цикле не бесплатен, и это повод проектировать без него (generics, виртуальный метод), а не «просто кастить».

### 2. Type testing: is / as

`is` — булева проверка типа: `x is Dog` → `true`, если `x` имеет тип `Dog` или производный (или реализует интерфейс `Dog`, если это интерфейс). Две ловушки: `null is T` всегда `false` (включая `null is object`), и `is` матчит **по подтипам**.

С C# 7.0 есть pattern-форма: `if (x is Dog d) { d.Bark(); }` — за одну операцию проверяет тип, кастит и связывает переменную `d`, видимую там, где проверка истинна. Это сегодня предпочтительная идиома.

`as` — мягкое приведение: `var d = x as Dog;` вернёт объект как `Dog` или `null`, если не подходит, без исключения. Работает **только с reference types и nullable value types** (`o as int?` — ок; `o as int` не компилируется). Важно: `as` _не_ вызывает user-defined conversion operators и _не_ делает numeric conversions — только reference/box-проверку. Поэтому `as` ≠ «безопасный каст вообще»; это «безопасная ссылочная проверка».

---
### Внизу один IL-инструкшн

csharp

```csharp
obj as Animal      // IL: isinst Animal
obj is Animal      // IL: isinst Animal + проверка на null
(Animal)obj        // IL: castclass Animal  ← другой инструкшн!
```

`isinst` делает ровно одно: «проверь тип, верни ссылку если подходит, иначе null». Без эксепшнов.

- `as` отдаёт тебе результат `isinst` напрямую (ссылку или null)
- `is` берёт тот же `isinst` и сверху добавляет `!= null`, отдавая тебе `bool`
- А вот `(Animal)` — это уже `castclass`, который **кидает** `InvalidCastException` на промахе

Вот почему их пара: `is` и `as` — это две обёртки над одним и тем же безопасным механизмом. `is` спрашивает «да/нет», `as` спрашивает «дай ссылку или null». Один корень, разный возврат.

---

Какую форму выбрать:

- `if (x is Dog d)` — одна проверка типа (`isinst`), переменная в scope. **Дефолт.**
- `var d = x as Dog; if (d != null)` — тоже одна проверка. Уместно, когда `null` — нормальный продолжающийся путь, а не «не тот тип».
- `if (x is Dog) { var d = (Dog)x; }` — **антипаттерн**: тип проверяется дважды (`isinst` + `castclass`). С C# 7 так не пишут.

**`GetType()` vs `is`.** `x.GetType() == typeof(Dog)` — точное совпадение типа, наследников **не** матчит; `x is Dog` — матчит и наследников. Если `Puppy : Dog`, то `puppy is Dog` → `true`, а `puppy.GetType() == typeof(Dog)` → `false`. Плюс `GetType()` кинет NRE на `null`. Точный `GetType` нужен там, где важна инвариантность конкретного типа (классически — в `Equals`); `is` — для полиморфной проверки.

> **🔬 Глубже.** С C# 9 `is` оброс логическими и `not`-паттернами (`x is not null`, `x is Dog or Cat`) — но это территория главы про pattern matching; здесь нам нужен только type-test. На уровне IL антипаттерн `is`+`(T)` буквально генерирует два прохода по method table (`isinst`, затем `castclass`), тогда как `is T x` и `as` — один `isinst`.

### 3. Оператор каста `()` целиком

`()` — самый «всеядный» инструмент, и поэтому его путают с `as`. Он покрывает:

- **reference conversions** (up/down) — через `castclass`;
- **numeric conversions** — `(int)3.99` → `3` (усечение дробной части), `(byte)300` → переполнение;
- **boxing/unboxing** — `(object)i`, `(int)o` (см. ниже);
- **user-defined conversion operators** — если у типа объявлен `public static explicit operator Foo(Bar b)`, каст его вызовет.

Вот где проходит настоящая граница cast vs as: `as` делает **только** ссылочную/box-проверку и возвращает `null`; он не зовёт user-defined operators и не конвертит числа. `(int)myDecimal` сработает (numeric/user-defined), `myDecimal as int` — даже не скомпилируется. Поэтому «использую `as` вместо `()` ради безопасности» верно лишь для **ссылочных** приведений.

> **🔬 Глубже.** Numeric overflow и `checked`/`unchecked`: по умолчанию числовые сужения идут в `unchecked` — биты вне диапазона просто отбрасываются: для `int n = 300;` результат `(byte)n` равен `44` (300 mod 256). В `checked`-контексте (`checked { ... }` или компиляция с `/checked`) то же сужение бросит `OverflowException` **в рантайме**. Нюанс: с константой-литералом (`(byte)300`) перебор ловится ещё на **компиляции** — это compile error, а не runtime-исключение. Для целочисленных сужений из внешних данных оборачивай в `checked` или валидируй заранее. И помни: дробное → целое через `(int)` — всегда усечение, а не округление (округление — уже `Math.Round`/`Convert`).

### 4. Boxing / unboxing

**Boxing** — превращение value type в объект на куче. `object o = 42;` (или присваивание интерфейсу, который реализует value type) аллоцирует на куче «коробку»: object header (method table pointer + sync block index) плюс копия значения — и возвращает ссылку на неё. **Unboxing** — обратно: `int i = (int)o;` проверяет, что в коробке лежит ровно `int`, и копирует значение на стек.

Два момента, которые спрашивают:

1. **Цена.** Каждый boxing — это heap-аллокация → давление на GC. В горячем коде или цикле это главный скрытый источник мусора. Сам unboxing дёшев (проверка + копия); дорога именно аллокация при boxing.
2. **Unboxing требует точного типа.** Коробку с `int` нельзя анбоксить в `long`, даже хотя implicit-конверсия `int`→`long` существует: `long x = (long)(object)42;` → `InvalidCastException`. Сначала анбокс в исходный тип, потом конверсия: `long x = (int)(object)42;`.

![[Pasted image 20260623115654.png]]

> **🔬 Глубже.** Где boxing прячется незаметно:
> 
> - **Value type через интерфейс по ссылке.** `IComparable c = 42;` — boxing. А через generic с constraint (`where T : IComparable`) JIT специализирует код под value type и boxing не делает. Отсюда правило: `List<int>` (generic, хранит `int` инлайн) не боксит, а `ArrayList` (хранит `object`) боксит каждый элемент.
> - **`enum` и struct в `object`-API** — старый `string.Format(object)`, нестроготипизированные коллекции, события.
> - **Идиома избегания:** generics вместо `object`-API; для строк современная интерполяция (.NET 6+/C# 10) через `DefaultInterpolatedStringHandler` умеет избегать boxing многих value-типов, тогда как старое `"x=" + obj` / `string.Concat(object)` боксит.

### 5. Convert / Parse / TryParse

`Convert` — это вообще **не приведение типа**, а набор методов, которые _производят новое значение_, конвертируя между несвязанными базовыми типами через `IConvertible`: `Convert.ToInt32("42")`, `Convert.ToInt32(true)` → `1`. Поэтому он умеет то, чего не умеют ни `()`, ни `as`: `(int)"42"` даже не компилируется — между `string` и `int` нет conversion на уровне CLR, а `Convert` парсит.

Чем `Convert` отличается от `Parse` / `TryParse` для строк:

- `int.Parse("x")` — бросит `FormatException` на плохой строке.
- `int.TryParse("x", out var n)` — вернёт `false`, без исключения. **Предпочтительно для недоверенного ввода** (исключения дороги на горячем пути).
- `Convert.ToInt32(s)` — похож на `Parse`, но толерантен к `null` (`Convert.ToInt32((string)null)` → `0`); при этом на мусорной строке всё равно бросит `FormatException`.

Развенчание: «`Convert` безопаснее каста» — полуправда. Он умеет **больше** (несвязанные типы), но не «безопаснее» в смысле «не бросает»: `Convert.ToInt32("abc")` упадёт. Безопасный путь для строк — `TryParse`.

> **🔬 Глубже.** `Convert` использует **banker's rounding** (округление к чётному): `Convert.ToInt32(2.5)` → `2`, `Convert.ToInt32(3.5)` → `4`. Это дефолт `Math.Round` (`MidpointRounding.ToEven`) и он отличается от «школьного» round-half-up — частый сюрприз в денежных расчётах. Плюс многие `Convert`-методы культуро-зависимы (парсинг чисел/дат) и ломаются при смене локали.

### Interview layer

Вопросы по убыванию частоты. Формат: вопрос → likely follow-up → trap; затем shallow-ответ (почему плох) и tight model answer (RU + EN).

**Q1 (очень часто). В чём разница `is` и `as`?**  
→ FU: что вернёт `as` при неудаче? (`null`) А что бросит каст `()`? (`InvalidCastException`)  
→ trap: `null is object`? (`false`) / можно ли `as` с `int`? (нет — только nullable: `as int?`)

- _Shallow:_ «`is` проверяет, `as` кастит». Недостаточно — нет `null`/исключений/nullable, не сказано, что `as` не зовёт user-defined.
- _RU:_ «`is` — булева проверка типа, с C# 7 ещё и связывает переменную (`x is Dog d`). `as` — мягкое приведение к reference- или nullable-value-типу: возвращает объект или `null`, без исключения, и не вызывает user-defined/numeric conversions. Каст `()` — приведёт или бросит `InvalidCastException`. Современная идиома — `if (x is Dog d)`: одна проверка типа и переменная в scope.»
- _EN:_ «`is` is a boolean type test; since C# 7 it also binds (`x is Dog d`). `as` is a soft conversion to a reference or nullable value type — returns the object or `null`, never throws, and doesn't invoke user-defined or numeric conversions. A `()` cast either converts or throws `InvalidCastException`. The modern idiom is `if (x is Dog d)` — one check plus a scoped variable.»
- _Senior add-on:_ `is`+последующий cast проверяет тип дважды (`isinst`+`castclass`); `is T x` и `as` — один раз.

**Q2 (часто). Что такое boxing/unboxing и когда происходит?**  
→ FU: какова цена? (heap-аллокация → GC)  
→ trap: boxed `int` можно анбоксить в `long`? (нет — точный тип)

- _Shallow:_ «boxing — это упаковка value type в object». Верно, но без «аллокация на куче» и «точный тип при unbox» не переживёт follow-up.
- _RU:_ «Boxing — аллокация на куче коробки (object header + копия значения) при превращении value type в `object`/интерфейс; unboxing — копирование значения обратно с проверкой, что в коробке ровно этот тип. Главная цена — аллокация и давление на GC. Анбокс требует точного типа: `(long)(object)42` бросит `InvalidCastException`. Поэтому generics (`List<int>`) предпочтительнее `object`-API (`ArrayList`) — они не боксят.»
- _EN:_ «Boxing allocates a heap box (object header + a copy of the value) when a value type is converted to `object` or an interface; unboxing copies the value back, checking the box holds exactly that type. The real cost is the allocation and GC pressure. Unboxing needs the exact type: `(long)(object)42` throws `InvalidCastException`. That's why generics (`List<int>`) beat `object`-based APIs (`ArrayList`) — no boxing.»
- _Senior add-on:_ boxing прячется при вызове value type через интерфейс по ссылке; constrained generic его убирает.

**Q3 (часто). Upcast vs downcast — что безопасно и почему?**  
→ FU: что происходит с объектом при касте? (ничего — меняется только static type / view)  
→ trap: стоит ли downcast чего-то в рантайме? (да — `castclass`, проверка типа)

- _Shallow:_ «вверх безопасно, вниз нет». Верно, но без механизма слабо.
- _RU:_ «Upcast (наследник→база) неявен и безопасен: компилятор уже знает отношение типов, runtime-проверки нет, объект не меняется — меняется только static type переменной. Downcast (база→наследник) требует явного каста и runtime-проверки реального типа (`castclass`); при несовпадении — `InvalidCastException`. Объект всегда несёт реальный тип в method table pointer, поэтому виртуальные вызовы работают одинаково через любую переменную.»
- _EN:_ «Upcast (derived→base) is implicit and safe: the compiler already knows the relationship, no runtime check, the object is unchanged — only the variable's static type changes. Downcast (base→derived) needs an explicit cast and a runtime type check (`castclass`); a mismatch throws `InvalidCastException`. The object always carries its real type in its method-table pointer, so virtual dispatch behaves identically through any variable.»

**Q4 (mid). Зачем `Convert`, если есть `()` и `as`?**  
→ FU: `Convert` vs `Parse` vs `TryParse`?  
→ trap: `Convert.ToInt32("abc")` — что будет? (`FormatException` — не «безопасно»)

- _Shallow:_ «`Convert` безопаснее и универсальнее». Полуправда — надо уточнить.
- _RU:_ «`Convert` — не приведение, а методы, производящие новое значение между несвязанными типами через `IConvertible` (`string`→`int`, `bool`→`int`), чего `()`/`as` не умеют: `(int)"42"` не компилируется. Но "безопаснее" — миф: `Convert.ToInt32("abc")` бросит `FormatException`. Для строк из ненадёжного источника правильный инструмент — `int.TryParse`, возвращающий `false` вместо исключения.»
- _EN:_ «`Convert` isn't a cast — it's methods producing a new value between unrelated types via `IConvertible` (`string`→`int`, `bool`→`int`), which `()`/`as` can't: `(int)"42"` doesn't compile. But "safer" is a myth: `Convert.ToInt32("abc")` throws `FormatException`. For untrusted strings the right tool is `int.TryParse`, returning `false` instead of throwing.»

**Q5 (mid+). Чем `is T x` лучше `as`+null check?**  
→ trap: сколько раз проверяется тип в `is`+последующем cast? (дважды)

- _RU:_ «Обе формы делают одну проверку типа и обе валидны. `is T x` предпочитают за читаемость и узкий scope: переменная видна только там, где проверка истинна, и не "протекает" как `null`. Антипаттерн — `if (x is T) { var t = (T)x; }`: тип проверяется дважды (`isinst` + `castclass`). `as`+null уместен, когда `null` — легитимный продолжающийся путь.»
- _EN:_ «Both do a single type check and both are valid. `is T x` is preferred for readability and tight scoping — the variable lives only where the test is true and doesn't leak as `null`. The anti-pattern is `if (x is T) { var t = (T)x; }`, which checks twice (`isinst` + `castclass`). `as`+null fits when `null` is a legitimate continuing path.»

**Q6 (mid+, trap). `obj is Base` vs `obj.GetType() == typeof(Base)`?**

- _RU:_ «`is` матчит и подтипы: для `Derived : Base` `d is Base` → `true`. `GetType() == typeof(Base)` — точное совпадение, для `Derived` даст `false`, плюс кинет NRE на `null`. `is` — для полиморфной проверки; точный `GetType` — когда нужна инвариантность конкретного типа (часто в `Equals`).»
- _EN:_ «`is` matches subtypes: for `Derived : Base`, `d is Base` is `true`. `GetType() == typeof(Base)` is exact — `false` for `Derived`, and throws on `null`. Use `is` for polymorphic checks; exact `GetType` when you need invariance to the concrete type (often in `Equals`).»

**Q7 (🔬 senior). Где boxing прячется незаметно?**

- _RU:_ «Вызов value type через интерфейс по ссылке (`IComparable c = myStruct;`); `enum`/struct в `object`-API; элементы в нестроготипизированных коллекциях (`ArrayList`, `Hashtable`). Constrained generic (`where T : IComparable`) boxing убирает — поэтому generic-коллекции не боксят.»
- _EN:_ «Calling a value type through an interface reference (`IComparable c = myStruct;`); `enum`/struct passed to `object`-APIs; elements in non-generic collections (`ArrayList`, `Hashtable`). A constrained generic (`where T : IComparable`) removes the boxing — which is why generic collections don't box.»

**Q8 (🔬). Зовёт ли `as` user-defined conversion operators?**

- _RU:_ «Нет. `as`/`is` делают только ссылочную/box-проверку (`isinst`). User-defined `implicit`/`explicit operator` и numeric conversions вызывает только каст `()`. Это и причина, почему `as` ограничен reference- и nullable-value-типами.»
- _EN:_ «No. `as`/`is` do only a reference/box type check (`isinst`). User-defined `implicit`/`explicit operator` and numeric conversions are invoked only by the `()` cast — which is exactly why `as` is limited to reference and nullable value types.»
