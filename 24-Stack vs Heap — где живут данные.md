Эта глава — дом самого живучего мифа в C#: «value type живёт в стеке, reference — в куче». Миф настолько распространён, что его повторяют сами интервьюеры, — и именно поэтому умение его аккуратно разобрать выделяет тебя. Понимать, где на самом деле лежат данные, нужно не ради эрудиции: из этого вырастает интуиция про производительность (почему heap-доступ медленнее), про нагрузку на GC (что вообще порождает мусор), про замыкания, async и boxing. Все эти темы спрашивают как follow-up именно отсюда.

И снова про разделение с прошлой главой. Там был **контракт** типа (как копируется, как себя ведёт). Здесь — **физика**: какие есть области памяти, как они управляются, и где в итоге оказывается конкретное значение. Важно с самого начала: «где лежит» — это деталь реализации рантайма, а не семантика языка. Спека C# не обещает «в стеке» — она обещает value-семантику и время жизни. Это различие и есть ключ к разрушению мифа.

### Mental model

**Stack и heap — две области памяти с разным управлением временем жизни. Stack — LIFO, привязан к вызовам методов, освобождается сам. Heap — динамика, освобождается сборщиком мусора.**

А главное — формула, которая бьёт миф: **значение живёт там, где объявлено.** Локальная переменная — на стеке (или в регистре). Поле-value внутри объекта — в куче, внутри этого объекта. Элемент массива — в куче, внутри массива. Захваченная замыканием локальная — в куче, в объекте-замыкании. «Всегда в стеке» — неверно.

### Как это работает

#### Stack

Стек — это область памяти на **каждый поток свой**, работающая по принципу **LIFO**. На каждый вызов метода в нём создаётся **frame** (кадр), который хранит: локальные переменные, параметры, адрес возврата, сохранённые регистры и **ссылки** на объекты в куче. Доступ очень быстрый — это просто сдвиг указателя стека; освобождение тоже бесплатное — при выходе из метода кадр снимается целиком. Размер ограничен (по умолчанию ~1 МБ на поток на Windows; конфигурируется и зависит от платформы). Переполнение → `StackOverflowException`, которое **нельзя поймать** — процесс падает.

#### Heap

Куча — область для **динамических** аллокаций. В неё попадают объекты reference types (через `new`) и результаты boxing. Память освобождается **сборщиком мусора** — детерминированного `delete` нет (механика GC, поколения, compaction — следующая глава). Доступ медленнее: чтобы добраться до объекта, идём по ссылке (индирекция), и локальность кэша хуже, чем у плотно лежащих данных на стеке.

#### Где что живёт на самом деле (разрушаем миф)

Вот контекст, от которого зависит реальное место значения:

- **локальный value type** → стек (или вообще регистр, если JIT так решит);
- **value type как поле class'а** → внутри объекта, **в куче** (инлайнится в него);
- **value type как элемент массива** → внутри объекта-массива, **в куче**;
- **захваченная локальная** (lambda/async) → поднимается в heap-объект замыкания;
- **boxed value type** → **в куче**;
- **reference type**: сама _ссылка_ — там, где переменная (на стеке, если локальная; в куче, если это поле объекта), а _объект_ — всегда в куче.

Ключевой случай для follow-up'а — `struct` внутри `class`: struct-поле **инлайнится** в объект, без лишней индирекции и с хорошей локальностью. Обратный случай — `class` внутри `struct`: struct держит ссылку, а объект всё равно в куче. Посмотрим это на схеме:

#### 🔬 Глубже / на синьора

**Layout объекта на x64.** Каждый объект в куче несёт **header 16 байт**: 8 байт sync block index (блокировки, кэш хеш-кода, COM-интероп) + 8 байт указатель на method table (тип объекта, диспетчеризация виртуальных методов). Из-за выравнивания **минимальный размер объекта — 24 байта**, даже у класса без полей. У `struct` накладных расходов нет вообще: размер = сумма полей (плюс padding на выравнивание). Поэтому массив из миллиона struct'ов в разы компактнее массива из миллиона ссылок на объекты — нет ни заголовков, ни отдельных аллокаций, ни россыпи указателей.

**Истинная цена boxing — это память, а не только копия.** Boxed `int` на x64 ≈ **24 байта** (16 header + 4 байта значения + 4 padding) против **4 байт** «голого» int — то есть ×6 по памяти плюс отдельный объект, который GC обязан отслеживать и собирать. Когда говоришь «boxing дорогой», имей в виду именно это, а не мифический «оверхед на пометку». (Механика boxing — отдельная глава дальше.)

**Enregistration: «локал на стеке» — тоже упрощение.** JIT может держать локальную переменную целиком в **регистре CPU**, вообще не касаясь стека. Поэтому строго говоря даже «локальный value type → стек» неточно: он → стек _или регистр_, как решит JIT. Это ещё один довод, почему stack/heap — деталь реализации: спека не гарантирует ни того, ни другого, она гарантирует время жизни и value-семантику.

**Стек: размер и падение.** Дефолт ~1 МБ на поток на Windows; задаётся через конструктор `Thread(maxStackSize)` и отличается на других платформах (на Linux главный поток обычно больше). Глубокая рекурсия или огромный `stackalloc` → `StackOverflowException`, которое **по дизайну неперехватываемо** и убивает процесс: восстановиться из повреждённого стека нельзя.

**`stackalloc` / `Span<T>` (teaser).** Можно осознанно положить буфер на стек и работать с ним без аллокаций в куче — снимает нагрузку на GC в горячем пути. Полный разбор (`ref struct`, ограничения, escape-анализ) — в отдельной главе позже.

### Interview layer

#### Вопросы по частоте

1. **«В чём разница между stack и heap?»** — базовый. Ждут: LIFO/на поток/быстрый/сам освобождается против динамики/GC/индирекции.
2. **«Правда ли, что value types всегда в стеке?»** — **главная ловушка**. Правильный ответ — «нет», с контрпримерами.
3. **«Где лежит struct, объявленный полем класса?»** — проверка того же мифа под другим углом.
4. **«Почему доступ к heap медленнее?»** — индирекция + локальность кэша.
5. (синьор) **«Что произойдёт при stack overflow? Можно ли его поймать?»** / **«Из чего состоит заголовок объекта?»**

#### Follow-up tree

> **Q:** Чем отличаются stack и heap?  
> **→ likely:** А что именно попадает на стек, а что в кучу?  
> **→ trap:** «Значит, value type всегда в стеке?» (Нет — поле объекта, элемент массива, замыкание, boxing → куча.)

> **Q:** Где живёт `struct`, если он поле класса?  
> **→ likely:** А почему тогда говорят «struct в стеке»?  
> **→ trap:** «А `class` внутри `struct` — где объект?» (struct держит ссылку; объект в куче.)

> **Q:** Дорогой ли boxing?  
> **→ likely:** В чём именно цена?  
> **→ trap:** «Сколько памяти займёт boxed int против обычного?» (≈24 vs 4 байта на x64 + работа для GC.)

#### Частые поверхностные ответы и почему проваливаются

- **«Value type → стек, reference type → куча».** Главный миф. Неверно для value-поля объекта, элемента массива, захваченной переменной, boxed-значения. Подменяет «где лежит» (контекст) на «какой тип» (семейство).
- **«Stack быстрее, потому что меньше».** Не поэтому. Быстрее, потому что аллокация/освобождение — это сдвиг указателя, плюс данные лежат плотно и дружат с кэшем; индирекции нет.
- **«В куче лежат только классы».** Ещё там boxed value types, объекты-замыкания, массивы (включая `int[]`), и инлайн value-поля внутри объектов.
- **«Stack overflow можно обработать try/catch».** Нет — `StackOverflowException` неперехватываемо и валит процесс by design.

#### Model answer (mid+)

**RU.** Stack и heap — две области памяти с разным управлением. Stack — на каждый поток свой, работает по LIFO: на вызов метода создаётся кадр с локальными переменными, параметрами, адресом возврата и ссылками на объекты; аллокация и освобождение — это сдвиг указателя, поэтому очень быстро, но размер ограничен (~1 МБ). Heap — для динамических объектов (reference types через `new` и boxing), память освобождает GC; доступ медленнее из-за индирекции и худшей локальности. Главное: «value type всегда в стеке» — миф. Значение живёт там, где объявлено: локальное — в стеке или регистре, но value-поле класса лежит в куче внутри объекта, элемент массива — в куче, захваченная замыканием переменная — в heap-объекте замыкания, boxed-значение — в куче. У reference type в стеке (если локальная) лежит только ссылка, а объект — в куче.

_Senior add-on._ Где именно — деталь реализации, а не семантика языка: JIT может вообще держать локал в регистре. На x64 у объекта заголовок 16 байт (sync block + method table ptr) и минимум 24 байта; boxed `int` ≈ 24 байта против 4 — отсюда реальная цена boxing.

**EN.** Stack and heap are two memory regions with different management. The stack is per-thread and LIFO: each method call pushes a frame holding locals, parameters, the return address and references to objects; allocation and freeing are just a pointer move, so it's very fast but bounded (~1 MB). The heap holds dynamic objects (reference types via `new`, plus boxing) and is reclaimed by the GC; access is slower due to indirection and worse cache locality. The key point: "value types always live on the stack" is a myth. A value lives where it's declared: a local sits on the stack or in a register, but a value-typed field of a class lives on the heap inside that object, an array element lives on the heap, a captured local lives in a heap closure object, and a boxed value lives on the heap. For a reference type, only the reference is on the stack (if it's a local); the object is on the heap.

_Senior add-on._ Where exactly is an implementation detail, not language semantics — the JIT may keep a local entirely in a register. On x64 an object has a 16-byte header (sync block + method table ptr) and a 24-byte minimum; a boxed `int` is ~24 bytes versus 4 — that's the real cost of boxing.
